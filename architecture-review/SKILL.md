# 技术架构审查

> 💡 本 skill 可在 agent-pipeline 流程中使用，也可独立用于任何评审场景。

## 评审者思维模式：架构审计师

我不是来夸架构的。我是来给它做压力测试的。

"架构不是一次性的设计，是一系列权衡的持续演进。"

我的大脑默认运行在3年后的时间线。你给我看现在的系统，我看到的是：

- 100用户 → 哪些设计会变成绊脚石？
- 10万用户 → 哪个服务先熔断？
- 100万用户 → 数据库撑不撑得住？缓存策略还灵吗？

**每个架构决策都是一个时间炸弹，区别在于引信长短。**

### 扩展性压力测试
- 数据库：单表行数到1000万时查询还快吗？有分库分表预案吗？
- 服务间通信：同步RPC还是异步消息？链路超时怎么处理？
- 缓存层：缓存穿透、雪崩、击穿的防护做了吗？
- 存储层：文件存储到TB级别怎么办？对象存储成本算过吗？

### 技术债雷达
- 每个TODO和FIXME都是一笔隐性贷款，利率是团队速度
- 循环依赖 = 架构里的信用卡透支
- 缺少抽象的copy-paste代码 = 技术高利贷
- 没有监控的服务 = 盲人开高速

### 团队承载力
- 新人入职多久能改第一个feature？超过一周就有问题
- 核心模块只有一个人懂 = 公交车因子为1
- 文档是写给3个月后的自己看的，不是写给code review看的

### 审查风格
1. **画增长曲线** — 先标注当前架构的舒适区边界
2. **找第一个断裂点** — 哪里最先扛不住
3. **评估迁移成本** — 到时候重构要多大代价
4. **检查过渡路径** — 从A到B有没有不停机的方案

### 黑名单
- ❌ "以后再说" — 技术债的起点
- ❌ "先跑起来再优化" — 没有回头的路
- ❌ "这个简单直接写" — 三个月后你会恨自己
- ❌ "照着XX项目抄就行" — 上下文不同，照抄是灾难

### 输出格式

每个问题给三个维度：
- **影响范围：** 局部/系统级/全局
- **紧迫度：** 立即修/本迭代内/下个季度
- **修复成本：** 小修/中等重构/需要专门sprint

## 评估标准

# 架构审查评估标准 v2.0

## ⚠️ 关键声明：本清单不是全部

这份标准是你的**最低要求**，不是**全部要求**。

**你必须自由发挥，关注标准之外的任何问题**，包括但不限于：
- 架构设计里"感觉不对"但说不上来为什么的地方
- 看着合理但实际运维时会踩坑的设计
- 原作者可能因为偷懒而跳过的架构考量
- 行业最佳实践（如DDD、微服务模式、云原生12要素）中提到但清单没写的
- 真实生产环境会遇到的边缘情况

**如果只有这份清单里写了的问题才被你发现 = 你不合格。**

## 评分规则

| 分数 | 含义 |
|------|------|
| 10 | 几乎不可能。架构超越行业标杆，有你从清单外发现的关键亮点 |
| 9 | 有明确亮点 + 清单外发现，无任何黄/红 |
| 8 | 无黄/红，扎实可靠 |
| 7 | 默认起点。有小问题但整体可用 |
| 6 | 有多个🟡 |
| 5 | 有🔴或大量🟡 |
| ≤4 | 架构需重做 |

### 严重级别

- 🔴 **阻断**：必须修复才能继续，一票否决项
- 🟡 **重要**：强烈建议修复，不修会在生产环境出事
- 🟢 **建议**：锦上添花，有时间就改

### 一票否决（出现即≤5分）
1. 架构无法支撑核心业务场景
2. 单点故障无容灾方案
3. 安全架构有致命缺陷（如敏感数据明文传输、无认证即可访问核心接口）
4. 数据一致性方案缺失（涉及资金/库存等关键数据时）

## 1. 架构合理性（12%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| AR-01 | 关注点分离是否清晰：业务逻辑、基础设施、接入层是否混杂在一起 | 🟡 |
| AR-02 | 模块边界是否明确：每个模块是否有单一职责，模块间的接口是否最小化 | 🟡 |
| AR-03 | 层次划分是否合理：Controller/Service/Repository 等分层是否严格，是否存在跨层调用 | 🟡 |
| AR-04 | 依赖方向是否正确：高层模块是否依赖低层模块，是否存在反向依赖或双向依赖 | 🟡 |
| AR-05 | 是否存在循环依赖：模块A→B→A 或间接循环，工具链能否检测 | 🔴 |
| AR-06 | 架构风格是否适配业务：单体/微服务/Serverless/事件驱动的选择是否有依据 | 🟡 |
| AR-07 | 领域模型是否与技术框架解耦：核心业务逻辑是否依赖了特定的ORM/框架 | 🟡 |
| AR-08 | 是否有无状态的缩放点：有状态的组件（如本地缓存、Session）是否会阻碍水平扩展 | 🟡 |
| AR-09 | 基础设施关注点是否下沉：日志、监控、配置、安全等横切关注点是否通过中间件/AOP统一处理 | 🟢 |
| AR-10 | 包/模块的命名是否能反映业务概念而非技术实现 | 🟢 |
| AR-11 | 是否存在上帝类/God Module：单个类或模块承担过多职责（超过500行或接口超过20个） | 🟡 |
| AR-12 | 是否考虑了限界上下文（Bounded Context）：不同子域之间的模型是否做了隔离 | 🟢 |
| AR-13 | 是否有清晰的架构决策记录（ADR）：关键决策的why是否有文档 | 🟢 |
| AR-14 | 是否考虑了组件的替换成本：某个数据库/消息队列/MQ被替换时，需要改多少代码 | 🟢 |

## 2. 技术选型（12%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| TS-01 | 选型依据是否文档化：为什么选A不选B，有对比矩阵还是凭感觉 | 🟡 |
| TS-02 | 是否存在过度追新：引入了<2年且社区小的技术，收益是否足以覆盖风险 | 🟡 |
| TS-03 | 与团队技术栈匹配度：引入新技术后团队的学习曲线和招聘成本是否考虑 | 🟡 |
| TS-04 | 社区活跃度：GitHub stars、issue响应速度、release频率是否健康 | 🟢 |
| TS-05 | 长期维护风险：该技术是否由商业公司单方面控制、是否有vendor lock-in | 🟡 |
| TS-06 | 是否评估了至少2个替代方案 | 🟡 |
| TS-07 | 数据库选型是否合理：关系型/文档型/时序/图数据库是否匹配数据特征 | 🟡 |
| TS-08 | 消息队列选型：Kafka/RabbitMQ/RocketMQ的选择是否基于吞吐量、延迟、可靠性的权衡 | 🟢 |
| TS-09 | 前端框架/构建工具版本是否在LTS范围内 | 🟢 |
| TS-10 | 是否存在不必要的多样性：类似功能用了多个同类技术（如同时用Redis和Memcached做缓存） | 🟡 |
| TS-11 | 第三方依赖的许可证是否合规（GPL/AGPL等传染性许可证） | 🔴 |
| TS-12 | 是否考虑了依赖的安全漏洞扫描机制（如Dependabot/Snyk） | 🟢 |
| TS-13 | 引入的中间件运维复杂度是否可接受：需要专门的运维人员吗 | 🟢 |

## 3. API设计（12%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| AP-01 | RESTful规范：URL是否用名词复数、HTTP方法是否语义正确（GET无副作用、POST创建） | 🟡 |
| AP-02 | API版本管理：是否有版本策略（URL路径/Header），旧版本废弃策略是否明确 | 🟡 |
| AP-03 | 错误码体系：是否有统一的错误码规范，错误信息是否包含足够的排查信息但不过度暴露内部细节 | 🟡 |
| AP-04 | 分页规范：是否统一使用cursor-based或offset-based，大集合下offset的性能问题是否考虑 | 🟡 |
| AP-05 | 排序过滤：多字段排序、范围查询、模糊搜索是否支持，SQL注入风险是否防护 | 🟡 |
| AP-06 | 批量操作：是否支持批量创建/更新/删除，批量上限是多少，是否有防误操作 | 🟢 |
| AP-07 | 幂等设计：POST/PUT是否考虑了重复提交场景，有无幂等Key机制 | 🟡 |
| AP-08 | 限流策略：API是否有速率限制，限流粒度（用户/IP/租户），超限后的响应码 | 🟡 |
| AP-09 | 请求/响应格式：是否统一使用JSON，字段命名规范（camelCase/snake_case）是否全局一致 | 🟢 |
| AP-10 | 认证信息传递：Token放在Header还是Query，是否有暴露风险 | 🔴 |
| AP-11 | 接口文档：是否有OpenAPI/Swagger规范，文档与代码是否自动同步 | 🟡 |
| AP-12 | 部分更新：PATCH请求是否支持JSON Patch或字段级更新 | 🟢 |
| AP-13 | 长任务处理：耗时操作是否返回任务ID+状态查询接口，而非同步等待 | 🟡 |
| AP-14 | API兼容性承诺：是否有破坏性变更的升级路径和通知机制 | 🟢 |

## 4. 数据模型（12%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| DM-01 | 表结构是否满足3NF，反范式设计是否有明确理由和性能需求支撑 | 🟡 |
| DM-02 | 索引设计：是否有覆盖索引避免回表，复合索引的字段顺序是否遵循最左匹配 | 🟡 |
| DM-03 | 分区策略：大表是否有分区方案（按时间/哈希），分区键选择是否合理 | 🟢 |
| DM-04 | 数据迁移方案：DDL变更是否有线上迁移方案（pt-online-schema-change/gh-ost），大表变更是否会锁表 | 🔴 |
| DM-05 | 软删除vs硬删除：是否有统一的删除策略，软删除数据如何定期清理，唯一索引如何处理 | 🟡 |
| DM-06 | 审计字段：create_time/update_time/create_by/update_by/is_deleted 是否统一 | 🟡 |
| DM-07 | 时区处理：时间存储用UTC还是本地时间，展示层如何转换，夏令时是否考虑 | 🟡 |
| DM-08 | 主键策略：自增ID vs UUID vs 雪花算法，对索引性能和分布式的影响是否评估 | 🟡 |
| DM-09 | 字段类型选择：金额用Decimal而非Float，手机号用VARCHAR而非INT，枚举值管理方式 | 🔴 |
| DM-10 | 外键约束：线上是否使用外键（建议应用层保证），级联删除的风险 | 🟡 |
| DM-11 | 大字段处理：TEXT/BLOB字段是否考虑分离表或使用对象存储 | 🟢 |
| DM-12 | 数据生命周期：冷热数据分离方案，归档策略，数据保留期限 | 🟢 |
| DM-13 | 并发控制：乐观锁（version字段）还是悲观锁，高并发场景的死锁风险评估 | 🟡 |

## 5. 扩展性（10%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| EX-01 | 水平扩展能力：无状态服务是否能随意增减实例，有状态组件的扩展方案 | 🟡 |
| EX-02 | 配置外部化：硬编码配置是否清除，配置中心（Nacos/Apollo/Consul）是否引入 | 🟡 |
| EX-03 | 插件化能力：核心流程是否支持通过SPI/插件机制扩展，而非修改核心代码 | 🟢 |
| EX-04 | 多租户支持：数据隔离方案（共享DB共享表/共享DB独立表/独立DB）是否明确 | 🟢 |
| EX-05 | Feature Flag：是否支持灰度发布，新功能是否能按比例/白名单开启 | 🟢 |
| EX-06 | 数据库分库分表：是否预留分片能力，分片键选择是否合理 | 🟡 |
| EX-07 | 服务注册发现：是否使用服务注册中心而非硬编码IP | 🟡 |
| EX-08 | 负载均衡策略：轮询/权重/一致性哈希，对有状态服务的会话保持是否处理 | 🟢 |
| EX-09 | 是否遵循云原生12要素：配置分离、无状态、端口绑定、并发扩展、日志流、进程平等 | 🟡 |
| EX-10 | 国际化（i18n）：是否预留多语言支持能力，文案是否外部化管理 | 🟢 |
| EX-11 | Schema Evolution：数据库/消息格式的变更是否向后兼容，如何处理消费端版本不一致 | 🟡 |

## 6. 安全性设计（10%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| SE-01 | 认证架构：登录方案（JWT/Session/OAuth2）选择是否合理，Token刷新机制是否完善 | 🔴 |
| SE-02 | 授权模型：RBAC/ABAC权限模型是否覆盖所有接口，是否存在越权访问风险 | 🔴 |
| SE-03 | 数据加密：传输层（TLS）和存储层（字段级加密）加密方案是否完整 | 🔴 |
| SE-04 | 密钥管理：密钥是否硬编码，是否使用KMS/Vault等密钥管理系统，密钥轮换策略 | 🔴 |
| SE-05 | 审计日志：敏感操作是否有不可篡改的审计记录，日志是否脱敏 | 🟡 |
| SE-06 | 安全Headers：CSP/X-Frame-Options/X-Content-Type-Options/HSTS是否配置 | 🟡 |
| SE-07 | CORS策略：是否白名单制而非`*`，是否区分了简单请求和预检请求 | 🟡 |
| SE-08 | SQL注入防护：是否全部使用参数化查询，ORM是否有原生SQL拼接风险 | 🔴 |
| SE-09 | XSS防护：输入是否过滤+输出是否转义，Content-Security-Policy是否配置 | 🟡 |
| SE-10 | CSRF防护：是否有CSRF Token，Cookie的SameSite/Site属性是否合理 | 🟡 |
| SE-11 | 敏感数据处理：日志中是否脱敏（手机号/身份证/银行卡），接口返回是否按需脱敏 | 🟡 |
| SE-12 | 接口鉴权一致性：是否有统一的鉴权拦截器，是否存在未鉴权的接口被遗漏 | 🔴 |
| SE-13 | 密码存储：是否使用bcrypt/argon2而非MD5/SHA1，是否有密码强度校验 | 🔴 |
| SE-14 | 文件上传安全：是否限制文件类型/大小，上传文件是否存放在非Web可访问目录，是否扫描恶意文件 | 🟡 |

## 7. 性能设计（10%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| PF-01 | 缓存架构：缓存层级（本地缓存→分布式缓存→CDN）是否合理，缓存穿透/击穿/雪崩是否防护 | 🟡 |
| PF-02 | 异步处理：非实时操作是否异步化（消息队列/事件驱动），同步调用的超时风险 | 🟡 |
| PF-03 | CDN策略：静态资源是否上CDN，缓存策略（Cache-Control/ETag）是否合理 | 🟢 |
| PF-04 | 数据库读写分离：读多写少场景是否做了读写分离，主从延迟如何处理 | 🟡 |
| PF-05 | 连接池配置：数据库/HTTP/RPC连接池大小是否合理，是否有连接泄漏检测 | 🟡 |
| PF-06 | 超时设置：所有外部调用是否有超时，超时后的降级策略，是否会级联超时 | 🟡 |
| PF-07 | 批量操作优化：循环调用是否改为批量，N+1查询问题是否存在 | 🟡 |
| PF-08 | 资源压缩：HTTP响应是否开启Gzip/Brotli，图片是否有WebP/懒加载 | 🟢 |
| PF-09 | 是否有性能基准测试：核心接口的P99延迟目标是多少，是否有压测数据和容量规划 | 🟡 |
| PF-10 | JVM/运行时调优：堆内存/GC策略是否根据业务特点调整，而非使用默认值 | 🟢 |
| PF-11 | 延迟敏感路径：核心交易链路是否有性能预算，每个环节的耗时上限是否明确 | 🟡 |
| PF-12 | 大数据量导出：报表导出是否有流式处理，是否会OOM | 🟡 |

## 8. 可观测性（7%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| OB-01 | 日志规范：日志级别（DEBUG/INFO/WARN/ERROR）使用是否合理，是否有关联ID串联请求链路 | 🟡 |
| OB-02 | 结构化日志：日志是否JSON格式化，便于ELK/Loki检索，是否有统一的日志框架 | 🟢 |
| OB-03 | 监控指标：是否有RED指标（Rate/Error/Duration）和USE指标（Utilization/Saturation/Errors） | 🟡 |
| OB-04 | 链路追踪：是否接入OpenTelemetry/Jaeger/SkyWalking，跨服务调用是否可追踪 | 🟡 |
| OB-05 | 告警策略：告警阈值是否基于SLA设定，是否存在告警风暴，告警是否有分级（P0/P1/P2） | 🟡 |
| OB-06 | 健康检查：Liveness/Readiness探针是否配置，依赖组件（DB/Redis/MQ）的健康状态是否纳入检查 | 🟡 |
| OB-07 | SLA定义：核心服务的可用性目标是几个9，对应的MTTR/MTBF是否定义 | 🟡 |
| OB-08 | Dashboard：是否有业务大盘（QPS/错误率/延迟分布），运维是否能在1分钟内判断系统状态 | 🟢 |
| OB-09 | 日志采样：高QPS服务的日志采样策略，错误日志是否保证不采样 | 🟢 |

## 9. 容灾设计（5%）

### 检查清单

| # | 检查项 | 级别 |
|---|--------|------|
| DR-01 | 降级方案：核心功能不可用时的降级策略（返回默认值/排队/关闭非核心功能） | 🟡 |
| DR-02 | 熔断机制：外部依赖故障时是否有熔断器（Hystrix/Sentinel/Resilience4j），熔断后如何恢复 | 🟡 |
| DR-03 | 重试策略：失败重试次数、间隔（固定/指数退避）、是否可重试（GET可重试、POST需幂等） | 🟡 |
| DR-04 | 数据备份：数据库备份策略（全量+增量），备份恢复时间（RTO），恢复演练频率 | 🔴 |
| DR-05 | 灾备切换：是否有多机房/多云容灾，切换流程是否文档化并演练过 | 🟡 |
| DR-06 | 故障隔离：单个服务故障是否会级联（雪崩效应），是否有bulkhead隔离 | 🟡 |
| DR-07 | 幂等消费：消息队列重复投递时消费端是否能幂等处理 | 🟡 |
| DR-08 | 优雅上下线：服务重启时是否有graceful shutdown，正在处理的请求是否能完成 | 🟢 |
| DR-09 | Chaos Engineering：是否有故障注入测试计划（如定期关闭某个节点观察系统行为） | 🟢 |

## 12. 一眼AI套路检查（25项）

### 眼睛直觉类（EYE-01~10）

| # | 模式 | 说明 |
|---|------|------|
| EYE-01 | 结构化强迫症 | 每个段落都是三段论：背景→方案→总结，读起来像教科书 |
| EYE-02 | 列表依赖 | 一个观点不用列表就不会表达，5个要点硬凑成10个 |
| EYE-03 | 过度分类 | 什么都要分三级五类，简单问题复杂化 |
| EYE-04 | 比喻泛滥 | "就像建造一座大厦""如同指挥一个交响乐团"——资深工程师写文档不用比喻 |
| EYE-05 | 格式极致统一 | 所有标题长度相近、所有列表项结构一致——真人写的不会这么整齐 |
| EYE-06 | 无主语 | 大量"建议考虑""推荐使用"——谁建议？谁推荐？ |
| EYE-07 | 假装谦虚 | "本文仅供参考""可能存在不足"——然后下面写得跟权威指南一样 |
| EYE-08 | 没有具体数字 | "适当增加""合理配置""根据实际情况调整"——是多少？ |
| EYE-09 | 过度封装 | 一个简单的JSON示例非要加上完整的项目上下文和说明 |
| EYE-10 | 首尾呼应病 | 开头说"本文将探讨X"，结尾说"综上所述X"，像高考作文 |

### 可疑模式类（SUS-01~07）

| # | 模式 | 说明 |
|---|------|------|
| SUS-01 | 完美的"不过" | 每个优点后面都跟一个"不过"——真人写文档会直接说问题 |
| SUS-02 | 万能免责 | "具体实现需要根据业务场景调整"——这是废话，不是结论 |
| SUS-03 | 技术词堆砌 | 在一段话里塞满技术名词，像是在证明"我懂这些" |
| SUS-04 | 没有踩坑记录 | 所有方案都描述得顺风顺水，没有"我们踩了X坑"的真实经历 |
| SUS-05 | 过度展望 | 文档花大量篇幅描述未来规划，当前问题一笔带过 |
| SUS-06 | 参考文献造假 | 列了一堆参考资料但格式不统一或链接已失效 |
| SUS-07 | 一致性异常 | 有些地方写得很详细（AI擅长），有些地方一笔带过（AI偷懒） |

### 文档风格类（DOC-01~07）

| # | 模式 | 说明 |
|---|------|------|
| DOC-01 | 表情符号滥用 | 每个标题都带emoji，看起来像产品宣传稿而非技术文档 |
| DOC-02 | 过度强调 | 一个文档里用了20+个"**加粗**"，全部强调等于没有强调 |
| DOC-03 | 引用块过度 | 大量使用blockquote，仿佛每句话都是名言 |
| DOC-04 | 代码注释废话 | `// 设置用户名` `// 返回结果`——这种注释不如不写 |
| DOC-05 | README模板感 | 看起来像是从某个awesome-xxx仓库复制来的模板 |
| DOC-06 | 没有TODO/FIXME | 真实项目的文档里一定有未完成项，AI生成的全是"完成态" |
| DOC-07 | 变更记录完美 | CHANGELOG每条都格式工整，没有紧急热修复、没有"撤销上次变更" |
