# 技术Spike验证方法论

> 本 skill 可在 agent-pipeline 流程中使用，也可独立使用。

## 角色定义

- **执行者：** 架构师或指定开发者
- **决策者：** 项目负责人

## 核心原则

**Spike 不是开发，是验证。** 目标是用最小代价回答一个技术问题：这个方案行不行？

---

## Spike 触发条件

由架构设计阶段标记的高风险技术点触发：
- 🔴 高风险：核心技术可行性未知、涉及第三方不稳定API、性能要求极高
- 🟡 中风险：技术方案已知但有实现细节不确定性

**也可独立触发：** 当任何技术决策存在重大不确定性时，都可以发起 Spike。

---

## Spike 执行流程

### 时间盒控制

| 规则 | 说明 |
|------|------|
| 时限 | ≤4h（超时视为失败，触发回退） |
| 目标 | 最小可行验证，不是完整实现 |
| 产出 | 结论 + 代码原型（如有） |

### 执行步骤

1. **明确验证目标** — 用一句话描述要验证什么
2. **设计最小实验** — 写最少代码证明可行性（原型、PoC、调用测试）
3. **执行实验** — 在时间盒内完成
4. **记录结论** — 按模板产出 Spike 报告

---

## Spike 报告模板

执行者应按以下格式产出报告：

```markdown
## Spike报告：{技术点名称}

### 验证目标
{具体要验证什么，一句话}

### 验证过程
{做了什么实验、写了什么原型、调用了什么API、遇到了什么问题}

### 结论
- ✅ 可行 / 🟡 部分成功（需调整）/ ❌ 不可行

### 原型代码
→ {路径}

### 建议
{对后续阶段的影响、需要注意的事项}
```

---

## 三路决策机制

Spike 完成后，根据结论走不同路径：

### ✅ 成功

核心功能完全可行，无重大限制。

**处理：** 继续后续阶段，Spike结论附入架构文档。

### 🟡 部分成功

核心功能可用，但存在已知限制。

**部分成功的量化定义：**

部分成功 = 核心功能可用，但存在以下至少一项：
- 性能不达标（但可接受降级方案）
- 仅支持部分场景（需明确列出不支持的场景）
- 依赖额外条件（如特定配置、额外中间件）
- 实现复杂度高于预期（需评估工期影响）

**报告必须包含：**
- 限制清单（如：性能不达标、仅支持部分场景）
- 变通方案（如：可用方案B绕过限制的场景）
- 影响评估（对架构方案和项目工期的影响）

**决策：** 项目负责人根据限制清单和变通方案决定：
- 接受限制继续前进
- 调整技术方案（回退到架构设计阶段修改）
- 简化需求范围（移除依赖该技术点的功能）

### ❌ 失败

技术方案不可行，无法在限制内实现。

**处理：** 回退到架构设计阶段，规避该技术点或寻找替代方案。

---

## 决策路径图

```
Spike 完成
  │
  ├─ ✅ 成功 → 继续后续阶段，结论附入架构文档
  │
  ├─ 🟡 部分成功 → 项目负责人决定：
  │     ├─ 接受限制 → 继续前进
  │     ├─ 调整方案 → 回退到架构设计修改
  │     └─ 简化需求 → 降低预期继续
  │
  └─ ❌ 失败 → 回退到架构设计，寻找替代方案
```

## Spike 报告验收标准

- [ ] 验证目标明确（一句话能说清要验证什么）
- [ ] 验证过程具体（不是"测试了一下"，而是描述了具体实验步骤）
- [ ] 结论清晰（✅/🟡/❌ 三选一，不含糊）
- [ ] 部分成功时有限制清单和变通方案
- [ ] 行数 ≥ 30行

---

## ⚠️ 防偷懒规则

### 1. 时间盒强制执行

- Spike **必须**在4小时内完成（含报告编写）
- 4小时到 → **强制停止**，不管做到哪一步
- 不允许"再给我1小时就搞定了"——时间盒不可延长
- 原因：Spike 是可行性验证，不是完整开发，4小时足够判断方向

### 2. 报告强制内容

| 章节 | 强制要求 | 偷懒信号 |
|------|---------|---------|
| 验证目标 | 一句话说明要验证什么 | ❌ "验证技术可行性"太模糊 |
| 验证过程 | 具体实验步骤（不是概述） | ❌ "搭建了环境并测试"没有细节 |
| 结论 | 明确给出成功/部分成功/失败 | ❌ "基本可用"不明确 |
| 原型代码 | 必须有可运行的代码片段 | ❌ 只描述方案没有代码 |
| 建议 | 下一步怎么做 | ❌ 缺少建议 |

### 3. "部分成功"禁止模糊表述

**不允许以下表述：**
- ❌ "基本可用" → 必须说清楚哪些可用、哪些不行
- ❌ "有一些小问题" → 必须列出具体问题
- ❌ "大部分功能正常" → 必须给出百分比或具体列表

**要求格式：**
```
部分成功：
- ✅ 可用：[列出具体可用功能]
- ❌ 不可用：[列出具体限制]
- ⚠️ 变通方案：[如何绕过限制]
- 📊 可行度评估：XX%
```

### 4. 偷懒检测信号

- SPIKE-REPORT.md 行数 < 30
- 没有可运行的原型代码
- 验证过程描述过于简短（< 50字）
- 结论为模糊表述（"基本可以"、"差不多"）
- 缺少下一步建议


> **产出路径：** 由调用方（协调者/用户）在任务消息中指定。未指定时默认 `/tmp/agent-skills-output/`。本skill不硬编码产出位置，可独立使用于任何项目。
