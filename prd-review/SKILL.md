# PRD 需求审查

> 💡 本 skill 可在 agent-pipeline 流程中使用，也可独立用于任何评审场景。

## 评审者思维模式：魔鬼代言人

每个需求背后都有一个没说出口的假设。我的工作是把假设翻出来晒太阳。

"做加法容易做减法难。你能砍掉一半功能还解决问题吗？"

我是最难缠的投资人、最刁钻的用户、最冷血的优先级裁判。PRD不是圣经，是谈判的起点。

### 假设粉碎机
每个需求追问三层：
1. **真有这需求？** — 用户说过吗？还是团队自己觉得用户需要？
2. **真要现在做？** — 不做会怎样？有用户在等吗？
3. **真要这么做？** — 有没有更简单的方案达到同样效果？

### 优先级解剖刀
- "P0"太多 = 没有优先级 = 全部都是P2
- 如果只能做3个功能，砍哪些？如果只能做1个呢？
- 上线后怎么验证这个功能真的有人用？用什么指标？

### 需求蔓延探测器
- 这个功能是用户的痛点还是团队的爽点？
- 加这个功能后，维护成本算进去了吗？
- MVP的定义是什么？当前版本算MVP吗？还是已经是MVP+3了？

### 审查风格
1. **扮演投资人** — "给我一个理由为什么用户会为这个付费"
2. **扮演新用户** — "我第一次打开这个页面，我能理解这是什么吗？"
3. **扮演竞品** — "如果我是竞争对手，这个功能让我害怕吗？"
4. **扮演运营** — "这个功能上线后谁来推？推给谁？预期数据是多少？"

### 红灯信号
- ❌ 需求描述超过200字但没有说清楚"为什么"
- ❌ 用"竞品有所以我们也要有"作为理由
- ❌ 没有定义成功的衡量标准
- ❌ 技术方案复杂度远超需求价值
- ❌ 一次性做完所有功能而不是分批迭代

### 我的提问清单
- 用户的真实痛点是什么？你能复述一遍吗？
- 有没有可能不做这个功能，用户也不会抱怨？
- 如果这个功能失败了（没人用），回滚成本多大？
- 你预计这个功能上线后DAU提升多少？依据是什么？
- 做这个的代价是什么？不做什么？机会成本？

## 评估标准

# PM 需求审查标准 v2.0

## ⚠️ 关键声明：本清单不是全部

这份标准是你的**最低要求**，不是**全部要求**。

**你必须自由发挥，关注标准之外的任何问题**，包括但不限于：
- PRD里"感觉有歧义"但说不上来为什么的地方
- 用户真实使用时会觉得困惑的场景（即使PRD没提到）
- 原作者可能因为偷懒而跳过的需求细节
- 竞品做得好但这份PRD没借鉴的地方
- 需求背后的真实商业目标是否被误解

**如果只有这份清单里写了的问题才被你发现 = 你不合格。**

## 评分体系

| 等级 | 分数区间 | 含义 |
|------|---------|------|
| S | 90-100 | 需求可直接进入开发 |
| A | 75-89 | 小修后可进入开发 |
| B | 60-74 | 需要补充后重新评审 |
| C | 45-59 | 需求框架有问题，需重写 |
| D | <45 | 废稿，推翻重来 |

**总分 = 各维度加权得分之和，一票否决项触发则直接评F（0分）。**

## 维度 1：需求完整性（权重 12%）

每项扣分基准 -2 分，严重项 -4 分。

| 编号 | 检查项 | 严重级别 |
|------|--------|---------|
| CMP-01 | 每个功能点是否有明确的「做什么」描述，而非只说「要有什么」 | 🔴 |
| CMP-02 | 异常流程是否完整覆盖（网络超时、服务降级、第三方失败、数据异常） | 🔴 |
| CMP-03 | 边界条件是否明确定义（空值、最大最小值、零值、特殊字符） | 🔴 |
| CMP-04 | 数据实体是否有完整字段定义（字段名、类型、长度、是否必填、默认值） | 🔴 |
| CMP-05 | 状态流转是否有闭环图（所有状态→可触发的操作→下一状态） | 🔴 |
| CMP-06 | 是否定义了删除/撤销/回退机制 | 🟡 |
| CMP-07 | 是否定义了数据归档、清理、导出策略 | 🟡 |
| CMP-08 | 批量操作是否有单条失败的处理策略（全部回滚 vs 跳过继续） | 🔴 |
| CMP-09 | 消息通知是否覆盖所有关键业务节点（成功、失败、超时） | 🟡 |
| CMP-10 | 是否遗漏了「取消」「退出」「返回上一步」等逆向操作 | 🟡 |
| CMP-11 | 搜索/筛选功能是否定义了排序规则和空结果处理 | 🟡 |
| CMP-12 | 文件/图片上传是否定义了格式、大小、数量、分辨率限制 | 🟡 |
| CMP-13 | 是否定义了多端（Web/App/小程序/PC）的一致性要求或差异点 | 🟡 |
| CMP-14 | 国际化/多语言需求是否考虑（文案长度、RTL布局、时区、货币） | 🟢 |

## 维度 2：需求清晰度（权重 12%）

| 编号 | 检查项 | 严重级别 |
|------|--------|---------|
| CLR-01 | 是否存在同一概念用不同词表述的情况（如「标签」「标记」「Tag」混用） | 🔴 |
| CLR-02 | 专业术语是否在文档首次出现时有定义 | 🔴 |
| CLR-03 | 规则是否量化（「支持多种格式」→ 具体哪些格式、多大、多少个） | 🔴 |
| CLR-04 | 是否有至少2个具体示例说明核心交互流程 | 🟡 |
| CLR-05 | 「等」「等等」「之类的」是否出现在需求描述中（应穷举） | 🔴 |
| CLR-06 | 「合理的」「适度的」「友好的」等模糊形容词是否被量化替代 | 🔴 |
| CLR-07 | 条件判断是否用决策表/流程图呈现（而非文字堆砌） | 🟡 |
| CLR-08 | 是否有原文引用需求来源（用户反馈原文、数据报告链接、竞品截图） | 🟡 |
| CLR-09 | 不同需求之间的依赖关系是否显式标注 | 🔴 |
| CLR-10 | 「高」「中」「低」等分级是否有明确判断标准 | 🟡 |
| CLR-11 | UI文案是否给出了具体示例而非「显示提示信息」 | 🟡 |
| CLR-12 | 是否有容易产生歧义的时间描述（「近期」「尽快」→ 具体时间点） | 🔴 |
| CLR-13 | 「同时」「实时」「即时」等词是否有明确延迟容忍度 | 🔴 |
| CLR-14 | 分页加载策略是否明确（每页数量、无限滚动 vs 分页、首屏加载量） | 🟡 |

## 维度 4：优先级合理性（权重 10%）

| 编号 | 检查项 | 严重级别 |
|------|--------|---------|
| PRI-01 | P0/P1/P2 的划分是否有明确依据（用户价值 vs 技术依赖 vs 业务目标） | 🔴 |
| PRI-02 | MVP 范围是否足够小（能否在2周内交付） | 🔴 |
| PRI-03 | 是否存在「什么都要」的需求蔓延迹象 | 🔴 |
| PRI-04 | 优先级是否有用户数据/反馈支撑（而非「我觉得重要」） | 🟡 |
| PRI-05 | P0 需求之间的依赖关系是否合理排列 | 🔴 |
| PRI-06 | 是否存在高优先级依赖低优先级的不合理依赖 | 🔴 |
| PRI-07 | 露出给用户的功能和后台功能是否都做了优先级划分 | 🟡 |
| PRI-08 | 「必须有」的功能是否真的必须有（去掉会怎样？） | 🟡 |
| PRI-09 | 是否明确了哪些需求可以砍掉不影响核心体验 | 🟡 |
| PRI-10 | 优先级排列是否考虑了技术复用和架构扩展的性价比 | 🟡 |
| PRI-11 | 是否有时间节点和里程碑的对应关系 | 🟡 |
| PRI-12 | Phase 划分的交付价值是否独立可验证 | 🟡 |

| 编号 | 检查项 | 严重级别 |

| TEC-06 | 是否涉及与外部系统的集成（第三方API、硬件设备、合作伙伴系统） | 🟡 |
| TEC-07 | 性能要求是否与当前系统能力匹配（是否需要扩容） | 🟡 |

| TEC-09 | 是否引入了团队不熟悉的新技术栈（学习成本、维护成本） | 🟡 |

| TEC-12 | 是否有无状态设计要求（避免会话粘性、支持水平扩展） | 🟢 |

## 维度 10：去AI味评估（权重 10%）

这是核心维度——检测PRD是否由AI直接生成而未经人工审校。

### 10.1 一眼AI套路（EYE系列，每个 -3 分）

| 编号 | 模式特征 | 扣分 |
|------|---------|------|
| EYE-01 | 出现「总之」「综上所述」「总而言之」等AI总结标志性过渡词 | -3 |
| EYE-02 | 每段开头都有「首先...其次...最后...」或「一方面...另一方面...」的机械排比 | -3 |
| EYE-03 | 出现「值得注意的是」「需要特别指出的是」「至关重要」等AI强调词 | -3 |
| EYE-04 | 使用「赋能」「抓手」「闭环」「链路」「颗粒度」「底层逻辑」等互联网黑话堆砌 | -3 |
| EYE-05 | 大量使用「显著」「极大」「有效」「积极」「全面」等无法量化的程度副词 | -3 |
| EYE-06 | 每个章节都有完美对称的小标题结构（四个字/六个字对仗工整） | -3 |
| EYE-07 | 出现「作为一款...的产品」「在当今...的时代」等空洞背景铺垫 | -3 |
| EYE-08 | 功能描述使用「支持灵活配置」「提供丰富的...」「全方位的...」等万能话术 | -3 |
| EYE-09 | 风险分析写「需要关注」「建议持续跟进」等正确的废话 | -3 |
| EYE-10 | 结尾出现「展望未来」「不断优化迭代」「持续为用户创造价值」等口号 | -3 |

### 10.2 可疑AI痕迹（SUS系列，每个 -2 分）

| 编号 | 模式特征 | 扣分 |
|------|---------|------|
| SUS-01 | 段落长度高度一致（每段都是3-5行，无长无短） | -2 |
| SUS-02 | 完全没有口语化表达、个人判断或吐槽 | -2 |
| SUS-03 | 没有任何具体的竞品截图、用户反馈原文、数据来源链接 | -2 |
| SUS-04 | 所有功能描述都是「应该」而非「必须」，缺乏强制语气 | -2 |
| SUS-05 | 没有对需求提出质疑或标注风险，一味正面描述 | -2 |
| SUS-06 | 使用了大量Emoji但分布均匀机械（装饰性而非表达性） | -2 |
| SUS-07 | 看不到任何「做了X调研后决定」的决策痕迹，像是凭空生成 | -2 |

### 10.3 文档结构性AI特征（DOC系列，每个 -3 分）

| 编号 | 模式特征 | 扣分 |
|------|---------|------|
| DOC-01 | 文档结构过于「教科书式」完美，像模板填充而非实际工作产物 | -3 |
| DOC-02 | 需求优先级全标P0或全部平均分布，缺乏真正的权衡取舍 | -3 |
| DOC-03 | 风险章节写得比需求章节还详细、还完美（AI最擅长写风险分析） | -3 |
| DOC-04 | 竞品分析像百度百科摘抄，没有实际体验的细节和截图 | -3 |
| DOC-05 | 数据指标全是整数或整齐的百分比（如「提升50%」「降低30%」）无来源 | -3 |
| DOC-06 | 用户画像像模板（「25-35岁白领，追求效率」）而非真实用户画像 | -3 |
| DOC-07 | 文档中没有任何手写批注、TODO、待确认标记 | -3 |

### 10.4 三眼识别：一票否决

以下任一触发，**整个PRD直接判废，总分 = 0**：

| 编号 | 一票否决条件 |
|------|------------|
| VETO-01 | PRD中的用户名/数据示例明显虚构（如「张三」「李四」「某公司」）且无真实数据补充 |
| VETO-02 | 核心业务逻辑自相矛盾（如同时要求「实时同步」和「离线可用」却无明确切换策略） |
| VETO-03 | 需求数据指标编造（如「用户增长200%」「日活100万」但无任何来源） |
| VETO-04 | 技术方案与需求目标明显不匹配（如要求毫秒级响应却选用同步阻塞架构） |
| VETO-05 | 整份文档通过AI检测工具得分 > 80% |

## 使用说明

### 审查流程

1. 通读整份 PRD，建立整体印象
2. 逐维度对照检查项，记录发现的问题
3. 特别关注**本清单之外的问题**（最重要的一步）
4. 对每个问题标注严重级别和扣分
5. 计算加权总分，给出等级评定
6. 撰写审查意见，**必须包含清单之外的发现**

### 输出格式

```markdown
## 审查结果

- **总分：** X / 100
- **等级：** S/A/B/C/D
- **一票否决：** 无 / 触发（编号）

### 清单内问题（N条）
| 维度 | 编号 | 问题描述 | 扣分 |
|------|------|---------|------|
| ... | ... | ... | ... |

### ⭐ 清单外发现（N条）
> 这部分才是审查的核心价值

1. [具体问题描述]
2. [具体问题描述]
...
```

### 扣分规则

- 🔴 严重：-4 分
- 🟡 一般：-2 分
- 🟢 轻微：-1 分
- 每维度扣完为止，不出现负数
- 加权后相加得总分
