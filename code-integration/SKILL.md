# 代码集成与冒烟验证方法论

> 本 skill 可在 agent-pipeline 流程中使用，也可独立使用。

## 角色定义

- **集成执行者：** dev3（全栈，前后端都懂，适合解决集成冲突）
- **冒烟验证执行者：** 创业助手（startup-helper）

## 流程概览

```
多Agent并发开发完成
  │
  ▼
代码合并 → 解决冲突
  │
  ▼
接口一致性验证 → 编译通过
  │
  ▼
P0冒烟验证（逐项执行）
  │
  ├─ ✅ 全部通过 → 进入审查阶段
  └─ ❌ 失败 → 打回开发者修复
```

---

## 第一步：代码合并

### 合并策略

1. **收集各Agent产出** — 确认每个开发者的代码已提交到各自分支
2. **合并到集成分支** — 按依赖顺序合并（无依赖的可并行）
3. **解决合并冲突** — 优先保留双方意图，无法判断时询问开发者

### 冲突解决规则

| 冲突类型 | 处理方式 |
|---------|---------|
| 同一文件不同区域 | 自动合并 |
| 同一函数不同修改 | 分析意图，合并两方逻辑 |
| 同一函数相同位置 | 询问对应开发者，不自行决定 |
| 配置文件冲突 | 以最新修改为准，确认不影响其他模块 |

---

## 第二步：接口一致性验证

合并后必须验证模块间接口一致：

1. **API 契约检查** — 前端调用的接口签名（URL、参数、返回值）与后端定义一致
2. **数据模型检查** — 实体字段、类型、命名在前后端一致
3. **状态管理检查** — 前端状态流转与后端数据变更一致
4. **导入完整性** — 所有跨模块引用的类/函数/类型已正确导入

---

## 第三步：编译验证

```bash
# 后端
mvn compile -q

# 前端
npx vue-tsc --noEmit  # 或 tsc --noEmit

# 全量构建
npm run build  # 或对应项目的构建命令
```

**编译不通过 = 不进入冒烟**。先修编译错误，再做冒烟。

---

## 第四步：P0 冒烟验证

### 核心原则

**必须逐项执行，不可跳过、不可合并、不可"看起来没问题"了事。**

### 冒烟验证格式

每一项必须输出（与 qa-workflow 测试报告格式统一）：

```
=== [功能模块] ===
✅/🔴 [测试项名称]
- 前置条件：[环境/数据准备]
- 执行步骤：[具体操作]
- 预期结果：[应该发生什么]
- 实际结果：[实际发生了什么]
```

### 冒烟验证范围

只验证 P0（核心流程），不验证 P1/P2：

1. **核心业务流程端到端** — 启动 → 主操作 → 结束，能走通
2. **关键 API 可调通** — 核心接口返回正确状态码和数据结构
3. **页面可加载** — 主要页面无白屏、无控制台致命错误
4. **数据可流转** — 创建/查询/更新/删除基本操作正常

### 冒烟结果处理

- ✅ **全部通过** → 进入阶段6.5开发审查
- ❌ **任一项失败** → 打回对应开发者修复，禁止进入审查环节

---

## 产出

- **集成报告：** `.contracts/{project}/integration-report.md`
  - 合并概况（参与Agent、文件数、冲突数）
  - 接口一致性检查结果
  - 编译结果
  - 冒烟验证逐项结果
  - 通过/失败判定
