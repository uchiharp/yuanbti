# qa-review-workflow — QA 测试质量审查

## 是什么

审查测试代码质量的完整工作流：9维度评分标准 + Playwright 实操审查流程。

**本 skill 无任何外部依赖，可独立使用。** 不依赖 agent-pipeline、agent-discipline 或任何其他 skill。

## 触发条件

| 场景 | 谁触发 | 说明 |
|------|--------|------|
| 收到测试审查任务 | QA评审官 / main agent | 按流程审查 |
| 流水线阶段5/6/8 | 协调者 | 审查测试计划/用例/报告 |
| 流水线阶段7 | 协调者 | Playwright 实操审查 |
| 用户要求审查测试 | main agent | 不走流水线直接用 |

## 执行流程

### 纯代码审查（测试计划/用例/报告）

```
1. 读取被审查的测试代码或报告
2. 按 9 维度逐项评分（每维度必须给分数+扣分原因）
3. 去AI味检查（第9维度 + 一眼AI识别）
4. 输出审查报告
5. 签收（≥8分无🔴）或打回（附具体问题）
```

### Playwright 实操审查

```
1. 确认应用运行（curl localhost:3000）
2. 执行检查清单：页面加载/导航/表单/响应式/控制台错误
3. 每页截图，问题截图留证
4. 输出审查报告
5. 有🔴问题 → 通知修复 → 最多3轮循环
```

## 约束

1. 必须真实执行（Playwright 场景真跑，不能只看代码推断）
2. 每个维度必须评分（不允许模糊评价）
3. 每条扣分必须引用 文件名:行号 + 代码片段
4. 报告 ≥30行，不足自动打回
5. Playwright 场景每页至少1张截图
6. 一轮最多3次循环，超过升级

## 评分规则

- 起点7分，扣分制
- ≥8分且无🔴 → 签收
- <8分或存在🔴 → 打回
- 一票否决：核心逻辑零覆盖 / 测全绿但功能有bug

```markdown

| 边界覆盖(10%) | X/10 | 🔴X 🟡X |

| 可维护性(10%) | X/10 | 🔴X 🟡X |
| 独立性(10%) | X/10 | 🔴X 🟡X |

```
