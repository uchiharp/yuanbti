# Code Quality Guard — 代码质量守护

> 本 skill 是**预防性检查清单**，不是测试执行工具。
> 测试编写和执行 → 见 `qa-workflow`
> 系统化调试 → 见 `qa-workflow` Part B
> 项目测试执行 → 见 `run-tests`

## 核心原则

代码必须满足两个条件：
1. ✅ **能够编译通过** — 无语法错误、类型错误、依赖问题
2. ✅ **业务逻辑正确** — 实现预期功能、处理边界情况、异常流程

## 🔍 编码检查清单（写代码时用）

### 编译相关

- [ ] 包声明正确且在第一行
- [ ] 所有 import 的类存在，无未使用的导入
- [ ] 方法返回类型匹配
- [ ] 参数类型正确，泛型类型一致
- [ ] `@Autowired` / `@RequiredArgsConstructor` 正确，final 字段都已注入
- [ ] 循环依赖检查
- [ ] 括号匹配、分号完整、花括号成对、字符串闭合

### 业务逻辑相关

- [ ] **数据一致性：** 事务边界正确、并发安全、外键关系维护
- [ ] **状态流转：** 状态转换合法、无非法状态、状态机完整
- [ ] **边界条件：** null 值、空集合、数值边界（0/负数/最大值）、字符串边界（空串/超长）
- [ ] **异常处理：** 捕获正确的异常、信息清晰、记录日志、必要时重试
- [ ] **权限验证：** 身份验证、资源归属、操作权限、防越权

## 🚨 常见问题模式速查

### 编译问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 缺失字段 | 使用不存在的字段 | 先在实体类中添加 |
| 方法签名不匹配 | 调用不存在的方法 | 确认正确方法名 |
| 依赖未注入 | 字段为 null | 添加 `final` + 构造器注入 |
| 包不存在 | 文件路径和 package 声明不一致 | 对齐路径和声明 |

### 业务逻辑问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 事务边界错误 | readOnly 中执行写操作 | 去掉 readOnly 或拆分方法 |
| 竞态条件 | 先读后写非原子操作 | 用原子 SQL 或 `@Version` 乐观锁 |
| 资源泄露 | 未关闭流/连接 | try-with-resources |
| 状态遗漏 | 新状态没有对应转换规则 | 补充状态机 |
| 空值穿透 | null 没有在入口拦截 | 参数校验 + `@Nullable` 注解 |

### 前端防抖/节流问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 重复提交 | 用户连点提交按钮产生多条记录 | 点击后立即 `disabled` + debounce 300ms |
| 搜索风暴 | 每次 keyup 都发请求 | `debounce(500)` 输入停止后才发 |
| 滚动性能 | scroll 事件触发频率过高 | `throttle(200)` 限制触发频率 |
| resize 重排 | 窗口缩放频繁触发布局计算 | `throttle(300)` + `requestAnimationFrame` |
| 定时器泄漏 | 组件卸载后 timer 仍在执行 | `onUnmounted(() => clearTimeout(...))` |

### 并发控制问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 重复下单 | 用户快速点击创建多笔订单 | 幂等键（前端 UUID）+ 数据库唯一约束 |
| 库存超卖 | 并发扣减导致负库存 | Redis `DECR` 原子操作或分布式锁 |
| 状态并发覆盖 | 两人同时修改同一记录 | `UPDATE ... WHERE version=?` 乐观锁 |
| 定时任务重复执行 | 多实例同时触发定时任务 | Redis `SET NX EX` 分布式锁 |
| SELECT-then-INSERT 竞态 | 检查不存在后插入，但并发时都通过 | 数据库唯一约束 + `INSERT IGNORE` 或 `ON DUPLICATE KEY` |

### 边界条件问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 空集合 NPE | 遍历 null 集合 | 返回 `Collections.emptyList()`，不用 null |
| 分页越界 | page=0 或 page=-1 | `page = Math.max(1, page)` |
| 字符串截断 | 输入超长导致数据库报错 | 前端计数 + 后端 `@Size(max=255)` |
| 金额精度丢失 | double 运算产生 0.3000000004 | 全程 `BigDecimal`，禁止 double |
| 日期时区错乱 | 不同环境时区不一致 | 统一 `Asia/Shanghai`，数据库存 UTC |
| 数值溢出 | ID 超过 int 范围 | 前后端统一 Long/string 处理大 ID |

### 超时配置问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 无限等待 | 外部调用没设超时，线程阻塞 | `connectTimeout=3s, readTimeout=10s` |
| 连接池耗尽 | DB 连接泄漏未归还 | HikariCP `connectionTimeout=3s` + `leakDetectionThreshold` |
| Redis 阻塞 | 慢命令阻塞整个连接池 | 设置 `timeout=2s`，避免 `KEYS *` |

### 缓存问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 缓存穿透 | 查不存在的数据每次打 DB | 缓存空值 TTL=60s 或布隆过滤器 |
| 缓存雪崩 | 大量 key 同时过期 | TTL + random(0,300s) 偏移 |
| 缓存击穿 | 热 key 过期并发穿透 | `SETNX` 互斥锁或逻辑过期 |
| 双写不一致 | 先更新缓存再更新 DB | Cache Aside：先 DB 后删缓存 |

### 限流/文件上传/降级问题

| 模式 | 问题 | 修复 |
|------|------|------|
| 无限流 | API 被爬虫打挂 | 令牌桶限流，超限 429 |
| 文件路径穿越 | 用户上传文件名含 `../` | 重命名为 UUID，禁止原始文件名 |
| 文件类型伪造 | 只看后缀名 | 校验 content-type + magic bytes |
| 无限重试 | 网络异常无限重试 | 指数退避 + 最大 3-5 次 + 死信队列 |
| 密码明文 | 密码 MD5 或明文存储 | bcrypt 加密存储 |

## 📋 代码审查清单（Review 时用）

### 编译层面
- [ ] 无语法/类型/依赖错误
- [ ] Maven/Gradle 编译成功

### 业务逻辑层面
- [ ] 功能完整实现
- [ ] 边界条件处理
- [ ] 异常流程处理
- [ ] 数据一致性保证
- [ ] 权限验证完整
- [ ] 并发安全

### 代码质量层面
- [ ] 命名清晰（见名知义）
- [ ] 注释合理（业务导向，不是翻译代码）
- [ ] 结构清晰（单一职责）
- [ ] 无重复代码（DRY）
- [ ] 性能可接受（无 N+1、无大循环内 IO）

## 📐 代码规范（人类常用）

### 命名规范

| 类型 | 规则 | 好的例子 | 差的例子 |
|------|------|---------|---------|
| 类名 | 大驼峰，名词 | `UserService`, `OrderItem` | `UserProcess`, `DoOrder` |
| 方法名 | 小驼峰，动词开头 | `findByName`, `calculateTotal` | `userList`, `total` |
| 变量 | 小驼峰，有意义 | `retryCount`, `maxFileSize` | `i`, `temp`, `data2` |
| 常量 | 全大写下划线 | `MAX_RETRY_COUNT` | `maxRetry`, `MaxRetry` |
| 布尔值 | is/has/can/should 开头 | `isEnabled`, `hasPermission` | `enable`, `permission` |
| 集合 | 复数名词 | `users`, `orderItems` | `userList`, `orderItemArray` |
| 包名 | 全小写，点分隔 | `com.film.auth.user` | `com.film.Auth.User` |

**核心原则：** 看到名字就知道它是什么、做什么。不确定就写长一点，别缩写。

### 代码格式

| 规则 | 说明 |
|------|------|
| 缩进 | 项目用什么就用什么（2空格/4空格/Tab），全项目统一 |
| 行宽 | ≤120 字符，超了换行 |
| 空行 | 逻辑段落之间 1 个空行，方法之间 1 个空行，不超过 2 个 |
| 括号 | 左括号不换行（Java 风格）或换行（JS 风格），全项目统一 |
| import | 不用 `*`，按需导入，删除未使用的 |
| 链式调用 | 每个方法调用一行，便于调试和断点 |

### 注释规范

| 场景 | 做法 |
|------|------|
| 方法名已经清晰 | 不写注释 |
| 复杂业务逻辑 | 写**为什么**，不写**做了什么** |
| 非直觉的 hack | 必须写原因和 TODO |
| 公开 API | 简短说明参数和返回值 |
| 临时注释 | 标记 `// TODO:` 或 `// FIXME:` 并附日期 |

**禁止：**
- ❌ `// 获取用户信息`（方法名已经说了）
- ❌ `// 循环遍历列表`（代码本身清晰）
- ❌ 注释和代码不一致（注释说是A，代码做的是B）

### Git 提交规范

格式：`<type>(<scope>): <subject>`

| type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(auth): 添加JWT登录` |
| `fix` | 修复 | `fix(user): 修复手机号校验` |
| `refactor` | 重构 | `refactor(service): 抽取公共方法` |
| `test` | 测试 | `test(auth): 补充登录单元测试` |
| `docs` | 文档 | `docs: 更新API文档` |
| `chore` | 杂项 | `chore: 升级依赖版本` |

**规则：**
- subject ≤50 字符，中文或英文均可
- 不以句号结尾
- 有 body 时空一行再写（说明为什么改，不是改了什么）

## 📊 质量评分

- **编译质量：** 0 或 10（无错误=10，有错误=0）
- **业务逻辑质量：** 0-10（功能完整=10，有缺陷=5，错误=0）
- **最低要求：** 编译 10 + 逻辑 ≥ 8
