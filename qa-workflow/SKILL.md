# QA Workflow — 统一测试与调试工作流

## 三条铁律

### 铁律1：单元通过 ≠ 功能正常
Mock环境测过不代表真实数据库/网络下正常。三层测试缺一不可：
- **单元测试**（Mock）→ 逻辑分支
- **集成测试**（真实DB/缓存）→ 数据一致性
- **E2E测试**（浏览器）→ 完整用户体验

报告时必须分开说明测试层级和覆盖范围。

### 铁律2：不找到根因不修代码
```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```
随机修bug浪费时间并引入新bug。症状修复等于失败。

### 铁律3：测试被跳过 = 没有测试
不允许静默跳过测试。如必须跳过，注释中写明：原因、负责人、计划修复日期。

## Part A：测试编写流程

```
源码分析 → 功能清单 → 用例设计 → 脚本编写 → 独立验证 → 测试报告
```

### A1. 功能分析

1. 读前端页面文件 → 提取交互事件（click/tap/input/confirm等）
2. 读后端 Controller → 提取 API 端点
3. 读状态管理（store/state） → 提取状态流转
4. 读 API 封装层 → 提取前端调用的接口

**输出格式：**
```markdown
### 模块: xxx

| # | 用例ID | 功能点 | 测试类型 | 前置条件 |
|---|--------|--------|---------|---------|
| 1 | XX-01 | 描述 | 正向/逆向/边界/安全/界面/数据流转/跨模块 | xxx |
```

**测试类型：**

| 类型 | 说明 |
|------|------|
| 正向 | 正常操作流程 |
| 逆向 | 错误操作/异常输入（null、缺失、格式错） |
| 边界 | 极端值/空状态/新用户首次/大数据量 |
| 安全 | 越权/XSS/注入/未认证访问 |
| 界面 | 加载中/空/错误/成功 各状态 |
| 数据流转 | 前后端一致性、跨模块联动 |
| 跨模块 | 多页面/多功能的端到端联动 |

**遗漏检查清单：**
- [ ] 每个交互事件都有测试
- [ ] 每个表单验证都有逆向用例（含 null）
- [ ] 每个 API 都有成功 + 失败 + 超时
- [ ] 空状态 / 加载中 / 错误状态有 UI 测试
- [ ] 删除操作有确认弹窗
- [ ] 文件上传有格式 + 大小 + null 测试
- [ ] 认证过期 / 无权限有安全测试
- [ ] 跨模块联动有端到端流程测试

### A2. 用例设计

**测试层级（对每个功能点）：**
1. **功能层** — 正向 + 逆向 + 边界 + 数据流转
2. **界面层** — 各种 UI 状态
3. **接口层** — HTTP 请求/响应
4. **非功能层** — 性能 + 安全
5. **回归层** — 修改后重跑受影响模块
6. **用户旅程层** — 完整业务流程（最重要）

**用户旅程测试（最高优先级）：**
必测旅程：
- **新用户首次使用**: 注册/登录 → 空状态 → 添加第一个数据 → 看到结果
- **核心业务流**: 完整的主线操作链路
- **数据管理**: 创建 → 编辑 → 删除 → 验证统计更新
- **异常恢复**: 网络断开 → 恢复 → 数据不丢失；认证过期 → 重新登录 → 回到之前页面

**用例编号规则：** `{模块缩写}-{序号}`，如 `L-01`, `JRN-01`

### A3. 脚本编写

**通用原则：**
1. **先读源码确认选择器/URL/字段名**，再写测试
2. 断言要具体（`assertEquals(201, status)`），不要空断言（`assertNotNull`）
3. 用测试框架的自动等待机制，避免硬编码 `sleep` / `waitForTimeout`
4. 每个关键步骤截图，方便排查

**单元测试：**
- Mock 外部依赖（数据库、缓存、外部 API）
- 每个测试类至少 3 个方法：正常 + 异常 + 边界

**集成测试：**
- 连真实数据库和缓存
- 验证数据库中的实际数据，不只是 API 返回值

**E2E 测试：**
- 先读组件确认 class 名和 URL
- 公共操作（登录、导航）抽成辅助函数
- AI/外部 API 用例标记 `[SEQUENTIAL]`

**截图规范：** `{功能}-{序号}-{状态}.png`

### A4. 执行策略

```
第一轮（并行）: 单元测试
第二轮（并行）: 集成测试（需要真实 DB/缓存）
第三轮（并行）: E2E 非 AI 测试
第四轮（串行）: E2E 外部 API 测试（间隔 30s）
```

**只跑受影响的用例：**
修改后不全量跑，只跑：直接修改的模块 + 依赖该模块的下游 + 涉及该模块的用户旅程。

### A5. 测试报告

格式见 [references/report-template.md](references/report-template.md)。

**报告铁律：**
1. 说明测试范围和环境：不能把 Mock 测试写成"全部通过"
2. 分层级报告：单元/集成/E2E 分别列出
3. 失败必须附真实错误日志
4. E2E 用例必须有截图
5. 跳过的用例必须说明原因和修复计划

### A6. 代码修改后自动验证

每次代码变更后自动执行验证流程：

**第一阶段：静态检查（不需要服务运行）**
1. 前端编译检查：`npx vue-tsc --noEmit`
2. 后端编译检查：`mvn compile -q`
3. Import 完整性扫描：检查使用的 API 是否已导入
4. 跨端兼容性检查：扫描 `fetch`/`document`/`window`/`localStorage`（跨端不兼容）
5. 页面/路由完整性检查

**第二阶段：API 冒烟测试（需要后端运行）**
- 未授权访问 → 应返回 401/403
- 参数错误 → 应返回 400
- 核心接口 → 应返回 200 和正确 JSON 结构

**第三阶段：后端测试**
```bash
mvn test -q 2>&1 | tail -10
```

**第四阶段：Playwright E2E（需要前后端都运行）**
```bash
npx playwright test --reporter=list --timeout=30000 2>&1
```

**E2E注意事项：**

| 陷阱 | 说明 |
|------|------|
| 登录注入 | 必须注入完整会话数据（token、过期时间、用户信息等） |
| Tab页导航 | 框架Tab页不能用`page.goto()`直接跳转，需要用框架API |
| 测试数据 | 用`beforeAll`通过API创建真实数据，不要硬编码ID |
| AI/外部API | 串行跑，避免触发限流 |
| 等待渲染 | 用`waitForLoadState('networkidle')` + `waitForTimeout` |

**验证输出格式：**
```
=== QA Report ===

🔴 BLOCKER: [必须修复，阻塞使用]
   - 文件:行号: 具体问题

🟡 WARNING: [应该修复，不影响基本功能]
   - 文件:行号: 具体问题

✅ PASS: [验证通过]
   - 具体项

总计: X BLOCKER, Y WARNING, Z PASS
结论: ✅ 可发布 / ❌ 需修复
```

## Part B：系统化调试

### 何时使用

任何技术问题：
- 测试失败、生产 Bug、异常行为
- 性能问题、构建失败、集成问题

**尤其适用于：** 时间压力下"快速修一下"看似明显时、之前修复没效果时。

### 四阶段流程

```
根因调查 → 模式分析 → 假设验证 → 修复确认
```

#### B1. 根因调查（修之前必须完成）

1. **仔细读错误信息**
   - 不要跳过 warning，完整读 stack trace
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？精确步骤是什么？
   - 不能复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - `git diff`、最近提交、新依赖、配置变更

4. **多组件系统中的证据收集**
   ```
   对每个组件边界：
     - 记录输入数据
     - 记录输出数据
     - 验证配置/环境传播
   跑一次看哪里断裂，再深入调查那层
   ```

5. **追踪数据流**
   - 坏值从哪来？谁调用了它？
   - 沿调用栈向上追溯到源头
   - **在源头修复，不在症状处修补**

#### B2. 模式分析（修之前找规律）

1. 在代码库中找类似的工作代码
2. 与参考代码对比 — 完整读，不要扫
3. 列出工作代码和坏代码之间的所有差异
4. 理解所有依赖和假设

#### B3. 假设 + 测试（一次一个）

1. 形成一个清晰假设："我认为 X 是根因，因为 Y"
2. **写下来**
3. 设计一个最小测试来证明/证伪
4. 运行测试
5. 证伪 → 新假设，重复
6. 证实 → 进入修复

#### B4. 修复 + 验证

1. 在根因处修复，不在症状处
2. 写一个能捕获这个 bug 的测试
3. 验证修复有效
4. 验证无回归（跑全量测试）
5. 提交时写清根因

### 调试反模式

| 反模式 | 为什么错 |
|--------|---------|
| "试试这个" | 随机修复没有根因 |
| "应该是X吧" | 猜测没有证据 |
| "先打个补丁" | 症状修复留下根因 |
| 同时改多个地方 | 无法知道哪个生效了 |
| 跳过复现步骤 | 调试无法稳定触发的问题 |

### 调试触发条件

当出现以下情况时，**必须**走完整的四阶段流程：
- 同一个 bug 修了 2 次以上还在
- 修了 A 却导致 B 坏了
- 测试在本地通过但 CI 失败（或反过来）
- 无法稳定复现的问题
