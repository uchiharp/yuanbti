# 需求分析方法论

> 本 skill 可在 agent-pipeline 流程中使用，也可独立使用。

## 角色定义

- **执行者：** 需求分析师（PM）
- **评审官：** 需求评审官（魔鬼代言人思维，挑战需求真实性和合理性）
- **交叉评审方：** 架构师、QA、开发、UI设计师、UX设计师

## 流程概览

```
用户需求
  │
  ▼
需求分析师梳理需求 → 产出 PRD
  │
  ▼
需求评审官审查（最多2轮迭代 → 签收）
  │
  ▼
交叉评审（架构师/QA/开发/UI/UX 并发，24h时限）
  │
  ▼
需求分析师修订 → 评审官快速复核（1轮）→ 最终签收
```

---

## 第一步：产出 PRD

需求分析师应基于用户需求，产出完整的 PRD 文档：

```markdown
## PRD：{功能名称}

### 1. 背景与目标
为什么要做这个？解决什么问题？

### 2. 用户故事
作为{角色}，我想要{功能}，以便{价值}。

### 3. 功能清单
| # | 功能 | 类型 | 优先级 | 验收标准 |
|---|------|------|--------|---------|
| 1 | 用户注册 | must-have | P0 | 邮箱+密码，收到验证邮件 |
| 2 | 第三方登录 | nice-to-have | P1 | 支持微信登录 |
| 3 | 找回密码 | must-have | P0 | 邮箱验证码重置 |

### 4. 非功能需求
- 性能：页面加载<2s
- 安全：密码加密存储
- 兼容：iOS/Android/PC

### 5. 不做什么
- 不做社交分享（V2考虑）
- 不做消息推送（V2考虑）
```

---

## 第二步：需求评审官审查

需求评审官从以下维度审查 PRD：
- 需求是否真实存在（不是我们觉得用户需要）
- MVP 范围是否够小（能不能砍掉一半还解决问题）
- 验收标准是否可量化
- 是否有需求蔓延（nice-to-have 混进 must-have）
- "不做什么"是否明确

最多2轮迭代，签收后进入交叉评审。

---

## 第三步：交叉评审

PRD 影响所有下游角色。评审官从"需求真实性"角度审查，但无法替代下游角色的专业视角。在架构设计和开发之前发现 PRD 问题，改造成本是最低的。

**模式：** 并发评审，不做迭代，只提意见
**时限：** 24h — 各角色有24h窗口期提交评审意见，超时视为无异议

### 各角色评审维度

| 角色 | 评审视角 | 典型问题 |
|------|---------|---------|
| 架构师 | 技术可行性 | 这个需求技术上能实现吗？有没有性能隐患？是否需要引入新中间件？ |
| QA | 可测试性 | 验收标准够具体吗？我能据此写测试用例吗？异常场景覆盖了吗？ |
| 开发 | 实现复杂度 | 工作量合理吗？有遗漏的边界情况吗？接口设计有矛盾吗？ |
| UI设计师 | 视觉可行性 | 页面复杂度合理吗？需要自定义组件吗？设计上有没有矛盾？ |
| UX设计师 | 交互完整性 | 用户流程完整吗？有死路吗？错误恢复路径有吗？ |

### 评审输出格式（每个角色）

```markdown
## {角色}对PRD的评审意见

### ✅ 清晰明确的部分
- {无异议的功能/标准}

### 🟡 建议澄清
| PRD条目 | 问题 | 建议修改 |
|----------|------|----------|
| 功能#3 | "实时通知"不够具体 | 定义"实时"是轮询还是WebSocket |

### 🔴 认为有问题
| PRD条目 | 问题 | 原因 |
|----------|------|------|
| 功能#5 | 与功能#2的权限模型矛盾 | 用户既是"管理员"又是"普通用户"时逻辑不清 |

### ➕ PRD遗漏
- {PRD没提到但该角色认为需要的功能/约束}
```

### 汇总处理

1. 收集所有角色的评审意见
2. 去重（多个角色提了同一个问题）
3. 分类：必须修改（🔴）、建议修改（🟡）、记录备查（➕）
4. 交给需求分析师修订 PRD
5. 需求评审官快速复核修订版（1轮，确认没引入新问题）
6. 最终签收

---

## PRD 验收标准

需求分析师交付 PRD 时，必须满足以下标准：

- [ ] 覆盖所有用户故事（每个核心功能有对应的用户故事）
- [ ] 优先级标注（P0/P1/P2）
- [ ] 非功能性需求（性能、安全、兼容性）
- [ ] 术语表（避免歧义）
- [ ] 行数 ≥ 100行

---

## 防空签收规则

签收报告必须包含：
- 至少3个具体检查过的点（写明章节号/功能点）
- 至少1个🟢建议或观察（即使觉得完美也要写）
- 各维度评分（不能只给总分）

打回报告必须包含：
- 每个问题的具体位置（章节号/条目号）
- 具体的修改建议（不能只说"需要修改"）
- 预期修改后的效果

**以下视为空签收（不合格）：**
- ❌ "看起来没问题，签收"
- ❌ "PRD写得很好，通过"
- ❌ "大家都没有意见"
