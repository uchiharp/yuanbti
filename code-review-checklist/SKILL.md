# Code Review Checklist — 代码审查清单

你是代码审查者（可以是开发互审、架构师审查、QA审查等任何角色）。你的职责是发现代码中的问题，确保质量。

**审查原则：**
1. 逐文件审，不要跳过
2. 发现问题必须指出具体文件和行号
3. 区分"必须修改"和"建议修改"
4. 不要只说"有问题"，要说清问题是什么、怎么修

---

## A. 代码质量检查

### A1. 编译与基础

- [ ] 包声明/模块导入是否正确完整
- [ ] 类型是否匹配（参数、返回值、泛型）
- [ ] 依赖注入是否完整（无未注入的依赖）
- [ ] 注解使用是否正确（框架特定注解）
- [ ] 语法：括号匹配、分号、花括号成对

### A2. 架构对齐

- [ ] 代码实现是否符合架构设计方案（技术选型、分层、接口定义）
- [ ] 接口定义是否和设计文档一致
- [ ] 新增的依赖是否在架构方案允许范围内

### A2.1 分层架构检查

- [ ] Controller 是否直接调用了 Repository/Mapper（应通过 Service）
- [ ] Service 是否依赖了 HttpServletRequest/Response（框架对象）
- [ ] Repository 是否包含业务逻辑（应只做 CRUD）
- [ ] 各层之间是否通过 DTO/VO 传递（不直接暴露数据库实体）
- [ ] 模块间是否通过接口（Interface）通信（不直接引用实现类）

**检查方法：**
```bash
# Controller 是否直接 import Repository
grep -r "import.*Repository\|import.*Mapper" src/**/controller/
# Service 是否 import HttpServletRequest
grep -r "import.*HttpServletRequest\|import.*HttpServletResponse" src/**/service/
# 模块间是否直接引用 ServiceImpl
grep -r "import.*ServiceImpl" src/ | grep -v "import.*self"
```

### A2.2 模块解耦检查

- [ ] 每个业务模块是否独立目录（不跨模块引用内部类）
- [ ] 模块间是否通过接口通信（注入 Interface 而非 Impl）
- [ ] 是否有循环依赖（A 模块依赖 B，B 又依赖 A）
- [ ] 公共组件是否抽取到 common/ 目录（不在各模块重复实现）

### A2.3 可复用组件检查

- [ ] 认证逻辑是否抽取为过滤器（非每个 Controller 重复检查）
- [ ] 权限检查是否使用注解（非硬编码 if/else）
- [ ] API 响应是否使用统一 Result 包装
- [ ] 异常处理是否全局统一（非每个方法 try-catch）
- [ ] 审计日志是否通过切面自动记录（非手动调用）

### A2.4 扩展性检查

- [ ] 新增业务系统接入是否只需改配置/加注解（不改核心代码）
- [ ] 新增权限是否只需注册编码（不改权限检查逻辑）
- [ ] 新增角色是否只需在管理后台创建（不改代码）

### A2.5 SOLID 原则检查

- [ ] **S** 单一职责：一个类只做一件事（Service 不混杂缓存逻辑、Controller 不写业务计算）
- [ ] **O** 开闭原则：新增功能是否通过扩展而非修改现有代码（加策略/加事件 vs 改 if-else）
- [ ] **L** 里氏替换：子类能否安全替换父类（不覆写父类契约行为）
- [ ] **I** 接口隔离：接口是否过胖（一个接口 10+ 方法 → 拆分）
- [ ] **D** 依赖反转：上层是否依赖抽象而非实现（注入 Interface 而非 Impl）

### A2.6 设计模式使用检查

- [ ] 架构方案中标注的模式是否正确使用（如标注用 Strategy 实际写成 if-else）
- [ ] 模式使用是否过度（简单场景套复杂模式）
- [ ] Observer/Event 是否有循环触发风险
- [ ] Chain of Responsibility 是否有终止条件

### A3. 逻辑正确性

- [ ] 业务逻辑是否完整实现（对照PRD/任务描述）
- [ ] 是否有遗漏的分支（if/else、switch、异常路径）
- [ ] 状态流转是否正确（状态机完整、无非法跳转）
- [ ] 数据计算是否准确（边界值、精度、类型转换）

### A4. 可维护性

- [ ] 代码是否可读（命名清晰、结构简洁、不过度嵌套）
- [ ] 是否有重复代码可以抽取公共方法
- [ ] 单个方法/文件是否过长（超过100行考虑拆分）
- [ ] 过度耦合：是否依赖了不该依赖的模块

### A5. 资源与异常

- [ ] 文件/连接/流是否正确关闭（try-with-resources / finally）
- [ ] 异常是否被正确捕获和处理（不吞异常、日志记录）
- [ ] 空值/空集合是否在入口处拦截
- [ ] 事务边界是否正确（readOnly vs 读写）

## B. 安全扫描

### B1. 输入安全

| 检查项 | 怎么查 | 常见漏洞 |
|--------|--------|---------|
| XSS | 搜索用户输入是否直接渲染到页面 | `v-html`、`innerHTML`、模板拼接 |
| SQL注入 | 搜索字符串拼接SQL | `"SELECT * FROM users WHERE name='" + name + "'"` |
| 命令注入 | 搜索 `exec`/`Runtime.exec`/`os.system` | 用户输入拼接到系统命令 |
| 路径遍历 | 搜索文件操作是否使用了用户输入 | `../../../etc/passwd` |

### B2. 认证与授权

| 检查项 | 怎么查 |
|--------|--------|
| 未认证访问 | 敏感接口是否检查了登录状态 |
| 水平越权 | 访问资源时是否验证了资源归属（`userId` 匹配）|
| 垂直越权 | 普通用户是否能调用管理员接口 |
| Token 安全 | Token 是否有过期时间、是否能被伪造 |
| 权限粒度 | 操作权限是否按最小权限原则 |

### B3. 数据安全

| 检查项 | 怎么查 |
|--------|--------|
| 敏感信息泄露 | 错误信息/日志/响应中是否暴露内部实现 |
| 密码存储 | 是否明文存储密码（应 bcrypt/argon2）|
| API响应 | 是否返回了不该返回的字段（如完整密码hash）|
| 日志安全 | 日志中是否记录了敏感数据（token、密码）|

## C. 健壮性检查

### C1. 异常输入

| 场景 | 检查什么 |
|------|---------|
| null 值 | 必填参数是否校验了 null |
| 空集合 | 批量操作时空集合是否处理 |
| 超长输入 | 字符串字段是否有长度限制 |
| 特殊字符 | 搜索/正则是否处理了特殊字符 |
| 负数/零 | 数值参数是否校验了范围 |
| 非法格式 | 日期、邮箱、手机号等是否校验格式 |
| 类型错误 | 接口是否校验了参数类型 |

### C2. 并发安全

| 检查项 | 怎么查 |
|--------|--------|
| 竞态条件 | 读-改-写操作是否加了锁/乐观锁 |
| 数据库事务 | 并发修改同一条记录时是否安全 |
| 幂等性 | 重复请求是否会导致重复写入 |
| 死锁风险 | 多个锁的获取顺序是否一致 |

### C3. 资源管理

| 检查项 | 怎么查 |
|--------|--------|
| 连接泄露 | 数据库/HTTP连接是否在 finally/try-with-resources 关闭 |
| 内存溢出 | 大集合/大文件是否分页/流式处理 |
| 线程泄露 | 异步任务是否有超时和取消机制 |

### C4. 降级处理

| 检查项 | 怎么查 |
|--------|--------|
| 外部服务不可用 | 调用外部API是否有 fallback/重试/超时 |
| 数据库不可用 | 是否有降级策略（缓存、默认值）|
| 网络超时 | HTTP请求是否设置了超时时间 |

## D. 去AI味检查

### D1. 注释

| AI味特征 | 人类写法 |
|---------|---------|
| `// 获取用户信息` | 删除（方法名已经说明）或写业务背景 |
| `// 处理异常` | 写明为什么在这里捕获，业务原因是什么 |
| `// TODO: 优化性能` | 写明具体要优化什么，当前瓶颈在哪 |
| `// 创建新订单` | 删除（代码本身清晰）或补充业务规则说明 |

### D2. 命名

| AI味特征 | 人类写法 |
|---------|---------|
| `processUserData` | `validateProfile` / `syncUserPrefs`（具体化）|
| `handleRequest` | `acceptInvitation` / `rejectProposal`（业务化）|
| `DataManager` | `PlaylistStore` / `ConfigCache`（领域化）|
| `result` / `data` / `item` | 用业务名替代 |

### D3. 代码结构

| AI味特征 | 人类写法 |
|---------|---------|
| 每个方法都有完整的 Javadoc | 只在复杂/公开方法写注释 |
| 过度的空行分隔 | 逻辑段落间一个空行够了 |
| 所有异常都 try-catch 并 log | 只捕获预期异常，让意外的冒泡 |
| 千篇一律的 if-else-null 检查 | 用 Optional / 断言 / 早期返回 |

### D4. 异常处理

| AI味特征 | 人类写法 |
|---------|---------|
| `catch (Exception e) { log.error("error", e); }` | 捕获具体异常，针对性处理 |
| 每层都吞异常返回空值 | 在合适的层统一处理 |
| 错误信息 `"操作失败"` | `"余额不足，当前100，需要200"` |

```markdown
| 文件:行号 | 维度 | 问题 | 建议修复 |
|-----------|------|------|---------|
| src/service/X.java:42 | 安全 | SQL拼接存在注入风险 | 改用参数化查询 |

### 🟡 建议修改
| 文件:行号 | 维度 | 问题 | 建议 |
|-----------|------|------|------|
| src/service/X.java:88 | 健壮性 | 未处理空列表 | 添加 isEmpty() 检查 |

### 🟢 观察记录
- {非问题但值得注意的点}

### 📊 统计
- 审查文件数：{N}
- 必须修改：{N}（安全{N} / 健壮性{N} / 质量{N} / AI味{N}）
- 建议修改：{N}
- 总体评分：{X}/10
```
