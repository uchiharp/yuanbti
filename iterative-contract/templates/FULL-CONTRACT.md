# Agent 迭代合同协议 v1

## 核心理念

传统 agent 协作是"做完→扔过去→下一个做→再扔过去"，没有结构化的交接和质量保证。

**迭代合同 = 每个 agent 产出一个可验证的交付物，接收方验证后签收或打回，循环直到质量达标。**

不是一次性跑完所有步骤，而是**每一步都有验证、确认、和反馈权**。

---

## 合同结构

每次 agent 之间的协作，都必须使用以下合同格式：

```markdown
## 📋 迭代合同 #{轮次}

### 发包方：{Agent名称}
### 承包方：{Agent名称}
### 创建时间：YYYY-MM-DD HH:mm
### 合同类型：{开发/审查/测试/修改}

---

### 一、交付物清单

| # | 交付物 | 类型 | 路径 | 验证标准 |
|---|--------|------|------|---------|
| 1 | {文件/功能名} | 代码/文档/设计 | {绝对路径} | {具体可验证的条件} |

### 二、假设与约束

本交付基于以下假设：
- {假设1}
- {假设2}

如果假设不成立，本合同作废，需要重新协商。

### 三、验证要求

承包方必须检查：
- [ ] {验证项1}
- [ ] {验证项2}
- [ ] 自由发挥：发现清单之外的问题

### 四、质量门槛

本迭代合同的通过门槛：**{具体标准}**

| 指标 | 门槛值 |
|------|--------|
| 🔴 阻断问题 | 0 |
| 🟡 重要问题 | ≤ 2 |
| 总评分 | ≥ 7.0 |
| 去AI味评分 | ≥ 6.0 |

### 五、合同状态

⏳ **待签收** — 承包方尚未验证

---

### 签收记录

**承包方签收：**
- 时间：
- 结果：✅ 签收 / ❌ 打回
- 发现问题：
- 打回原因：
```

## 合同生命周期

```
发包方产出合同 → 承包方验证 → 签收 or 打回
                                       ↓ 打回
                              发包方修改 → 重新发合同
                                       ↓ 签收
                              进入下一迭代 or 合同终止
```

### 状态流转

```
⏳ 待签收 → ✅ 已签收（进入下一阶段）
         → ❌ 已打回（返回发包方修改）
         → ❌ 作废（假设不成立，重新协商）
         → ⏹️ 已终止（质量达标，流程结束）
```

### 终止条件

**合同在以下情况终止：**
1. 承包方签收 + 质量门槛全部达标
2. 迭代次数达到上限（默认3轮）
3. 发包方和承包方双方确认无需继续

**迭代次数上限：**
| 合同类型 | 最大迭代数 | 超限处理 |
|---------|----------|---------|
| 代码→审查 | 3轮 | 升级到协调者仲裁 |
| 审查→修改 | 3轮 | 升级到协调者仲裁 |
| 修改→测试 | 3轮 | 记录未通过项，继续前进 |
| 测试→修改 | 3轮 | 标记为已知问题 |

## 标准协作流程

### 流程 A：代码开发 + 评审审查 + 测试

```
迭代 1：开发 Agent → 产出代码 → 发合同给评审官
迭代 2：评审官 → Playwright 实操 + 代码审查 → 签收 or 打回
迭代 3（如打回）：开发 Agent → 根据反馈修改 → 重新发合同
迭代 4：测试 Agent → 更新用例 + 执行 → 签收 or 打回
迭代 5（如打回）：开发 Agent → 修复测试失败 → 重新发合同
```

**每个迭代的合同模板：**

### 四、质量门槛
| 指标 | 门槛值 |
|------|--------|
| 🔴 阻断问题 | 0 |

## 合同存储

所有合同存储在项目级别的 `.contracts/` 目录下：

```
.contracts/
├── {feature-name}/
│   ├── contract-01-dev-to-reviewer.md
│   ├── contract-02-reviewer-to-dev.md
│   ├── contract-03-dev-to-tester.md
│   └── contract-summary.md  ← 最终总结
```

### 合同编号规则

```
contract-{迭代号}-{发包方简称}-{承包方简称}.md
```

### 合同总结

当所有迭代完成后，生成合同总结：

```markdown
## 📋 合同总结：{功能/任务名称}

### 总迭代次数：X 轮
### 最终状态：✅ 通过 / ❌ 未通过（已知问题）

### 迭代历史
| 轮次 | 合同类型 | 结果 | 关键发现 |
|------|---------|------|---------|
| 1 | 开发→审查 | ❌ 打回 | 3个阻断问题 |
| 2 | 审查→修改 | ✅ 签收 | 全部修复 |
| 3 | 修改→测试 | ✅ 签收 | 测试全部通过 |

### 最终评分
| 维度 | 分数 |
|------|------|
| 总评分 | X.X/10 |
| 去AI味 | X.X/10 |

### 经验教训（供后续迭代参考）
- {教训1}
- {教训2}
```

## 协调者职责

当多个 agent 通过迭代合同协作时，需要一个**协调者**（通常是 main agent 或 startup-helper）：

1. **发起合同** — 明确第一个迭代的目标
2. **监控进度** — 追踪合同状态，防止卡住
3. **仲裁争议** — 当发包方和承包方对质量有分歧时裁决
4. **终止决策** — 判断是否达到终止条件
5. **汇总报告** — 生成最终合同总结

## 与现有机制的集成

1. **与 task-decomposer 集成** — 大任务拆解后，每个子任务都用迭代合同包装
2. **与 Playwright 审查集成** — 评审官的实操结果直接作为合同签收依据
3. **与去AI味评估集成** — 去AI味评分作为合同质量门槛的一部分
4. **与 ClawFlow 集成** — 合同状态作为 flow 的节点状态

---

**核心原则：不盲目信任任何 agent 的产出。每一步交付都要验证，每一个合同都要签收。**
