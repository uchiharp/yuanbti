# 任务分解与测试规划方法论

> 本 skill 可在 agent-pipeline 流程中使用，也可独立使用。

## 角色定义

- **任务分解执行者：** 任务规划师
- **测试计划执行者：** QA（与任务规划师并行工作）
- **评审官：** 任务评审官（冷血投资人思维，挑战任务合理性和完整性）
- **冒烟验证执行者：** 任务规划师（开发完成后执行P0冒烟）

## 流程概览

```
PRD + 架构方案 + UX方案 + UI设计（已签收）
  │
  ├──→ 任务规划师：产出开发任务清单（tasks.md）
  │         │
  └──→ QA（并行）：产出测试计划（test-plan.md）
            │
            ▼
各评审官审查（最多2轮迭代 → 签收）
  │
  ▼
开发确认（确认任务合理 → 修订 → 最终签收）
  │
  ▼
开发执行（按批次并发）
  │
  ▼
代码集成完成
  │
  ▼
任务规划师冒烟验证（P0维度）
  │
  ▼
审查 → 测试 → 交付
```

---

## 第一步：产出开发任务清单

任务规划师应整合所有上游文档（PRD、架构方案、UX方案、UI设计），分解为可执行的开发任务：

```markdown
## 开发任务清单

### 依据文档
- PRD：{路径}
- 架构方案：{路径}
- UX方案：{路径}
- UI设计：{路径}

### 批次规划

#### 批次1（无依赖，并发）
| 任务# | 操作 | 文件 | 验收标准 | 风险 | 负责人 | 参考文档 |
|-------|------|------|---------|------|--------|---------|
| 1.1 | 创建数据库表 | schema.sql | 表结构与架构一致 | 🟡 | 开发A | 架构§3 |
| 1.2 | 搭建项目框架 | pom.xml, build.gradle | 编译通过 | 🟢 | 开发B | 架构§2 |
| 1.3 | 设计Token | entity/User.java | 字段完整 | 🟢 | 开发C | 架构§3 |

#### 批次2（依赖批次1）
| 任务# | 操作 | 文件 | 验收标准 | 风险 | 负责人 |
|-------|------|------|---------|------|--------|
| 2.1 | 实现注册API | AuthController.java | 返回JWT | 🔴 | 开发A |
| 2.2 | 实现登录API | AuthController.java | 返回JWT | 🔴 | 开发A |
| 2.3 | 注册页面 | pages/register.vue | 表单可操作 | 🟡 | 开发B |
| 2.4 | 登录页面 | pages/login.vue | 表单可操作 | 🟡 | 开发B |

#### 批次3（依赖批次2）
...
```

## 第二步：QA 并行产出测试计划

QA 在任务规划师工作的同时，基于 PRD 产出测试计划：

```markdown
# 测试计划：{项目名}

## Part A：测试维度清单（按优先级）
| 优先级 | 模块 | 测试维度 | 验收标准 |
|--------|------|---------|--------|
| P0 | 登录 | 功能+边界+安全 | 核心流程无阻断 |
| P1 | 设备管理 | CRUD+异常+E2E | 数据一致性 |
| P2 | 性能/安全 | 响应时间/并发 | API<200ms/100并发通过 |

## Part B：测试数据需求
| 级别 | 类型 | 场景 |
|------|------|------|
| L1 | 种子数据 | 标准用户/订单/商品 |
| L2 | 快照数据 | 边界场景数据 |
| L3 | 真实设备 | 多端兼容性 |

## Part C：异常场景清单
| 场景 | 测试维度 | 预期结果 |
|------|---------|--------|
| 空值输入 | 边界 | 友好提示 |
| 并发操作 | 安全 | 数据一致 |
| 错误格式 | 健壮性 | 400错误 |
```

## 第三步：评审官审查

### 任务评审官审查维度

- 任务粒度（不大不小，一个任务1-3个文件）
- 验收标准（可量化，不是"功能正常"）
- 依赖关系（无循环，无遗漏）
- 风险分级（🔴🟡🟢标注合理）
- 是否遗漏了隐含工作（配置、异常处理、数据迁移）
- 与上游文档的一致性（PRD说的功能都有对应任务）

最多2轮迭代 → 签收。

---

## 第四步：开发确认

评审官签收后，实际执行开发的人员应快速确认：
- 分配给我的任务合理吗？
- 工作量预估准确吗？
- 依赖关系有遗漏吗？
- 和架构/UI设计有矛盾吗？

汇总意见 → 任务规划师修订 → 评审官快速复核 → 最终签收。

---

## 第五步：冒烟验证

开发完成并集成后，任务规划师执行 P0 冒烟验证，拦截阻断性问题。

### 冒烟验证流程

1. 按测试计划中 P0 维度执行冒烟测试
2. 验证核心流程端到端可通（启动→主流程→结束）
3. ✅ 通过 → 进入正式审查和测试
4. ❌ 失败 → 打回开发者，定位阻断问题，修复后重新冒烟

### 冒烟验证要点

- 只验证 P0（核心流程），不验证 P1/P2
- 关注"能不能走通"，不关注细节和边界
- 快速执行，发现阻断性问题立即打回

### 产出

冒烟验证报告（P0维度通过/失败清单）。

---

## 质量保障三层防线

| 防线 | 阶段 | 内容 | 负责人 |
|------|------|------|--------|
| 防线1 | 开发阶段 | 编码时自检（代码质量、编译验证） | 开发者 |
| 防线2 | 集成后 | P0 冒烟验证，拦截阻断性问题 | 任务规划师 |
| 防线3 | 测试阶段 | 全量测试，确保质量达标 | QA |

---

## QA与任务规划师协作流程

1. **任务规划师拆开发任务** → tasks.md（含批次、依赖、风险）
2. **QA 同时拆测试任务** → test-plan.md（含测试维度、优先级、异常场景）
3. **各自评审官审查** → 签收
4. **开发执行时**：QA 同步读源码写测试脚本（与开发并行）
5. **开发完成后**：任务规划师执行 P0 冒烟
6. **冒烟通过后**：QA 执行全量测试（L1→L2→L3）

---

## 验收标准

### tasks.md 验收标准

- [ ] 每个任务有明确的负责人
- [ ] 任务粒度可执行（单个任务 ≤ 1天工作量）
- [ ] 依赖关系已标注（任务间的先后顺序）
- [ ] 每个任务有验收标准（"做完什么样算完成"）
- [ ] 风险分级合理（🔴🟡🟢）
- [ ] 行数 ≥ 50行

### test-plan.md 验收标准

- [ ] 覆盖 PRD 中所有 P0/P1 功能的测试维度
- [ ] 每个维度有明确的验收标准
- [ ] 优先级已标注（P0优先执行）
- [ ] 包含异常场景（边界值、空值、并发、错误输入）

---

## ⚠️ 防偷懒规则

### 1. 禁止跳过步骤

**以下步骤不允许跳过、合并、用"详见XXX"替代：**
- 读取 PRD 和架构文档（必须读，不能靠记忆）
- 识别任务依赖关系（必须显式标注，不能"反正我知道"）
- 编写 test-plan.md（不能只在 tasks.md 里加一行"QA自行规划"）
- 冒烟验证逐项执行（不能跳过任何一项）

### 2. 任务粒度强制检查

- 单个任务工作量 **≤ 1天**（超过则必须拆分）
- 每个任务必须包含：**任务描述 + 负责人 + 验收标准 + 依赖关系**
- 任务描述禁止模糊表述：
  - ❌ "完成登录功能"
  - ✅ "实现邮箱+密码登录API，包含参数校验、JWT签发、错误码定义，产出 `POST /api/auth/login`"

### 3. test-plan 强制覆盖

- test-plan **必须** 覆盖 PRD 中所有 P0 和 P1 功能点
- 每个 P0 功能至少包含：正向场景 + 1个异常场景
- **不允许**只写"QA自行决定测试内容"——必须列出具体测试维度
- 覆盖率检查：`grep` PRD 功能清单中 P0/P1 关键词，对比 test-plan 是否都有对应条目

### 4. 冒烟验证强制执行

- 冒烟验证 **逐项执行**，不允许跳过
- 每项结果必须记录：✅通过 / ❌失败 + 原因
- 冒烟不通过 = **不进入后续阶段**，不允许"先往下走后面再测"

### 5. 产出物最小要求

| 文件 | 最小行数 | 最小内容 |
|------|---------|---------|
| tasks.md | 50行 | ≥5个任务，每个含描述+验收标准 |
| test-plan.md | 30行 | ≥P0功能全部覆盖 |

### 6. 偷懒检测信号

**以下行为视为偷懒：**
- tasks.md 行数 < 50 且任务描述过于简短
- test-plan.md 只写了"QA自行规划"
- 冒烟验证没有逐项记录
- 任务依赖关系全部标为"无依赖"（几乎不可能）
- 没有标注任何风险（小型项目也可能有技术风险）


> **产出路径：** 由调用方（协调者/用户）在任务消息中指定。未指定时默认 `/tmp/agent-skills-output/`。本skill不硬编码产出位置，可独立使用于任何项目。
