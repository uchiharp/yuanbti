# 任务分解与测试规划方法论

> 本 skill 可在 agent-pipeline 流程中使用，也可独立使用。

## 角色定义

- **任务分解执行者：** 任务规划师
- **测试计划执行者：** QA（与任务规划师并行工作）
- **评审官：** 任务评审官（冷血投资人思维，挑战任务合理性和完整性）
- **冒烟验证执行者：** 任务规划师（开发完成后执行P0冒烟）

## 流程概览

```
PRD + 架构方案 + UX方案 + UI设计（已签收）
  │
  ├──→ 任务规划师：产出开发任务清单（tasks.md）
  │         │
  └──→ QA（并行）：产出测试计划（test-plan.md）
            │
            ▼
各评审官审查（最多2轮迭代 → 签收）
  │
  ▼
开发确认（确认任务合理 → 修订 → 最终签收）
  │
  ▼
开发执行（按批次并发）
  │
  ▼
代码集成完成
  │
  ▼
任务规划师冒烟验证（P0维度）
  │
  ▼
审查 → 测试 → 交付
```

## 第一步：产出接口契约

在拆任务之前，先锁定模块间接口契约。并行开发最大的集成冲突来源就是接口不一致。

```markdown
## 模块间接口契约

| 提供方 | 消费方 | 接口 | 关键字段 | 类型 | 说明 |
|--------|--------|------|---------|------|------|
| 用户模块 | 订单模块 | GET /api/user/:id | userId | Long | 用户唯一标识 |
| 用户模块 | 订单模块 | GET /api/user/:id | name | String | 用户昵称 |
| 订单模块 | 支付模块 | POST /api/order | orderId | String | 订单号 |
```

**规则：**
- 接口契约在批次1开发前锁定
- 改契约必须通知所有消费方
- 契约是任务验收的依据之一

---

## 第二步：产出开发任务清单

任务规划师应整合所有上游文档（PRD、架构方案、UX方案、UI设计），分解为可执行的开发任务。

**核心原则：拆出来的每个任务都是必做项。** 不做的不拆。不设优先级字段，按依赖关系和并发组控制执行顺序。

```markdown
## 开发任务清单

### 依据文档
- PRD：{路径}
- 架构方案：{路径}
- UX方案：{路径}
- UI设计：{路径}
- 接口契约：{见第一步}

### 任务列表

| 任务# | 操作 | 文件 | 验收标准 | 工时 | 负责人 | 并发组 | 依赖 | 可重分配 |
|-------|------|------|---------|------|--------|--------|------|---------|
| T-001 | 搭建项目框架 | pom.xml, build.gradle | 编译通过 | 0.5d | 小一 | infra | 无 | ❌ |
| T-002 | 创建用户表 | schema.sql | 表结构与架构一致 | 0.5d | 小一 | auth | T-001 | ❌ |
| T-003 | 创建订单表 | schema.sql | 表结构与架构一致 | 0.5d | 小二 | order | T-001 | ❌ |
| T-004 | 用户注册API | AuthController.java | 返回JWT | 1d | 小一 | auth | T-002 | ❌ |
| T-005 | 订单创建API | OrderController.java | 返回订单号 | 1d | 小二 | order | T-003 | ❌ |
| T-006 | 用户前端页面 | pages/user.vue | 表单可操作 | 0.5d | 小三 | frontend | T-004 | ✅ |
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 任务# | ✅ | 唯一编号 T-XXX |
| 操作 | ✅ | 具体做什么（禁止"完成XX功能"这种模糊描述） |
| 文件 | ✅ | 涉及的文件列表 |
| 验收标准 | ✅ | 可量化（不是"功能正常"） |
| 工时 | ✅ | 单个任务 ≤ 1天 |
| 负责人 | ✅ | 分配给谁 |
| 并发组 | ✅ | 同组串行，不同组并发。如 auth/order/frontend/infra |
| 依赖 | ✅ | 前置任务编号，无则填"无" |
| 可重分配 | ✅ | 失败时能否转给其他人 |

### 任务分配规则

**规则1：评审上下文复用（最高优先级）**
交叉评审阶段本身就是上下文建设过程。任务分配时，优先分给已在上游评审中了解过相关模块的开发者：
- 阶段1.5 PRD交叉评审：谁评审了哪些功能点 → 那些功能点的任务优先分给谁
- 阶段2.5 架构交叉评审：谁评审了哪些模块的设计 → 那些模块的开发任务优先分给谁
- 阶段5.5 分解确认：开发者确认任务时进一步巩固上下文

任务规划师在拆任务前，应先收集交叉评审记录（谁评审了哪些模块），作为分配依据。

**规则2：模块内上下文连续性**
同一模块的上下游任务优先分给同一个人。已有上下文的开发者不需要重新读文件，减少 token 消耗和理解偏差。

**规则3：工作量均衡**
按总工时均衡分配，不是按任务数。当上下文连续性和工作量冲突时，优先保证上下文连续性，用工时微调（±0.5d）平衡。

**规则4：核心依赖优先**
无依赖的基础任务（项目骨架、数据库初始化等）优先安排，它们是其他任务的前置条件。

**规则5：失败由原 agent 修复**
任务失败时由执行该任务的 agent 自行修复，不需要预留空闲 agent。

**规则6：失败重分配**
每个任务标记"可重分配"（✅）或"仅限本人"（❌）。
- 核心模块首次实现 → 仅限本人（上下文太重要）
- 修复类任务 → 可重分配（谁有空谁修）
- 开发者超时或产出不合格 → 协调者将可重分配任务转给空闲开发者

## 第二步：QA 并行产出测试计划

QA 在任务规划师工作的同时，基于 PRD 产出测试计划：

```markdown
# 测试计划：{项目名}

## Part A：测试维度清单
| 模块 | 测试维度 | 验收标准 |
|------|---------|--------|
| 登录 | 功能+边界+安全 | 核心流程无阻断 |
| 设备管理 | CRUD+异常+E2E | 数据一致性 |
| 性能/安全 | 响应时间/并发 | API<200ms/100并发通过 |

## Part B：测试数据需求
| 级别 | 类型 | 场景 |
|------|------|------|
| L1 | 种子数据 | 标准用户/订单/商品 |
| L2 | 快照数据 | 边界场景数据 |
| L3 | 真实设备 | 多端兼容性 |

## Part C：异常场景清单
| 场景 | 测试维度 | 预期结果 |
|------|---------|--------|
| 空值输入 | 边界 | 友好提示 |
| 并发操作 | 安全 | 数据一致 |
| 错误格式 | 健壮性 | 400错误 |
```

## 第三步：评审官审查

### 任务评审官审查维度

- 任务粒度（不大不小，一个任务1-3个文件）
- 验收标准（可量化，不是"功能正常"）
- 依赖关系（无循环，无遗漏）
- 风险分级（🔴🟡🟢标注合理）
- 是否遗漏了隐含工作（配置、异常处理、数据迁移）
- 与上游文档的一致性（PRD说的功能都有对应任务）

最多2轮迭代 → 签收。

---

## 第四步：开发确认

评审官签收后，实际执行开发的人员应快速确认：
- 分配给我的任务合理吗？
- 工作量预估准确吗？
- 依赖关系有遗漏吗？
- 和架构/UI设计有矛盾吗？

汇总意见 → 任务规划师修订 → 评审官快速复核 → 最终签收。

---

## 第五步：冒烟验证

开发完成并集成后，任务规划师执行 P0 冒烟验证，拦截阻断性问题。

### 冒烟验证流程

1. 按测试计划中 P0 维度执行冒烟测试
2. 验证核心流程端到端可通（启动→主流程→结束）
3. ✅ 通过 → 进入正式审查和测试
4. ❌ 失败 → 打回开发者，定位阻断问题，修复后重新冒烟

### 冒烟验证要点

- 只验证 P0（核心流程），不验证 P1/P2
- 关注"能不能走通"，不关注细节和边界
- 快速执行，发现阻断性问题立即打回

### 产出

冒烟验证报告（P0维度通过/失败清单）。

---

## 质量保障三层防线

| 防线 | 阶段 | 内容 | 负责人 |
|------|------|------|--------|
| 防线1 | 开发阶段 | 编码时自检（代码质量、编译验证） | 开发者 |
| 防线2 | 集成后 | P0 冒烟验证，拦截阻断性问题 | 任务规划师 |
| 防线3 | 测试阶段 | 全量测试，确保质量达标 | QA |

---

## QA与任务规划师协作流程

1. **任务规划师拆开发任务** → tasks.md（含批次、依赖、风险）
2. **QA 同时拆测试任务** → test-plan.md（含测试维度、优先级、异常场景）
3. **各自评审官审查** → 签收
4. **开发执行时**：QA 同步读源码写测试脚本（与开发并行）
5. **开发完成后**：任务规划师执行 P0 冒烟
6. **冒烟通过后**：QA 执行全量测试（L1→L2→L3）

---

## 验收标准

### tasks.md 验收标准

- [ ] 每个任务有明确的负责人
- [ ] 任务粒度可执行（单个任务 ≤ 1天工作量）
- [ ] 依赖关系已标注（任务间的先后顺序）
- [ ] 每个任务有验收标准（"做完什么样算完成"）
- [ ] 风险分级合理（🔴🟡🟢）
- [ ] 行数 ≥ 50行

### test-plan.md 验收标准

- [ ] 覆盖 PRD 中所有 P0/P1 功能的测试维度
- [ ] 每个维度有明确的验收标准
- [ ] 优先级已标注（P0优先执行）
- [ ] 包含异常场景（边界值、空值、并发、错误输入）

---

## ⚠️ 防偷懒规则

### 1. 禁止跳过步骤

**以下步骤不允许跳过、合并、用"详见XXX"替代：**
- 读取 PRD 和架构文档（必须读，不能靠记忆）
- 识别任务依赖关系（必须显式标注，不能"反正我知道"）
- 编写 test-plan.md（不能只在 tasks.md 里加一行"QA自行规划"）
- 冒烟验证逐项执行（不能跳过任何一项）

### 2. 任务粒度强制检查

- 单个任务工作量 **≤ 1天**（超过则必须拆分）
- 每个任务必须包含：**任务描述 + 负责人 + 验收标准 + 依赖关系**
- 任务描述禁止模糊表述：
  - ❌ "完成登录功能"
  - ✅ "实现邮箱+密码登录API，包含参数校验、JWT签发、错误码定义，产出 `POST /api/auth/login`"

### 3. test-plan 强制覆盖

- test-plan **必须** 覆盖 PRD 中所有 P0 和 P1 功能点
- 每个 P0 功能至少包含：正向场景 + 1个异常场景
- **不允许**只写"QA自行决定测试内容"——必须列出具体测试维度
- 覆盖率检查：`grep` PRD 功能清单中 P0/P1 关键词，对比 test-plan 是否都有对应条目

### 4. 冒烟验证强制执行

- 冒烟验证 **逐项执行**，不允许跳过
- 每项结果必须记录：✅通过 / ❌失败 + 原因
- 冒烟不通过 = **不进入后续阶段**，不允许"先往下走后面再测"

### 5. 产出物最小要求

| 文件 | 最小行数 | 最小内容 |
|------|---------|---------|
| tasks.md | 50行 | ≥5个任务，每个含描述+验收标准 |
| test-plan.md | 30行 | ≥P0功能全部覆盖 |

### 6. 偷懒检测信号

**以下行为视为偷懒：**
- tasks.md 行数 < 50 且任务描述过于简短
- test-plan.md 只写了"QA自行规划"
- 冒烟验证没有逐项记录
- 任务依赖关系全部标为"无依赖"（几乎不可能）
- 没有标注任何风险（小型项目也可能有技术风险）


## 共享上下文注入机制

任务规划师在产出 tasks.md 和接口契约后，协调者需要在后续阶段向各角色注入上下文。本机制定义上下文包的内容和注入时机。

### 两种注入场景

#### 场景A：交叉评审时的上下文注入

pipeline 的交叉评审阶段（1.5/2.5/3.5/4.5/5.5），协调者向评审者注入：

```
📋 交叉评审上下文包
━━━━━━━━━━━━━━━━━━
📄 被评审文档：{完整文档}
📎 相关上游文档：{评审者专业视角需要的上游文档片段}
🔍 评审检查清单：{对应角色的评审维度}
```

各阶段注入内容：

| 阶段 | 评审者 | 注入的上游上下文 | 评审视角 |
|------|--------|----------------|---------|
| 1.5 PRD评审 | 架构师 | 技术约束、非功能需求 | 技术可行性 |
| 1.5 PRD评审 | QA | 验收标准、用户故事 | 可测试性 |
| 1.5 PRD评审 | 开发 | 功能清单、优先级 | 工作量评估 |
| 2.5 架构评审 | 开发 | PRD功能清单+接口契约草案 | 接口合理性 |
| 2.5 架构评审 | QA | PRD验收标准+异常场景 | 可观测性 |
| 3.5 UX→UI | UI设计师 | PRD用户故事+设计约束 | 视觉可行性 |
| 4.5 UI→UX | UX测试 | PRD交互流程+UX方案 | 交互完整性 |
| 5.5 分解确认 | 开发×3 | 接口契约+架构API设计+各自任务 | 可实现性 |

#### 场景B：任务派发时的上下文补充

阶段6协调者向开发者派发任务时，开发者已在阶段1.5/2.5评审中建立上下文。协调者只需补充评审阶段未覆盖的信息：

```
📋 任务派发 | 任务 {编号}
━━━━━━━━━━━━━━━━━━━━━━━━
🎯 任务：{任务描述}
📁 涉及文件：{文件列表}
✅ 验收标准：{可量化的标准}

🔗 接口契约（评审后锁定版本）：
{你负责模块的接口定义，可能与评审时的草案有差异，以此为准}

⚠️ 变更提示：
{评审后接口/设计的变更点，你评审时的版本 vs 最终版本的差异}
```

**原则：评审阶段已确认的内容不重复注入，只补充变更和新增信息。**

### 上下文注入的裁剪原则

1. **只注入相关片段** — 不灌全文，只提取与当前任务/评审相关的段落
2. **接口契约必须完整** — 涉及的接口定义必须完整注入，不能裁剪字段
3. **按优先级裁剪** — P0任务注入完整上下文，P1任务可裁剪PRD用户故事部分
4. **控制总长度** — 单个上下文包 ≤ 2000 tokens，避免淹没核心信息

### 与 pipeline 阶段的上下文流转

```
阶段1.5 PRD交叉评审
  │ 架构师/QA/开发各自评审 → 建立需求上下文
  │ 记录：谁评审了哪些功能模块
  │
  ├──→ 阶段2.5 架构交叉评审
  │     开发/QA评审 → 建立技术设计上下文
  │     记录：谁评审了哪些模块的设计
  │
  ├──→ 阶段5 任务分解
  │     任务规划师读取评审记录 → 按评审上下文分配任务
  │     产出：tasks.md（含上下文来源列）+ 接口契约
  │
  ├──→ 阶段5.5 分解确认
  │     开发者已了解相关模块 → 快速确认，只看变更
  │
  ├──→ 阶段6 开发执行
  │     协调者只补充：接口契约锁定版 + 评审后变更点
  │     开发者已有上下文，无需重读上游文档
  │
  └──→ 阶段6.3 冒烟验证
        创业助手验证：接口契约一致性 + test-plan P0维度
```

**核心逻辑：评审即上下文建设，分配即上下文复用，派发只补变更。**
