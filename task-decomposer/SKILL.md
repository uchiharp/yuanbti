# 任务分解与多Agent调度 v2

## 核心原则

1. **先给用户看计划，确认后再执行** — 禁止未经用户确认就启动Agent
2. **先产出事实，再做判断** — 第一轮只产出清单/表格，第二轮基于清单做分析
3. **一个Agent只做一件事** — 读一个文件或一类操作，不做组合任务
4. **表格模板约束输出** — 用结构化表格让Agent无法写模糊文字
5. **输出可量化验证** — 数行数、查行号、验证字段完整性
6. **分轮执行，逐轮验证** — 每轮完成验证后再启动下一轮
7. **断点续传** — 任务计划持久化到文件，每完成一项更新状态，session 断开后可从断点继续
8. **挑刺官交叉审查** — 执行Agent完成任务后，派对应领域的挑刺官审查，执行+审查结果同时汇报

## ⚠️ 新增规则（基于实战教训）

### 规则1：执行前必须给用户看完整计划

**禁止行为：** 写完计划立刻 `sessions_spawn` 启动Agent。

**正确流程：**
```
1. 拆解任务 → 产出详细计划（表格）
2. 发给用户看
3. 用户确认/修改后
4. 才能启动Agent执行
```

**计划必须包含：**
- 每个Agent的具体任务（不是一句话，是详细说明）
- 产出文件的**绝对路径**
- 每个用例的方法名 + 操作 + 预期结果（不能只写"注册→200"）
- 执行顺序和依赖关系
- 预估时间

### 规则2：先验证环境再分发任务

**分发Agent之前，协调者必须自己确认：**

| 检查项 | 命令/方法 | 不通过时 |
|--------|----------|---------|
| 编译通过 | `mvn compile test-compile` | 先修编译错误 |
| 测试环境能启动 | `mvn test -Dtest="SimpleTest"`（跑1个简单测试） | 先修配置 |
| 依赖完整 | 检查pom.json/package.json | 先加依赖 |
| 环境变量有默认值 | grep `${` application-test.yml | 先加默认值 |

**教训：** 今天因为test profile没配好，8个测试类全军覆没。分发前1分钟验证能省1小时排错。

### 规则3：产出必须用绝对路径

❌ 错误：`e2e/auth.spec.js`
✅ 正确：`/Users/sunwenyong/projects/deer-flow/output/finder-app/uni-app/e2e/auth.spec.js`

所有文件路径在计划、Agent任务、最终报告中都必须用绝对路径。

### 规则4：先查功能是否存在再写测试

**教训：** 我声称"AI识别→添加到列表已测试"，但用户质疑功能是否做了。

**正确流程：**
```
写测试前：
1. grep确认功能代码存在
2. grep确认API接口存在
3. 确认功能可用（curl或手动验证）
4. 然后才写测试
```

**产出时标注：**
| 用例 | 功能状态 | 测试状态 |
|------|---------|---------|
| AI识别添加到列表 | ✅ search.vue:160 addToItems | ✅ ai-image-search.spec.js #3 |

### 规则5：用例必须细化

❌ 错误（太粗）：
```
| 1 | 注册 | POST /register → 200 |
```

✅ 正确（够细）：
```
| # | 方法名 | 操作 | 预期 |
|---|--------|------|------|
| 1 | register_newUser_returnsToken | POST /api/auth/register body:{"username":"test_时间戳","email":"test_xxx@test.com","password":"Test1234"} | 200, body含accessToken字段，非null |
```

### 规则6：Agent超时恢复策略

| 重试次数 | 操作 |
|---------|------|
| 第1次超时（10分钟） | 缩小任务范围，拆成2个小Agent |
| 第2次超时 | 换模型（GLM-5.1 → glm-5-turbo或反过来） |
| 第3次 | 自己做 |

**注意：** 超时的Agent可能已经产出了部分文件（文件存在但不完整）。下次要检查已有文件。

### 规则7：协调者只做编译验证

Agent返回后，协调者只做一件事：编译验证。
```
mvn compile test-compile  # 后端
npm run build                  # 前端
```
编译失败直接打回Agent修。不做其他检查。

**责任边界：** import完整性、文件非空、行数达标等是Agent自己的责任，应该在Agent任务指令里写清楚，不是协调者的事。协调者管计划和编译，Agent管代码质量。

## Step 0: 判断是否需要拆解

遇到任务时先判断：

| 信号 | 说明 |
|------|------|
| 任务描述含"全面"、"所有"、"整体"、"从头到尾" | 必须拆解 |
| 需要读3个以上文件才能完成 | 必须拆解 |
| 产出包含多种类型（清单+分析+修复） | 必须拆解 |
| 只需读1个文件、产出单一 | 不用拆，直接做 |

## Step 1: 确定维度

### 1.0 检查是否有未完成的任务计划

任务计划存储在 `workspace/task-plans/` 目录下，每个任务一个独立文件。

**文件命名规则：** `{任务名}-{YYYYMMDD}.md`
- 例：`e2e-fix-selectors-20260411.md`
- 例：`code-review-auth-20260410.md`

```bash
# 检查是否存在未完成的任务计划
grep -rl "🔄 进行中\|⏳ 待开始" workspace/task-plans/ 2>/dev/null
```

- **有未完成计划** → 询问用户：继续还是放弃？
- **没有未完成计划** → 正常开始新任务
- **不同任务的计划互不干扰**，各自独立文件

### 断点续传流程

```
1. 读取 task-plan.md
2. 找到第一个 ⏳ 或 🔄 的任务
3. 跳过所有 ✅ 的已完成任务
4. 从断点任务继续执行
5. 每完成一项 → 更新 task-plan.md 中的状态
```

### 任务计划文件格式（task-plan.md）

```markdown
# 任务计划：{任务名称}
创建时间：YYYY-MM-DD HH:mm
总体状态：🔄 进行中 | ✅ 已完成 | ❌ 已取消

## 任务概览
| 轮次 | 模块 | 状态 | 失败数 | 涉及文件 | Agent ID |
|------|------|------|--------|---------|----------|
| 1 | auth | ✅ 完成 | 4/4修复 | auth.spec.js | abc123 |
| 2 | home | 🔄 进行中 | 0/1 | home.spec.js | def456 |
| 3 | search | ⏳ 待开始 | - | search*.spec.js | - |
| 4 | AI | ⏳ 待开始 | - | ai-*.spec.js | - |

## 详细任务

### 轮次1: auth 注册/登录/登出
- 状态: ✅ 完成
- Agent task: {具体任务描述}
- 产出: {文件路径}
- 验证结果: 4/4 通过
- 完成时间: 2026-04-11 13:30

### 轮次2: home 首页
- 状态: 🔄 进行中
- Agent task: {具体任务描述}
- Agent session: {sessionKey}
- 开始时间: 2026-04-11 13:35

### 轮次3: search 搜索
- 状态: ⏳ 待开始
- Agent task: {具体任务描述}
```

### 状态流转规则

```
⏳ 待开始 → 🔄 进行中（Agent启动时）→ ✅ 完成（验证通过）或 ❌ 失败（验证不通过）
❌ 失败 → 🔄 进行中（重试时）
```

### 更新时机

| 事件 | 更新内容 |
|------|----------|
| 计划确认 | 创建 task-plan.md，所有任务标 ⏳ |
| Agent 启动 | 对应任务改 🔄，记录 session |
| Agent 完成 + 验证通过 | 对应任务改 ✅，记录结果 |
| Agent 完成 + 验证失败 | 对应任务改 ❌，记录错误 |
| 全部完成 | 总体状态改 ✅，归档文件 |
| Session 中断 | 文件保留，下次读取继续 |

### 归档与清理规则

**无需归档** — 每个任务已经是独立文件，完成后保留作为历史记录。

**定期清理**（加入 HEARTBEAT.md 巡检任务）：
```bash
# 超过 15 天的 plan 文件自动清理
find workspace/task-plans/ -name "*.md" -mtime +15 -delete
```

**完成标记：**
- 全部完成 → 总体状态改 `✅ 已完成`
- 用户取消 → 总体状态改 `❌ 已取消`
- 不需要移动文件，保留原位

### 决策规则

1. **审查/分析类** → 按关注点拆：安全、性能、规范、测试覆盖、可维护性
2. **修复类** → 按文件分组：同一文件的改动归一组，每组一个Agent
3. **开发类** → 按模块拆：前端、后端、数据库、配置
4. **测试类** → 按功能模块拆：认证、CRUD、搜索、AI、配额等
5. **对比/交叉类** → 按数据源拆：A数据 vs B数据，找出不一致

### 维度设计要求

- 每个维度 = 一个可独立验证的产出（清单、表格、报告）
- 维度之间尽量独立，减少互相依赖
- 每个维度可由一个Agent独立完成，不需要等其他Agent

### 任务粒度标准

什么事算"一件"：

| ✅ 一件 | ❌ 不是一件 |
|---------|-----------|
| 读一个文件 + 产出一张表 | 读文件 + 分析 + 写报告 |
| 读N个同类型文件 + 产出一张汇总表 | 读N个不同类型文件做综合分析 |
| 修复一个文件里的多个问题 | 修复跨文件的问题链 |
| 写一个模块的测试（2-3个文件） | 写整个项目的测试（8+个文件） |

**注意：** 任务太大会导致Agent超时（10分钟）。超过3个文件的写作任务必须拆分。

## Step 2: 第一轮 — 产出事实清单

每个维度一个Agent，**只要求读代码/文档 + 产出表格**。

```
第一轮（并行）
  ├─ Agent A → 读文件X → 产出清单A（表格）
  ├─ Agent B → 读文件Y → 产出清单B（表格）
  └─ Agent C → 读文件Z → 产出清单C（表格）
```

### 任务模板

```
你是{角色}。只做一件事，不要多做也不要少做。

**任务：{具体到文件+操作}**

文件路径：`{完整绝对路径}`

产出格式：
| 列1 | 列2 | 列3 |

要求：
- 不要猜测，只写代码/文档中实际看到的
- 每个xxx都要列出来，不能遗漏
- 直接输出表格，不需要开头结尾的客套话
```

## Step 3: 验证第一轮产出

对每个Agent的产出执行以下检查：

| 检查项 | 方法 | 不合格处理 |
|--------|------|-----------|
| 文件存在且非空 | `ls` + `wc -l` | 打回重做 |
| 表格行数 ≥ 预估值 | 数行数 | 打回补全 |
| 每列都填了值 | 检查空列 | 打回补充 |
| 数据分布合理 | "严重程度"不应全是P0或全是P2 | 打回重新评估 |
| 抽查1-2行准确性 | 对照源文件验证 | 打回修正 |

## Step 4: 第二轮 — 基于清单做分析/执行

第一轮的产出作为第二轮的输入。

```
第二轮（并行）
  ├─ Agent D → 读清单A + 代码 → 产出分析/修复
  ├─ Agent E → 读清单B + 代码 → 产出分析/修复
  └─ Agent F → 对比清单A+C → 找不一致
```

### 任务模板

```
你是{角色}。只做一件事。

**背景信息（第一轮产出）：**
- {清单A的关键数据}

**任务：基于以上信息，{具体操作}**

检查文件：
- `{绝对路径1}` — 关注{具体点}

产出文件：`{绝对路径}`

产出格式：
| 列1 | 列2 | 列3 |

要求：
- 包含必要的import语句
- 编译验证通过
- 产出文件路径必须是绝对路径
```

## Step 5: 第三轮 — 验证+汇总

```
第三轮
  ├─ 编译验证：mvn compile test-compile
  ├─ 跑测试：mvn test / npx playwright test
  ├─ 汇总结果 → 产出报告
  └─ 报告存到绝对路径
```

## 修复类任务的调度方式

与审查类不同，修复任务按文件分组：

```
修复任务分解：
  1. 汇总所有待修复项
  2. 按涉及文件分组（同一文件的改动手动合并）
  3. 每组一个Agent，任务包含：
     - 具体修改内容（精确到行号）
     - 修改后自己跑编译验证
  4. 协调者做集成验证（全量测试）
```

## 测试类任务的调度方式

```
测试任务分解：
  1. 先验证测试环境能启动（跑1个简单测试）
  2. 按功能模块分组，每组一个Agent
  3. 每组任务包含：
     - 要先读的文件（了解API/组件）
     - 产出文件的绝对路径
     - 每个用例的方法名 + 操作 + 预期
     - **明确要求：确保所有import完整，编译通过后再返回**
  4. 编译验证通过后再跑测试
```

## 防偷懒机制速查

| 手段 | 说明 |
|------|------|
| 表格模板 | 每列都有具体含义，无法用模糊文字填充 |
| 最少行数 | "至少N行" — 不够说明有遗漏 |
| 要求行号 | 必须标注代码行号，证明真的读了 |
| 不许客套 | "直接输出表格，不需要开头结尾" |
| 一个文件一个Agent | 任务越小越难偷懒 |
| 验证后再启动下一轮 | 防止错误累积 |
| 数据分布检查 | 不能全标P0或全标P2 |
| **验证编译** | 编译失败直接打回Agent |
| **检查文件非空** | `wc -l` 确认行数 |

## 通用调度命令

```
# 并行启动多个Agent（确认用户后）
sessions_spawn(label="{维度}", task="{模板}", mode="run", runTimeoutSeconds=300)

# 等待全部返回 → 检查质量 → 验证编译 → 启动下一轮
```

## 挑刺官交叉审查机制

### 8 个挑刺官

| 挑刺官 | Agent ID | 审查维度 | 适用场景 |
|--------|----------|---------|----------|
| 架构师挑刺官 | architect-reviewer | 架构设计、方案合理性 | 技术方案、API设计 |
| 后端挑刺官 | backend-reviewer | 后端代码质量、Java规范 | 后端开发、接口实现 |
| 前端挑刺官 | frontend-reviewer | 前端代码质量、Vue规范 | 前端开发、UI实现 |
| PM挑刺官 | pm-reviewer | 需求覆盖、PRD一致性 | 功能实现、需求验收 |
| QA挑刺官 | qa-reviewer | 测试覆盖、边界场景 | 测试编写、Bug修复 |
| UI设计师挑刺官 | ui-designer-reviewer | 视觉规范、交互一致性 | UI变更、页面开发 |
| UX测试挑刺官 | ux-tester-reviewer | 用户体验、使用流程 | 功能交付、交互优化 |
| Code Review | code-review | 通用代码质量、安全 | 任何代码变更 |

### 调度规则

```
执行Agent完成 → 协调者初步验证 → 派挑刺官审查
                                                    ↓
                                         挑刺官只做审查，不改代码
                                                    ↓
                                         输出：问题清单 + 严重程度
                                                    ↓
                                    协调者汇总执行结果+挑刺结果 → 一起汇报
```

### 挑刺官任务模板

```
你是{挑刺官名}。审查{执行Agent}的工作成果。

## 审查范围
- 文件：{绝对路径列表}
- 任务：{原始任务描述}
- 执行者产出：{产出的文件/修改}

## 审查要求

1. 逐一检查产出的每个文件/修改
2. 只挑刺，不改代码
3. 按严重程度分级：
   - 🔴 P0：功能错误、逻辑bug、安全隐患
   - 🟡 P1：规范问题、潜在风险、不够健壮
   - 🔵 P2：代码风格、命名、注释、可读性

## 产出格式
| # | 文件 | 行号 | 严重程度 | 问题描述 | 建议修复方式 |
|---|------|------|---------|---------|------------|

如果没有问题，输出："✅ 审查通过，未发现问题。"
```

### 挑刺官选择策略

| 任务类型 | 必选挑刺官 | 可选招刺官 |
|---------|-----------|-----------|
| 后端开发 | backend-reviewer, code-review | architect-reviewer |
| 前端开发 | frontend-reviewer, code-review | ui-designer-reviewer |
| API设计 | architect-reviewer, pm-reviewer | backend-reviewer |
| 测试修复 | qa-reviewer | code-review |
| UI变更 | ui-designer-reviewer, frontend-reviewer | ux-tester-reviewer |
| 功能实现 | pm-reviewer, ux-tester-reviewer | 对应技术挑刺官 |
| Bug修复 | code-review | 对应领域挑刺官 |

### 挑刺结果处理

| 挑刺结果 | 处理方式 |
|---------|----------|
| ✅ 无问题 | 标记任务完成 |
| 🔴 P0 | 必须修复，派执行Agent修复后再审查 |
| 🟡 P1 | 建议修复，询问用户是否处理 |
| 🔵 P2 | 记录，不阻塞流程 |

每个任务增加挑刺官字段：

```
| 轮次 | 模块 | 状态 | 执行Agent | 挑刺官 | 挑刺结果 |

| 1 | auth | ✅ 完成 | e2e-fix-1 | frontend-reviewer | ✅ 通过 |
| 2 | search | 🔄 挑刺中 | e2e-fix-2 | qa-reviewer | - |
```
