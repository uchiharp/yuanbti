# Reasoning RAG

当需要基于内部知识（MEMORY.md / memory/*.md）回答复杂问题时，使用此流程替代单次 memory_search。

## 触发条件

- 问题涉及多个方面或需要综合判断
- 单次搜索可能遗漏关键信息
- 需要对比、分析、推理（不是简单的事实查询）

## 工作流

### Step 1: 问题拆解

把用户的问题拆成 2-5 个子问题。每个子问题用不同的关键词搜索。

示例：
- 用户问 "RAG 和推理型 RAG 有什么区别，我该用哪个？"
- 拆解为：
  - 子问题1: "RAG 相关记录" → 搜索 "RAG 检索增强"
  - 子问题2: "推理型 RAG 相关记录" → 搜索 "推理型 RAG reasoning"
  - 子问题3: "之前做过的技术选型" → 搜索 "技术选型 决策"

### Step 2: 多轮检索

对每个子问题分别调用 memory_search，使用不同的 query：

```
memory_search(query="关键词1")
memory_search(query="关键词2")  
memory_search(query="关键词3")
```

### Step 3: 交叉验证

检查搜索结果：
- **矛盾检测** — 不同来源的信息是否冲突？如果有，哪个更新/更可信？
- **覆盖检测** — 有没有子问题没搜到答案？需要补充搜索吗？
- **时效检测** — 信息是否过时？最近有没有更新？

如果发现问题，执行补充搜索。

### Step 4: 综合回答

基于所有检索结果，组织回答：
1. 先给结论
2. 列出依据（标注来源：`Source: memory/2026-04-07.md#L10`）
3. 标注不确定性 — "这个信息来自 X 天前，可能已更新"
4. 如果有矛盾，说明冲突点并给出判断

## 与简单查询的判断标准

| 场景 | 用基础 RAG | 用推理型 RAG |
|------|-----------|-------------|
| "昨天聊了什么" | ✅ | ❌ |
| "上次那个项目的方案是什么" | ✅ | ❌ |
| "我该选 A 还是 B" | ❌ | ✅ |
| "总结一下最近的学习进展" | ❌ | ✅ |
| "之前做的技术决策哪些该调整" | ❌ | ✅ |

## 局限性

- 底层仍然是 memory_search（语义搜索），不是真正的向量数据库
- 检索质量取决于笔记的完整性和组织方式
- 如果信息从没记下来，推理型也搜不到
- 建议：重要信息及时记录到 memory/ 文件，方便后续检索
