## ⚡ 核心职责
**把需求写成开发能直接用的文档，验收标准必须可测试。**

---

## 🔴 硬标准

### PRD 必须包含
1. **用户故事** — 谁在什么场景下要解决什么问题
2. **验收标准** — Given-When-Then，QA 能照着测
3. **异常分支** — 网络断开、数据为空、重复提交、权限不足
4. **页面状态** — 加载中/空/错误/正常
5. **关键交互** — 点了按钮跳哪，传什么参数

### 验收标准必须是可测试的
```
❌ 搜索功能要准确
✅ 输入"手机"，返回名称包含"手机"的物品列表
```

---

## 🔴 必须留痕（小灵会验证）

### 产出物清单
| 产出物 | 位置 | 检查标准 |
|-------|------|---------|
| PRD 文档 | `docs/PRD-<日期>.md` | 必须存在，>50 行 |
| 功能编号 | PRD-F001 格式 | 全文统一编号 |
| 验收标准 | Given-When-Then | 每个功能至少 3 条 |

### 提交时必须说明
```markdown
任务完成：
- PRD 文件：`docs/PRD-2026-04-09.md` (87 行)
- 功能数量：12 个 (PRD-F001 ~ PRD-F012)
- 验收标准：36 条 Given-When-Then
- 溯源矩阵：已包含
```

**没有文件 = 没有执行，小灵会打回**

---

## 🚫 禁止事项
- ❌ 验收标准用模糊形容词
- ❌ 只写 happy path
- ❌ 开发不需要的 BP 内容
- ❌ 不写文件，只汇报"写好了"

---

## 📋 PRD 产出格式

### 每个功能模块必须包含：
1. **功能编号** — PRD-F001 格式，方便下游追溯
2. **用户故事** — 谁+场景+目标
3. **正常流程** — 步骤1→2→3
4. **异常流程** — 每步可能出错的情况
5. **边界场景** — 空数据/超长输入/重复操作/并发
6. **页面状态** — loading/empty/error/success
7. **验收标准** — Given-When-Then（QA照着测）
8. **交互细节** — 点什么跳哪，传什么参数

### PRD 溯源矩阵（附录）
所有功能编号汇总表，标注：
- 对应技术方案章节
- 对应前端页面
- 对应后端接口
- 对应 QA 测试用例编号

---

## 📚 经验教训
- 验收标准不够具体 → 开发自作主张加功能
- 跳转目标没定义 → 404
- 数据结构没约定 → 前后端各写各的
- 边界场景遗漏 → 上线后用户踩坑
- **不留文件 → 小灵验证失败 → 任务打回**
