# Agent 开发流水线 v4（骨架版）

## 核心理念

**不是让一个agent做完所有事，而是让一群专业agent各司其职、互相评审、迭代交付。**

每个环节都有"对手"来挑战产出质量。从PRD到代码，从设计到测试，每一步交付都经过专业评审官的严格审查。

### v4 新增特性
1. **项目规模裁剪** — 大/中/小型项目走不同流程长度，避免小项目过重
2. **技术Spike（阶段2.8）** — 高风险技术点前置验证，4h时限
3. **代码集成（阶段6.3）** — 多Agent并发开发后的合并与冲突解决
4. **异步交叉评审24h时限** — 交叉评审参与方有24h窗口期
5. **三级测试数据管理** — L1种子数据 / L2快照数据 / L3真实设备
6. **项目健康度看板** — 每阶段更新，可视化项目状态
7. **CI/CD集成**（可选）— GitHub Actions 模板，可跳过
8. **文件头YAML标记** — version 字段标识版本

## 项目规模裁剪（v4新增）

不同规模项目走不同流程长度，避免"杀鸡用牛刀"。

### 规模判定标准（阶段0由协调者判定）

| 规模 | 判定条件 | 示例 |
|------|---------|------|
| 🔴 大型 | 功能≥10、涉及≥3个子系统、预估工期≥2周 | 电商系统、社交App |
| 🟡 中型 | 功能5-9、1-2个子系统、工期1-2周 | 后台管理、小程序 |
| 🟢 小型 | 功能≤4、单子系统、工期<1周 | 工具页、H5活动页 |

## 🧠 Brainstorm 融合原则（Superpowers × Pipeline）

**核心理念：** 任何需要产出文档的阶段，都应先用 Brainstorm 的对话式讨论方式与用户确认，最后再整理成文档。

### 适用阶段

| 阶段 | Brainstorm 讨论什么 | 产出文档 |
|------|-------------------|---------|
| **阶段1（PRD设计）** | 用户是谁、解决什么痛点、功能边界、验收标准、和竞品的差异化 | PRD.md |
| **阶段2（架构设计）** | 技术选型、架构风格、数据模型、API设计、非功能需求（性能/安全/可扩展性） | ARCHITECTURE.md |
| **阶段5（任务分解）** | 任务拆分方式、优先级排序、依赖关系、每任务的验收标准 | TASK-LIST.md + test-plan.md |

### Brainstorm 讨论规则

1. **了解现状** — 先读项目文件、之前讨论、竞品分析
2. **逐个提问** — 每次只问一个问题，优先给选择题
3. **提出方案** — 给出 2-3 个方向，标注利弊和推荐
4. **分段确认** — 按模块一段一段讲，每段等用户确认再继续
5. **整理文档** — 讨论完毕后整理成正式文档提交 git

### 与评审流程的关系

```
Brainstorm 对话讨论（和用户一起）
    ↓ 讨论结果整理成文档
Pipeline 评审流程（评审 agent 审查文档）
    ↓ 通过后进入下一阶段
```

**Brainstorm 不替代评审** — 讨论确保文档内容正确，评审确保文档质量达标。两者先后关系，互补不重叠。

## Skill 引用容错规则

所有 skill 引用遵循以下容错机制（协调者和各角色 agent 均须遵守）：

1. **尝试加载 skill** — 按引用的路径读取 SKILL.md
2. **skill 存在** → 加载并按其流程执行
3. **skill 不存在或路径变更** → 回退到内置基础流程，不中断流水线

**具体容错行为：**

| Skill | 不存在时的回退 |
|-------|--------------|
| requirements-analysis | 按阶段1的简要说明直接执行（产出PRD.md，包含背景、用户故事、功能清单） |
| tech-architecture | 按阶段2的简要说明直接执行（产出ARCHITECTURE.md，包含技术选型、数据模型、API设计） |
| tech-spike | 按阶段2.8的简要说明直接执行（4h验证，产出SPIKE-REPORT.md） |
| task-decomposition | 按阶段5的简要说明直接执行（产出tasks.md + test-plan.md） |
| qa-workflow | 按阶段8描述的简化流程执行（编译→API冒烟→E2E）|
| code-review-checklist | 使用流水线内置的审查维度表（代码质量+安全+健壮性+去AI味）|
| code-quality-guard | 跳过详细检查，仅做基础编译验证（`mvn compile` / `tsc --noEmit`）|
| run-tests | 手动执行 `mvn test` + `npx playwright test` |
| humanize-code | 跳过去AI味检查，仅做代码质量和安全审查 |
| iterative-contract | 使用简化合同格式（3行），省略完整合同流程 |
| code-review-standard | 使用流水线内置的基础审查维度（代码质量+安全）|

**检查方式（协调者在阶段0执行）：**

```bash
SKILLS_DIR=~/.agents/skills
MISSING=""
for skill in agent-discipline qa-review-workflow requirements-analysis tech-architecture tech-spike task-decomposition qa-workflow code-review-checklist code-quality-guard run-tests humanize-code iterative-contract code-review-standard; do
  [ ! -f "$SKILLS_DIR/$skill/SKILL.md" ] && MISSING="$MISSING $skill"
done
if [ -n "$MISSING" ]; then
  echo "⚠️ 以下 skill 不存在，将使用回退流程：$MISSING"
fi
```

**原则：skill 是锦上添花，不是硬依赖。流水线在任何 skill 缺失的情况下都必须能继续执行。**

### 迭代合同核心规则

**⚠️ 协调者红线：以下规则不可跳过、不可绕过、不可降低等级。**

| 风险等级 | 合同类型 | 最大轮次 | 适用场景 |
|---------|---------|---------|---------|
| 🔴 高风险 | 完整合同（30行） | 3轮 | 安全相关、核心流程、数据一致性 |
| 🟡 中风险 | 标准合同（10行） | 2轮 | 新功能、UI交互、业务逻辑 |
| 🟢 低风险 | 简化合同（3行） | 1轮 | 文案修改、样式调整、文档更新 |

**不确定风险等级时，按高等级执行。**

**合同状态：** `in-progress` / `signed` / `rejected` / `escalated`

**轮次超限处理：** 同阶段达到最大轮次仍未签收 → 标记为 `escalated` → 协调者汇报用户决定继续或回退。**禁止协调者自行决定跳过未签收的阶段。**

**轮次计算：** "最多N轮" = 最多打回+修复+重新评审N次。第1次提交不算轮次，每次打回后重新提交算1轮。

**合同必须包含：** 交付物路径、评审官名称、签收/打回结果。详细格式见 `references/iterative-contract.md`。

**合同生命周期：** 创建 → 产出 → 提交 → 评审 → 签收/打回 → （打回则修改重提，轮次+1）→ 签收归档 → 下一阶段。

## 机制一：回退规则

流水线不是单向前进的。任何阶段发现上游问题，必须立即回退。

### 回退触发条件

| 发现问题的阶段 | 回退目标 | 典型场景 |
|---------------|---------|----------|
| 阶段6 开发 | 阶段2 架构 | 架构设计导致无法实现某个PRD功能 |
| 阶段6 开发 | 阶段1 PRD | 发现PRD遗漏了关键业务逻辑 |
| 阶段7 审查 | 阶段4 UI | 实际UI和设计稿差异大 |
| 阶段8 测试 | 阶段5 分解 | 测试发现任务清单遗漏了边界场景 |
| 阶段8 测试 | 阶段2 架构 | 性能测试发现架构瓶颈 |
| 阶段9 验收 | 任意 | 端到端走查发现任何阶段的问题 |

### 回退流程

```
发现问题 → 严重度评估 → 🔴阻断（立即回退）/ 🟡重要（本批次后回退）/ 🟢建议（记录备查）
回退到对应阶段 → 附带问题描述 → 上游修正 → 评审官复核（1轮）→ 从回退阶段重新走流程
```

### 回退次数限制

| 回退类型 | 最大额度 | 超限处理 |
|---------|---------|---------|
| 阶段间回退 | 总额度≤5次 | 暂停项目，与用户复盘 |
| 同阶段连续回退 | ≤2次 | 第3次升级到协调者+用户决策 |

每次回退在 `.contracts/{project}/changes/` 下记录（发现阶段、回退目标、严重度、问题描述、处理结果、额度消耗）。

## 机制二：端到端用户旅程验证

阶段9验收必须包含完整用户旅程走查。

**原因：** 各模块各自通过测试 ≠ 系统整体OK（模块衔接问题、状态丢失、体验断裂、数据不一致）。

**走查流程：**

协调者不是做重复测试（那是QA阶段8的工作），而是做**系统集成级验收**——验证各模块拼在一起后是否整体可用。

1. **定义核心用户旅程**（2-3个：新用户首次使用、老用户核心功能、异常路径）
2. **逐旅程走查以下维度：**
   - **模块衔接：** 页面跳转是否正常？状态是否丢失？（如：登录后跳首页，用户信息是否带上？）
   - **数据一致性：** 前端展示数据是否和后端一致？（如：列表页计数 vs 详情页数据）
   - **权限边界：** 不同角色的权限隔离是否生效？（如：普通用户能否访问管理员页面？）
   - **错误处理链路：** 任一环节出错，用户是否看到有意义的提示而不是白屏/崩溃？
   - **业务闭环：** 核心业务流程从头走到底是否通？（如：下单→支付→确认→通知）
3. **对比产出物与PRD：** 逐条检查PRD中P0/P1功能的实现状态，确保无遗漏
4. **对比测试报告与实际：** 检查阶段8测试报告中标记的问题是否真的修复了
5. **问题分类：**
   - 模块衔接问题 → 打回开发
   - 数据不一致 → 打回开发
   - 体验断裂 → 记录，由用户决定是否修复
   - 与PRD不符 → 打回对应阶段
6. **结果判定：** 全部走通✅ / 🔴问题修复后重查 / 🟡记录已知问题可交付

**执行方式：** 协调者通过读源码+调API+审查QA测试报告+审查各阶段产出物来完成走查，不依赖Playwright（那是QA的职责）。如需实际运行验证，spawn QA执行。

## 机制四：Agent 间沟通汇报

**所有 agent 之间的每一次沟通都必须记录并汇报给用户。**

### 沟通记录格式

存储在 `.contracts/{project}/communications/` 下，每条记录包含：时间、发送方、接收方、类型（签收/打回/评审/回退）、合同编号、阶段、沟通内容（具体问题描述+修改建议）、期望响应。

### 汇报规则

**核心原则：用户看到的是原始沟通记录，不是协调者的总结。协调者不可信，只信任原文。**

每个节点完成后，协调者必须发送两条：
1. **总结** — 谁→谁、结果（签收/打回+分数）、问题数量（禁止过滤/隐藏/美化问题）
2. **沟通记录原文** — `read` 文件全文，一字不改发送

**禁止：** 只发总结不发原文 / 只发原文不发总结 / 总结里说"没什么问题"但原文有🔴 / 跳过任何沟通记录

## 机制五：防偷懒强制约束

**LLM 会偷懒。必须用强制机制确保每步真实执行。**

### 5.1 文件检查点（不产出文件 = 没执行）
协调者进入下一阶段前必须执行 `scripts/pipeline-check.sh` 验证产出文件存在且行数达标。不执行 = 严重违规。

### 5.2 签收必须附带内容（禁止空签收）
签收报告必须包含：至少3个具体检查过的点、至少1个🟢建议、各维度评分。禁止"看起来没问题，签收"。

### 5.3 交叉评审独立输出（禁止合并为"大家说没问题"）
每个角色必须单独输出评审意见，至少3个检查点。全部通过也要写具体检查过的点。

### 5.4 并发执行的真实性（禁止假并发）
协调者优先 `sessions_send` 同时发出，或 `sessions_spawn` 在同一调用中启动。绝对禁止顺序等待再发下一条。

### 5.5 协调者随机抽查
当一个阶段"全部通过"时，随机选择评审官要求详细说明检查内容。同一项目连续2个阶段无异议，强制第3个阶段抽查。

### 5.6 产出物最小要求

| 产出物 | 最小要求 |
|--------|---------|
| PRD | ≥200行 |
| 架构设计 | ≥100行 |
| UX/UI设计 | ≥80行 |
| 任务清单 | ≥50行 |
| 评审报告 | ≥30行 |
| 交叉评审 | 每角色≥20行 |
| 测试报告 | ≥20行 |

不足20%警告，不足20%打回。

### 5.7 用户随时介入（默认不中断）

**流水线默认全程自主执行，不等待用户确认。** 协调者在每个阶段完成后向用户展示产出和沟通记录，然后自动进入下一阶段。

用户可随时打断：
- "暂停" → 立即暂停
- "看看进展" → 展示当前产出和沟通记录
- "跳过XX" → 记录用户决策并确认影响
- "重做" → 立即回退
- "有问题" → 协调者暂停并等待用户指示

**原则：agent自己做决定、自己推进，用户只看不问。有疑问时用户会主动打断。**
