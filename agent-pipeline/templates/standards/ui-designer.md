# UI 设计审查标准 v2.0

## ⚠️ 关键声明：本清单不是全部

这份标准是你的**最低要求**，不是**全部要求**。

**你必须自由发挥，关注标准之外的任何问题**，包括但不限于：
- 设计里"看起来还行但总觉得哪里怪"的地方
- 用户实际使用时可能遇到的视觉困扰
- 设计是否真的解决了问题还是只是"好看"
- 竞品设计中值得借鉴但原Agent没考虑的细节
- 不同设备/场景下设计是否会崩

**如果只有这份清单里写了的问题才被你发现 = 你不合格。**

## 1. 设计一致性（12%）

**核心问题：同一个产品，看起来像拼凑的还是一体的？**

| ID | 检查项 | 等级 |
|----|--------|------|
| CON-01 | 色彩使用统一：全局主色不超过3个，辅助色不超过5个，不存在游离色 | ❌ |
| CON-02 | 间距系统统一：使用 4/8/12/16/24/32/48 的倍数体系，无随意取值 | ❌ |
| CON-03 | 圆角统一：同类组件（按钮、卡片、输入框、弹窗）圆角一致，层级关系通过圆角大小区分（外层大内层小） | 🔶 |
| CON-04 | 阴影层级统一：最多3级阴影层级，从 elevation-1 到 elevation-3 逐级递增，不存在第4种阴影 | ❌ |
| CON-05 | 字体统一：全文不超过2种字体族，不存在中英文各用不同字体的随意混搭 | ❌ |
| CON-06 | 图标风格统一：同一页面不混用线性和填充图标，图标线宽一致，圆角风格一致 | ❌ |
| CON-07 | 间距节奏一致：相邻同类元素间距相同（如列表项间距、表单项间距），不存在忽大忽小 | 🔶 |
| CON-08 | 边框使用一致：要么统一用 border，要么统一用 divider，同一层级元素不混用两种分隔方式 | 🔶 |
| CON-09 | 按钮样式一致：主/次/幽灵/文字按钮的圆角、内边距、字体大小在全局统一 | ❌ |
| CON-10 | 标题层级一致：H1-H6 各层级字号/字重/间距规则明确且被严格遵守，不存在跳级使用 | ❌ |
| CON-11 | 对齐方式一致：同类内容对齐方式统一（全部左对齐或全部居中），不存在同一区域左对齐和居中混用 | 🔶 |
| CON-12 | 交互反馈一致：同类操作（点击、悬浮、长按）的视觉反馈风格一致，不存在有的地方有有的地方没有 | 🔶 |
| CON-13 | 分割线/留白选其一：同类模块间的分隔方式统一，不会出现左边用分割线右边用留白的混搭 | ⚪ |

## 2. 色彩系统（12%）

**核心问题：色彩不只是好看，还要能用、安全、包容。**

| ID | 检查项 | 等级 |
|----|--------|------|
| CLR-01 | 主色、辅色、功能色（成功/警告/错误/信息）定义完整且语义明确，每种功能色有 background + border + text 变体 | ❌ |
| CLR-02 | 暗色模式完整：不只是反色，而是独立设计色彩方案；暗色模式下文字对比度同样达标 | ❌ |
| CLR-03 | WCAG AA 对比度：正文文字与背景对比度 ≥ 4.5:1，大文字（≥18pt/14pt bold）≥ 3:1 | ❌ |
| CLR-04 | WCAG AAA 对比度（关键场景）：重要信息、错误提示、品牌标识等关键元素 ≥ 7:1 | 🔶 |
| CLR-05 | 色弱友好：红色/绿色不作为唯一信息传达手段（如状态不只用红绿圆点，附加文字或图标） | ❌ |
| CLR-06 | 色彩渐变规范：渐变角度统一（如统一45°或统一品牌角度），渐变起止色在色板内 | 🔶 |
| CLR-07 | 交互态色彩完整：每种颜色都有 hover / active / disabled 状态变体 | ❌ |
| CLR-08 | 背景色层级：至少定义 3 级背景色（页面底色、卡片底色、悬浮层底色），层级关系清晰 | 🔶 |
| CLR-09 | 透明度使用规范：统一使用 0.04 / 0.08 / 0.12 / 0.16 / 0.24 / 0.32 / 0.48 / 0.64 / 0.80 等档位，无随意透明度 | ⚪ |
| CLR-10 | 图片叠加层文字可读：图片上叠文字时，叠加层透明度和模糊值保证文字对比度 ≥ 4.5:1 | ❌ |
| CLR-11 | 不存在过度鲜艳的高饱和色大面积使用（≥30%屏幕面积）导致视觉疲劳 | 🔶 |
| CLR-12 | 超链接色彩可辨识：链接色与周围文字色有明显区分（色盲模拟下仍可区分） | ❌ |
| CLR-13 | 色彩情绪与产品调性匹配：金融类偏稳重冷色，社交类偏活力暖色，工具类偏中性灰色调 | ⚪ |

## 3. 排版规范（10%）

**核心问题：文字是 UI 的灵魂，排版乱了一切都乱。**

| ID | 检查项 | 等级 |
|----|--------|------|
| TYP-01 | 字体层级清晰：至少5级（Display/H1/H2/Body/Caption），每级字号差值合理（1.2~1.5倍递增），不存在相邻两级视觉差异太小 | ❌ |
| TYP-02 | 行高合理：正文行高 1.5~1.75，标题行高 1.2~1.4，不存在行高过大（≥2.0）导致段落松散或行高过小（≤1.2）导致拥挤 | ❌ |
| TYP-03 | 段落间距 ≥ 行高：段落间距大于等于正文行高，视觉上能区分段落 | 🔶 |
| TYP-04 | 中英混排处理：中英文之间有 1/4 em 间距（或通过 font-feature-settings 处理），不存在中文英文紧贴 | 🔶 |
| TYP-05 | 数字排版：数字使用等宽或 tabular-nums 特性，表格/列表中的数字对齐整齐 | 🔶 |
| TYP-06 | 最大行宽限制：正文阅读区最大宽度 640~720px，超宽屏幕不出现一行超100字的段落 | ❌ |
| TYP-07 | 文本对齐：中文正文用左对齐或两端对齐（不用右对齐和居中），英文正文左对齐 | ❌ |
| TYP-08 | 字间距（letter-spacing）：标题可微调（±0.5px），正文字间距为 0，不存在正文字间距过大导致阅读困难 | 🔶 |
| TYP-09 | 文本截断规范：单行截断用省略号 + tooltip 显示全文，多行截断明确行数限制（2-3行），不出现无截断的长文本溢出 | ❌ |
| TYP-10 | 标点挤压：中文标点使用全角，中英文混排时标点不出现在行首 | ⚪ |
| TYP-11 | 列表/表格标题不换行：表头、列表标题文字不换行或换行后对齐美观 | 🔶 |
| TYP-12 | 字体加载策略：有系统字体降级方案，FOIT 闪烁时间 <100ms 或使用 FOUT 策略 | 🔶 |

## 4. 组件规范性（12%）

**核心问题：组件是 UI 的积木，积木不靠谱，建筑就是危房。**

| ID | 检查项 | 等级 |
|----|--------|------|
| CMP-01 | **按钮**：主/次/幽灵/文字/危险 5种变体完整，各有 default/hover/active/focus/disabled/loading 6种状态 | ❌ |
| CMP-02 | **按钮尺寸**：至少 SM/MD/LG 三档，尺寸对应 padding、字号、圆角均明确 | ❌ |
| CMP-03 | **输入框**：支持 default/hover/focus/disabled/error/success 6种状态，error 状态有红色边框+错误文案 | ❌ |
| CMP-04 | **输入框附加元素**：前置图标、后置图标、清除按钮、密码切换、字数统计均有规范 | 🔶 |
| CMP-05 | **选择器**：下拉/多选/搜索下拉/级联/日期选择器，均支持 disabled/error/空状态 | ❌ |
| CMP-06 | **弹窗（Modal）**：有遮罩层、支持 ESC 关闭、点击遮罩可关闭（或明确不可关闭的原因）、焦点锁定、滚动锁定 | ❌ |
| CMP-07 | **抽屉（Drawer）**：支持左/右/上/下四个方向，宽度有 SM/MD/LG 规范，同 Modal 的无障碍要求 | 🔶 |
| CMP-08 | **Toast/通知**：有成功/警告/错误/信息四种类型，自动消失时间合理（3-5秒），支持手动关闭 | ❌ |
| CMP-09 | **空状态**：有插画/图标 + 说明文字 + 操作引导，不是空白或仅"暂无数据" | ❌ |
| CMP-10 | **加载状态**：骨架屏 > 加载动画 > 简单 spinner，避免整页白屏加载 | ❌ |
| CMP-11 | **错误状态**：网络错误/权限不足/404/500 等均有对应页面，有重试/返回/联系方式 | ❌ |
| CMP-12 | **表格**：列宽分配合理，支持横向滚动，空数据/加载中/排序/筛选/分页状态完整 | 🔶 |
| CMP-13 | **标签/徽标（Tag/Badge）**：颜色变体完整，关闭按钮大小足够点击（≥24×24px），溢出处理明确 | 🔶 |
| CMP-14 | **头像**：有占位头像方案（名字首字母/默认图标），不同尺寸圆角一致，在线/离线状态指示清晰 | ⚪ |

## 5. 交互设计（12%）

**核心问题：用户操作后发生了什么？他们知道吗？**

| ID | 检查项 | 等级 |
|----|--------|------|
| INT-01 | 即时反馈：每次用户操作（点击、提交、删除）在 100ms 内有视觉反馈（按钮按下、loading态、toast提示） | ❌ |
| INT-02 | 提交类操作有确认环节：删除、清空、不可逆操作需二次确认弹窗，确认文案明确告知后果 | ❌ |
| INT-03 | 操作可撤销：重要操作（删除、修改）提供撤销机制（undo toast / 软删除 / 撤销按钮） | ❌ |
| INT-04 | 动效自然：过渡动画时长 200-400ms，缓动函数统一（ease-out 用于进入，ease-in 用于退出），不存在生硬的线性动画 | 🔶 |
| INT-05 | 尊重系统动效偏好：`prefers-reduced-motion` 开启时关闭或大幅简化动画 | ❌ |
| INT-06 | 手势交互：移动端支持下拉刷新、左滑删除、长按菜单等原生手势，手势区域足够大（≥44×44px） | 🔶 |
| INT-07 | 键盘可操作：Tab 顺序逻辑合理，Enter/Space 激活按钮，Esc 关闭弹窗，箭头键导航列表/菜单 | ❌ |
| INT-08 | 表单实时验证：输入时/失焦时即时校验，错误提示紧贴对应字段，提交时兜底全量校验 | ❌ |
| INT-09 | 长列表优化：虚拟滚动或分页，避免一次渲染 100+ 条数据导致卡顿 | 🔶 |
| INT-10 | 焦点指示器可见：所有可交互元素有明确的 focus ring，focus ring 不被裁切，颜色与品牌色一致或高对比 | ❌ |
| INT-11 | 加载中禁止重复提交：按钮点击后变为 loading 态并 disabled，防止重复触发 | ❌ |
| INT-12 | 滚动体验：平滑滚动（scroll-behavior: smooth），吸顶/吸底元素不遮挡内容，滚动锚定合理 | 🔶 |
| INT-13 | 拖拽交互（如适用）：有拖拽手柄、拖拽中状态、放置区域高亮、拖拽取消/还原机制 | ⚪ |

## 6. 响应式适配（10%）

**核心问题：不是所有人的屏幕都跟你的一样大。**

| ID | 检查项 | 等级 |
|----|--------|------|
| RSP-01 | 断点体系合理：至少覆盖 Mobile(≤480) / Tablet(≤768) / Desktop(≤1024) / Wide(>1024)，断点值基于内容而非随意设定 | ❌ |
| RSP-02 | 移动端适配：触控目标 ≥ 44×44px，手指不会遮挡关键信息，单手可操作核心功能 | ❌ |
| RSP-03 | 导航适配：移动端使用汉堡菜单或底部 Tab 导航，桌面端使用侧边栏或顶部导航，切换自然 | ❌ |
| RSP-04 | 表格适配：小屏幕下表格支持横向滚动，或转换为卡片列表，关键信息不被截断 | ❌ |
| RSP-05 | 图片响应式：使用 srcset 或响应式图片方案，避免移动端加载桌面尺寸图片 | 🔶 |
| RSP-06 | 横竖屏切换：布局不崩溃，内容不溢出，横屏时不出现大面积空白 | 🔶 |
| RSP-07 | 极端尺寸处理：320px 超小屏不崩、2560px 超宽屏内容不散，最大内容宽度合理（≤1440px） | ❌ |
| RSP-08 | 字体响应式：标题在小屏适当缩小（clamp 或 media query），正文不低于 14px | ❌ |
| RSP-09 | 间距响应式：小屏幕间距适当收紧（但不低于 12px），大屏幕间距适当扩大 | 🔶 |
| RSP-10 | 弹窗/抽屉响应式：移动端弹窗变为底部弹出或全屏，抽屉宽度适配屏幕 | 🔶 |
| RSP-11 | 安全区域适配：iPhone 底部 Home Indicator 区域、刘海屏顶部不遮挡内容（env(safe-area-inset-*)） | ❌ |
| RSP-12 | 键盘弹出适配：移动端输入时页面不被键盘遮挡，输入框可见并自动滚动到可视区 | ❌ |

## 7. 无障碍设计（8%）

**核心问题：所有人都能用你的产品，包括看不见、点不准、听不到的人。**

| ID | 检查项 | 等级 |
|----|--------|------|
| A11Y-01 | 语义化 HTML：使用 header/main/nav/section/article/footer/button/a/input 等语义标签，不全是 div + span | ❌ |
| A11Y-02 | 图片替代文本：所有信息性图片有 alt 属性，装饰性图片 alt=""，不遗漏 | ❌ |
| A11Y-03 | ARIA 标签完整：自定义组件（tabs/dropdown/dialog）有 role + aria-label + aria-expanded + aria-selected 等属性 | ❌ |
| A11Y-04 | 焦点管理：弹窗打开时焦点移入弹窗、关闭时焦点回到触发按钮；Tab 顺序符合阅读顺序 | ❌ |
| A11Y-05 | 键盘完整可操作：不依赖鼠标完成任何操作，包括拖拽有键盘替代方案 | ❌ |
| A11Y-06 | 颜色不作为唯一信息载体：状态、错误、进度等信息通过文字/图标/纹理辅助传达 | ❌ |
| A11Y-07 | 表单关联：每个 input 有对应的 label（for/id 关联或 aria-label），错误信息通过 aria-describedby 关联 | ❌ |
| A11Y-08 | 跳过导航链接：页面顶部有"跳到主内容"链接，屏幕阅读器用户可跳过重复导航 | 🔶 |
| A11Y-09 | 文字可缩放：页面在 200% 缩放下不出现文字重叠、截断、功能不可用 | ❌ |
| A11Y-10 | 屏幕阅读器友好：动态内容（toast/live region）通过 aria-live 通知，隐藏内容用 aria-hidden 而非 display:none | 🔶 |
| A11Y-11 | 链接文本有辨识度：不存在多个"点击这里""了解更多"的链接文本，链接文本能独立传达目的 | 🔶 |

## 8. 设计系统完整性（10%）

**核心问题：设计能不能被工程团队准确还原？能不能持续维护？**

| ID | 检查项 | 等级 |
|----|--------|------|
| DS-01 | Design Token 完整：颜色、字体、间距、圆角、阴影、动画时长均以 token 形式定义，代码和设计稿 token 一一对应 | ❌ |
| DS-02 | 组件文档完整：每个组件有使用说明、Props/属性列表、交互状态说明、使用场景和不推荐用法 | 🔶 |
| DS-03 | 开发者友好：设计稿标注完整（间距/颜色/字号），切图资源齐全（1x/2x/3x），命名规范清晰 | ❌ |
| DS-04 | 场景覆盖：核心用户路径的每一个页面都有设计稿，不存在"开发自己看着办"的页面 | ❌ |
| DS-05 | 组件可组合：组件接口灵活（通过 slot/children 扩展），不存在只能满足一种场景的死板组件 | 🔶 |
| DS-06 | 状态覆盖：每个组件的所有交互状态（default/hover/active/focus/disabled/loading/error/empty）都有设计稿 | ❌ |
| DS-07 | 变更日志：设计系统有版本管理和变更记录，组件改动有影响面评估 | ⚪ |
| DS-08 | 设计和代码的一致性审查：有机制确保开发还原度（设计走查/视觉回归测试），不只是"差不多就行" | 🔶 |
| DS-09 | 命名规范统一：组件/变量/文件命名遵循同一套规范（BEM 或其他），不存在驼峰和短横线混用 | 🔶 |
| DS-10 | 暗色模式 token 完整：不是简单 invert，而是独立的暗色 token 体系，与亮色 token 一一对应 | ❌ |
| DS-11 | 动效规范文档化：缓动函数、时长、类型（进入/退出/强调）有文档，开发可复现 | 🔶 |
| DS-12 | 间距/网格系统可视化：设计稿中可查看网格线和间距标注，开发不需要猜 | ⚪ |

## 9. 品牌一致性（4%）

**核心问题：用户闭眼想你的产品，脑海里浮现的画面是否统一？**

| ID | 检查项 | 等级 |
|----|--------|------|
| BRD-01 | Logo 使用规范：最小尺寸、安全间距、不可拉伸/变形、不可变色（除非规范允许）、不可加背景（除非规范允许） | ❌ |
| BRD-02 | 品牌色贯穿：主品牌色出现在导航、CTA按钮、关键信息提示等核心触点 | 🔶 |
| BRD-03 | 品牌调性统一：整体视觉风格（圆钝vs方正、温暖vs冷峻、活泼vs专业）在所有页面保持一致 | 🔶 |
| BRD-04 | 品牌字体应用：品牌字体用于标题/Logo等品牌触点，非品牌字体用于正文阅读 | ⚪ |
| BRD-05 | 插画风格统一：空状态插画、引导页插画、错误页插画等风格一致（扁平/手绘/3D等） | 🔶 |
| BRD-06 | 图标风格匹配品牌：线性/填充/圆角/直角等图标风格与品牌调性一致 | ⚪ |
| BRD-07 | 品牌元素适度：品牌元素（品牌图案、吉祥物、口号）出现频率适当，不过度刷存在感 | ⚪ |
| BRD-08 | 社交分享图/启动页：品牌展示一致，分辨率和尺寸符合各平台规范 | 🔶 |
| BRD-09 | 加载页/闪屏：品牌展示 ≤2秒，动画流畅，品牌元素完整（Logo + slogan + 加载指示） | 🔶 |
| BRD-10 | 第三方集成页面：微信授权页、支付页等第三方页面的品牌元素不被覆盖或丢失 | ⚪ |

## 10. 去AI味评估（10%）

**核心问题：这个设计是 AI 默认输出的模板感，还是经过思考的定制设计？**

| ID | 检查项 | 等级 |
|----|--------|------|
| DAI-01 | 不存在千篇一律的 Hero Section：大字居中 + 渐变背景 + 3个功能卡片 + CTA 按钮的模板化布局 | ❌ |
| DAI-02 | 配色有个性：不使用默认的紫蓝渐变、Tailwind 默认蓝 #3B82F6、Dribbble 流行色板 | ❌ |
| DAI-03 | 插画不是 Dribbble 通用插画：空状态/引导页插画不是下载的通用素材，有产品特色 | 🔶 |
| DAI-04 | 字体选择有意图：不使用 Inter/Roboto/Poppins 作为唯一字体且无任何品牌考量 | 🔶 |
| DAI-05 | 布局有变化：不是所有页面都是"标题 + 间距 + 卡片列表"的重复结构 | ❌ |
| DAI-06 | 不存在过度精致：圆角过大（≥24px 到处用）、阴影过柔、间距过大导致信息密度过低 | ❌ |
| DAI-07 | 交互有巧思：至少有一个"让人眼前一亮"的交互细节（如有趣的微动效、创意的加载动画） | 🔶 |
| DAI-08 | 页面节奏有变化：信息密度有张有弛，不是每个页面都一样"舒适"到无聊 | 🔶 |
| DAI-09 | 深色模式不是简单反色：暗色方案是独立设计的，有层次感和氛围感 | 🔶 |
| DAI-10 | 不存在无意义装饰：没有"为了好看"但无任何功能的装饰元素（纯装饰圆点、无意义的线条） | ⚪ |
| DAI-11 | 空状态有创意：不使用通用的"空箱子"或"放大镜"插画，有产品语境的空状态设计 | ⚪ |
| DAI-12 | 数据展示有设计感：不只是默认图表库的输出，图表配色/样式与整体设计语言统一 | 🔶 |
| DAI-13 | 存在不完美但有味道的细节：如手写字体标题、略带手感的图标线条、不对称但和谐的布局 | ⚪ |
| DAI-14 | 整体感受：把设计拿给3个不同人看，是否有人说「这看起来像 AI 做的」——如果有人说，就不及格 | ❌ |

## 附录：一眼 AI 套路识别（一票否决）

### 三眼识别法

打开设计稿，花 **3 秒钟** 扫一眼。如果以下任一特征命中，直接否决，不用继续细看。

| 识别眼 | 触发条件 | 否决理由 |
|--------|---------|---------|
| **配色眼** | 紫蓝渐变主色、Tailwind 默认蓝满屏、粉紫配色 | Midjourney/DALL-E 默认色调 |
| **布局眼** | Hero 居中大标题 + 3 列等宽卡片 + CTA 居中 | ChatGPT/Claude 默认输出模板 |
| **质感眼** | 所有圆角 16-24px、所有阴影 ultra soft、所有间距超大 | AI "安全审美"——永不出错但永无个性 |

| EYE-04 | 评价区：3 个带头像的评价卡片，圆角 16px，左侧引号装饰 | ✅ |
| EYE-05 | 底部 CTA：深色背景 + 居中白色标题 + 亮色按钮 + 底部渐变模糊 | ✅ |

### SUS-01 ~ SUS-07：可疑设计模式

| ID | 可疑模式 | 说明 |
|----|---------|------|
| SUS-01 | 所有间距都是 24px/32px/48px 的倍数，无任何"不规则"间距 | 过于规整 = AI 生成 |
| SUS-02 | 每个页面都有恰好 3 个主要功能/特性展示 | "3" 是 AI 最爱的数字 |
| SUS-03 | 所有按钮都是 pill 形状（全圆角） | AI 偏好"安全"的圆角 |
| SUS-04 | 配色方案完美遵循 60-30-10 法则，精确到令人怀疑 | 人不会这么精确 |
| SUS-05 | 所有图片占位符都是 Unsplash/Pexels 的高清摄影 | 无真实产品截图 |
| SUS-06 | 字重只有 400/500/600/700，字间距全部为 0 | 无任何排版个性 |
| SUS-07 | 设计稿中没有任何"不完美"或"出乎意料"的元素 | 人工设计总有不完美 |

---

### DOC-01 ~ DOC-07：文档套路识别

| ID | 文档套路 | 说明 |
|----|---------|------|
| DOC-01 | Design Token 命名全部 `--color-primary-500` 格式，无品牌语义命名 | 照搬 Tailwind/Chakra |
| DOC-02 | 组件文档只有 Props 表格，无使用场景说明和反模式示例 | AI 生成的默认文档格式 |
| DOC-03 | 设计规范文档格式与 Ant Design/Material Design 几乎相同 | 照搬现有系统 |
| DOC-04 | 无任何设计决策记录（为什么选这个颜色/间距/字号） | AI 不会解释"为什么" |
| DOC-05 | 所有设计 Token 都是 8px 基准的倍数，无任何例外 | 过于系统化 |
| DOC-06 | 间距 Token 命名 `space-1/2/3/4/5` 无业务语义 | 纯技术命名无上下文 |
| DOC-07 | 无设计系统治理流程（谁审批、怎么升级、兼容策略） | AI 只生成静态文档 |

## 审查流程

```
1. 三眼识别 → 命中即否决，输出具体触发项
2. EYE 套路扫描 → 任一命中即否决
```
