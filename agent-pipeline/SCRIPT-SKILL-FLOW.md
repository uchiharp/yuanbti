# 脚本-Skill 穿插执行流程

## 执行模型

```
pipeline-run.sh（主控脚本，每阶段驱动）
  │
  ├── 1. 初始化阶段环境（创建目录、合同、配置）
  ├── 2. 输出任务指令 → 协调者读取后调度 agent
  │     └── 指令中包含：产出物规格 + 要加载的 skill + 约束条件
  ├── 3. 等待 agent 完成（协调者轮询产出物）
  ├── 4. 执行检查脚本（验证产出物合规）
  │     ├── 通过 → 推进下一阶段
  │     └── 不通过 → 输出缺失项 → 打回重做（轮次+1）
  └── 5. 调度子脚本（按阶段）
      ├── review-check.sh（阶段6.5/7）
      ├── contract-check.sh（阶段7/9）
      ├── test-report-check.sh（阶段8）
      └── acceptance-check.sh（阶段9）
```

**脚本管"做什么、做多少、做对了没"，skill 管"怎么做好"。**

## 全阶段穿插流程

### 阶段0：启动

```
┌─ pipeline-run.sh ──────────────────────────────────────────┐
│                                                            │
│  [脚本] 创建项目目录结构                                     │
│    mkdir -p .contracts/{project}/{phase-0..9,changes,comm} │
│    初始化 _index.json、_config.json                         │
│    初始化 CONTEXT.md 骨架（所有阶段段落预留标题）              │
│    初始化 health-dashboard.md                               │
│                                                            │
│  [脚本] 输出任务指令：                                       │
│    ┌──────────────────────────────────────────────┐        │
│    │ 任务：评估项目规模 + 产出项目计划              │        │
│    │ 产出：PROJECT-PLAN.md ≥30行                   │        │
│    │ 知识库：写入 CONTEXT.md 阶段0段落              │        │
│    │ 加载 skill：无                                │        │
│    └──────────────────────────────────────────────┘        │
│                                                            │
│  [协调者] 读指令 → 判定规模(🔴/🟡/🟢) → 写 PROJECT-PLAN   │
│           → 更新 CONTEXT.md 阶段0                           │
│           → 加载 MemPalace 历史教训（MCP 调用）              │
│                                                            │
│  [脚本] 检查：                                              │
│    ✓ PROJECT-PLAN.md 存在且 ≥30行                          │
│    ✓ CONTEXT.md 存在且包含"阶段0"段落                       │
│    ✓ health-dashboard.md 存在                              │
│    ✓ _index.json 合法 JSON                                 │
│                                                            │
│  [脚本] 通过 → 自动推进阶段1                                │
│  [脚本] 不通过 → 输出缺失项，协调者补全后重检                 │
└────────────────────────────────────────────────────────────┘
```

**脚本负责：** 目录初始化、产出物规格定义、检查通过/打回
**Skill 负责：** 无（阶段0是纯结构化工作）
**协调者负责：** 规模判定（需要判断力）、MemPalace 加载（需要 MCP）

### 阶段1：需求分析（PM）

```
┌─ pipeline-run.sh ──────────────────────────────────────────┐
│                                                            │
│  [脚本] 输出任务指令：                                       │
│    ┌──────────────────────────────────────────────┐        │
│    │ 任务：产出 PRD                                │        │
│    │ 产出：PRD.md ≥200行                           │        │
│    │ 合同：🟡中风险 标准合同 最多2轮                 │        │
│    │ 知识库：交付时更新 CONTEXT.md 阶段1段落         │        │
│    │              存入 MemPalace room=prd-decisions │        │
│    │ 加载 skill：requirements-analysis              │        │
│    │   （不存在则按内置基础流程执行）                 │        │
│    │ Brainstorm：先和用户讨论，再整理成文档           │        │
│    └──────────────────────────────────────────────┘        │
│                                                            │
│  [协调者] 读指令 → 调度 PM agent                           │
│                                                            │
│  [PM agent] ──────────────────────────────────────         │
│    [skill] 加载 requirements-analysis SKILL.md             │
│             → 获得需求分析方法论：                           │
│               · 用户画像怎么写                              │
│               · 功能优先级怎么排                             │
│               · 验收标准怎么定                               │
│               · 竞品分析框架                                 │
│    [skill] Brainstorm 讨论规则：                            │
│             · 逐个提问，优先给选择题                         │
│             · 给2-3个方向标注利弊                            │
│             · 按模块分段确认                                 │
│    [PM] 和用户讨论 → 整理 PRD.md → 提交评审                 │
│    [PM] 更新 CONTEXT.md 阶段1段落                           │
│    [PM] 存入 MemPalace prd-decisions                       │
│  ─────────────────────────────────────────────────         │
│                                                            │
│  [PM·评审官] 审查 PRD（魔鬼代言人思维）                     │
│    [skill] 无专属 skill（评审官思维模式由 SKILL.md 定义）    │
│             → 魔鬼代言人：需求真实吗？过度设计吗？MVP够小吗？│
│    [评审官] 产出评审意见 → 签收/打回                        │
│                                                            │
│  [脚本] 检查：                                              │
│    ✓ PRD.md 存在且 ≥200行                                  │
│    ✓ 评审报告存在且 ≥3检查点 + ≥1建议 + 评分               │
│      → review-check.sh 验证                                │
│    ✓ 合同状态 = signed（非 in-progress/escalated）          │
│      → contract-check.sh 验证轮次未超限                     │
│    ✓ CONTEXT.md 阶段1段落已更新（日期非空）                 │
│    ✓ communications/ 有本轮评审记录                         │
│                                                            │
│  [脚本] 打回 → 轮次+1，PM修改后重新检查（最多2轮）          │
│  [脚本] 超限 → 标记 escalated，输出指令给协调者汇报用户      │
│  [脚本] 通过 → 推进阶段1.5（如果规模≥🟡）或阶段2            │
└────────────────────────────────────────────────────────────┘
```

**脚本负责：** 产出物规格(≥200行)、检查点数量(≥3)、合同轮次(≤2)、知识库更新验证
**Skill 负责：** 需求分析方法论（怎么写好PRD）、Brainstorm讨论方法（怎么和用户聊）
**评审官思维：** 由 SKILL.md 定义（脚本管不了"怎么挑毛病"，只能管"挑了多少"）

### 阶段1.5：PRD交叉评审（🟡中型：架构+QA，🔴大型：全角色）

```
┌─ pipeline-run.sh ──────────────────────────────────────────┐
│                                                            │
│  [脚本] 输出任务指令：                                       │
│    ┌──────────────────────────────────────────────┐        │
│    │ 任务：交叉评审 PRD                            │        │
│    │ 产出：cross-review-pm.md                     │        │
│    │   每个参与角色独立段落 ≥20行                   │        │
│    │   每段 ≥3检查点                               │        │
│    │ 时限：24h（脚本记录起始时间）                   │        │
│    │ 加载 skill：无                                │        │
│    └──────────────────────────────────────────────┘        │
│                                                            │
│  [脚本] 记录起始时间戳到 _config.json                       │
│                                                            │
│  [协调者] 并发调度各角色 agent（sessions_send 同时发出）     │
│                                                            │
│  [各角色 agent] ──────────────────────────────────         │
│    [skill] 无（各角色用自己的专业视角评审，                  │
│             脚本规定了"评审多少"，不需要skill教"怎么评"）     │
│    [架构师] 技术可行性？性能隐患？                           │
│    [QA] 验收标准具体？能写测试用例？                         │
│    [开发] 工作量合理？边界遗漏？                             │
│    [UI/UX] 交互矛盾？页面复杂度？                            │
│    → 各自输出独立段落                                       │
│  ─────────────────────────────────────────────────         │
│                                                            │
│  [PM] 汇总意见 → 修订 PRD → PM·评审官复核（1轮）           │
│                                                            │
│  [脚本] 检查：                                              │
│    ✓ cross-review-pm.md 存在                               │
│    ✓ 每个参与角色有独立 ## 段落                              │
│    ✓ 每段 ≥3 检查点（🔴/🟡/🟢/编号列表/列表项）            │
│      → review-check.sh 验证交叉评审格式                     │
│    ✓ 24h时限未超（对比 _config.json 起始时间）              │
│    ✓ 修订后的 PRD.md 行数仍 ≥200                            │
│    ✓ communications/ 有评审记录                             │
│                                                            │
│  [脚本] 通过 → 推进阶段2                                    │
└────────────────────────────────────────────────────────────┘
```

**脚本负责：** 参与角色清单、每角色独立段落、检查点数量、时限检查
**Skill 负责：** 无（专业视角是 agent 自身能力，不需要方法论指导）
