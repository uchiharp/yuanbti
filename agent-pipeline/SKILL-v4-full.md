# Agent 开发流水线 v4

## 核心理念

**不是让一个agent做完所有事，而是让一群专业agent各司其职、互相评审、迭代交付。**

每个环节都有"对手"来挑战产出质量。从PRD到代码，从设计到测试，每一步交付都经过专业评审官的严格审查。

### v4 新增特性
1. **项目规模裁剪** — 大/中/小型项目走不同流程长度，避免小项目过重
2. **技术Spike（阶段2.8）** — 高风险技术点前置验证，4h时限
3. **代码集成（阶段6.3）** — 多Agent并发开发后的合并与冲突解决
4. **异步交叉评审24h时限** — 交叉评审参与方有24h窗口期
5. **三级测试数据管理** — L1种子数据 / L2快照数据 / L3真实设备
6. **项目健康度看板** — 每阶段更新，可视化项目状态
7. **CI/CD集成**（可选）— GitHub Actions 模板，可跳过
8. **文件头YAML标记** — version 字段标识版本

## 项目规模裁剪（v4新增）

不同规模项目走不同流程长度，避免"杀鸡用牛刀"。

### 规模判定标准（阶段0由协调者判定）

| 规模 | 判定条件 | 示例 |
|------|---------|------|
| 🔴 大型 | 功能≥10、涉及≥3个子系统、预估工期≥2周 | 电商系统、社交App |
| 🟡 中型 | 功能5-9、1-2个子系统、工期1-2周 | 后台管理、小程序 |
| 🟢 小型 | 功能≤4、单子系统、工期<1周 | 工具页、H5活动页 |

### 规模差异表

| 阶段 | 🔴 大型 | 🟡 中型 | 🟢 小型 |
|------|---------|---------|---------|

**关键并发点：**
- 阶段3（UX）和 阶段2（架构）的**后半段可以并发**（UX设计用户流程不需要等API设计完）
- 阶段4（UI）依赖阶段3（UX）签收
- 阶段6 的多个开发Agent可以并发
- 阶段6.3 集成在阶段6完成后执行
- 阶段6.5 的多个reviewer可以并发审查不同Agent的代码
- 阶段7 的多个评审官可以并发审查不同Agent的代码

## Skill 引用容错规则

所有 skill 引用遵循以下容错机制（协调者和各角色 agent 均须遵守）：

1. **尝试加载 skill** — 按引用的路径读取 SKILL.md
2. **skill 存在** → 加载并按其流程执行
3. **skill 不存在或路径变更** → 回退到内置基础流程，不中断流水线

**具体容错行为：**

| Skill | 不存在时的回退 |
|-------|--------------|
| qa-workflow | 按阶段8描述的简化流程执行（编译→API冒烟→E2E）|
| code-review-checklist | 使用流水线内置的审查维度表（代码质量+安全+健壮性+去AI味）|
| code-quality-guard | 跳过详细检查，仅做基础编译验证（`mvn compile` / `tsc --noEmit`）|
| run-tests | 手动执行 `mvn test` + `npx playwright test` |
| humanize-code | 跳过去AI味检查，仅做代码质量和安全审查 |

**检查方式（协调者在阶段0执行）：**

```bash
# 阶段0启动时检查所有依赖 skill 是否存在
SKILLS_DIR=~/.agents/skills
MISSING=""
for skill in qa-workflow code-review-checklist code-quality-guard run-tests humanize-code; do
  [ ! -f "$SKILLS_DIR/$skill/SKILL.md" ] && MISSING="$MISSING $skill"
done

if [ -n "$MISSING" ]; then
  echo "⚠️ 以下 skill 不存在，将使用回退流程：$MISSING"
  # 记录到 PROJECT-PLAN.md 的风险预案中
fi
```

**原则：skill 是锦上添花，不是硬依赖。流水线在任何 skill 缺失的情况下都必须能继续执行。**

**为什么要开发审查：**
QA评审官和架构评审官（阶段7）的视角是「业务验证+架构合规」，确保代码实现符合PRD需求且架构方案正确。阶段6.5由各开发者的专属reviewer负责代码质量和架构对齐的建设性审查。

**职责边界（重要）：**
- ✅ reviewer 负责：代码质量、安全扫描、健壮性检查、去AI味
- ❌ reviewer 不做：**Playwright E2E 测试** — 这是 QA评审官（阶段7）和 QA（阶段8）的职责
- 原因：Playwright 是浏览器级 E2E 测试工具，需要启动前后端服务、管理浏览器实例、维护测试数据，属于 QA 领域。reviewer 的核心能力是代码审查，不应混入测试执行。

**审查维度（全面覆盖）：**

**A. 代码质量（建设性审查）**

| 维度 | 检查内容 |
|------|----------|
| 代码风格 | 命名规范、注释质量、代码格式是否一致 |
| 架构对齐 | 代码实现是否符合架构设计方案（技术选型、分层、接口定义）|
| 逻辑正确性 | 业务逻辑是否正确实现、是否有遗漏的分支 |
| 边界条件 | null/空值/异常输入是否处理 |
| 可维护性 | 代码是否可读、是否过度耦合、是否方便后续扩展 |
| 代码复用 | 是否有重复代码可以抽取公共方法 |

**B. 安全扫描（对抗性审查）**

| 检查项 | 说明 |
|--------|------|
| XSS | 用户输入是否转义，是否存在注入点 |
| SQL注入 | 是否使用参数化查询 |
| 越权访问 | 水平越权（访问他人资源）/ 垂直越权（普通用户执行管理员操作）|
| 敏感信息泄露 | 日志/错误信息/响应中是否暴露内部实现 |
| 认证安全 | Token/Session 管理是否安全 |

**C. 健壮性测试**

| 检查项 | 说明 |
|--------|------|
| 异常输入 | 超长字符串、特殊字符、负数、非法格式 |
| 并发安全 | 竞态条件、乐观锁/悲观锁使用 |
| 资源管理 | 连接泄露、文件未关闭、内存溢出风险 |
| 降级处理 | 外部服务不可用时的 fallback |

| 登录-功能 | P0 | ✅ 已测试 | ✅ |
| 登录-边界 | P0 | ✅ 已测试 | ✅ |
| 设备管理-E2E | P1 | ✅ 已测试 | ✅ |
| 性能-并发 | P2 | ❌ 未测试 | ❌ |

功能测试维度覆盖率 = P0+P1已测试数 / P0+P1总数 × 100%

**三级测试数据管理：**

| 级别 | 名称 | 说明 | 适用规模 |
|------|------|------|---------|
| L1 | 种子数据 | 标准测试用例+基础场景（用户/订单/商品等）| 🟢🟡🔴 全部 |
| L2 | 快照数据 | 数据库快照+API响应录制回放，验证数据一致性 | 🟡🔴 中大型 |
| L3 | 真实设备 | 真机浏览器测试，覆盖不同OS/分辨率/网络条件 | 🔴 大型 |

**性能基准：**
- API响应 < 200ms
- 首屏加载 < 2s
- 100并发通过

**通过 → 进入阶段9 | 失败 → 打回开发Agent → 回到阶段6 | 最多3轮迭代**

**测试报告产出：** `test-report.md`（L1/L2/L3各自结果、性能基准对比、覆盖率对比）

**质量保障三层防线（v4修正17新增）：**
1. **防线1 — 开发者自检**（阶段6）：编码时按 `code-quality-guard` 检查
2. **防线2 — 冒烟验证**（阶段6后）：创业助手执行 P0 冒烟，拦截阻断性问题
3. **防线3 — 全量测试**（阶段8）：QA 执行完整测试维度，确保质量达标

### 迭代合同核心规则

**⚠️ 协调者红线：以下规则不可跳过、不可绕过、不可降低等级。**

#### 风险分级

| 风险等级 | 合同类型 | 最大轮次 | 适用场景 |
|---------|---------|---------|---------|
| 🔴 高风险 | 完整合同（30行） | 3轮 | 安全相关、核心流程、数据一致性 |
| 🟡 中风险 | 标准合同（10行） | 2轮 | 新功能、UI交互、业务逻辑 |
| 🟢 低风险 | 简化合同（3行） | 1轮 | 文案修改、样式调整、文档更新 |

**不确定风险等级时，按高等级执行。**

#### 阶段默认风险等级

| 阶段 | 默认风险 | 原因 |
|------|---------|------|
| 0. 启动 | 🟢 | 用户确认即可 |
| 1. PRD | 🟡 | 核心需求但通常不涉及安全 |
| 2. 架构 | 🔴 | 影响全局，技术选型错误代价大 |
| 2.8. Spike | 🟡 | 验证性质，非正式交付 |
| 3. UX | 🟡 | 交互影响用户体验 |
| 4. UI | 🟢 | 视觉可快速迭代 |
| 5. 任务分解 | 🟡 | 分解遗漏导致开发返工 |
| 6. 开发 | 🔴 | 代码质量直接影响产品 |
| 6.3. 代码集成 | 🟡 | 集成风险可控 |
| 6.5. 开发审查 | 🟡 | reviewer并行审查，代码质量和架构对齐 |
| 7. 代码审查 | 🔴 | 业务验证+架构合规+可读性 |
| 8. 测试 | 🔴 | 质量把关 |
| 1.5/2.5 交叉评审 | 🟡 | 评审意见，不直接产出 |
| 3.5/4.5 交接评审 | 🟢 | 快速确认 |

#### 合同状态

每份合同必须有且只有一个状态：
- `in-progress` — 进行中
- `signed` — 签收通过
- `rejected` — 打回，等待修改
- `escalated` — 超限，升级仲裁

#### 轮次超限处理

- 同阶段达到最大轮次仍未签收 → 合同状态标记为 `escalated`
- 协调者汇报用户，由用户决定：继续（加1轮）或回退到上游阶段
- **禁止协调者自行决定跳过未签收的阶段**

#### 合同必须包含的内容

即使是最简化的 🟢 合同（3行），也必须包含：
1. 交付物路径
2. 评审官名称
3. 签收/打回结果

标准合同（10行）和完整合同（30行）的详细格式见 `references/iterative-contract.md`。

#### 合同生命周期

```
1. 协调者创建合同（基于风险等级选择模板）
2. 原角色完成产出 → 提交合同
3. 评审官审查 → 签收或打回（附具体意见）
4. 打回 → 原角色修改 → 重新提交（轮次+1）
5. 签收 → 合同归档 → 进入下一阶段
```

## 机制一：回退规则

流水线不是单向前进的。任何阶段发现上游问题，必须立即回退。

### 回退触发条件

| 发现问题的阶段 | 回退目标 | 典型场景 |
|---------------|---------|----------|
| 阶段6 开发 | 阶段2 架构 | 架构设计导致无法实现某个PRD功能 |
| 阶段6 开发 | 阶段1 PRD | 发现PRD遗漏了关键业务逻辑 |
| 阶段7 审查 | 阶段4 UI | 实际UI和设计稿差异大，需要重新设计 |
| 阶段8 测试 | 阶段5 分解 | 测试发现任务清单遗漏了边界场景 |
| 阶段8 测试 | 阶段2 架构 | 性能测试发现架构瓶颈 |
| 阶段9 验收 | 任意 | 端到端走查发现任何阶段的问题 |

### 回退流程

```
发现上游问题
  │
  ├─ 问题严重度评估
  │   ├─ 🔴阻断（无法继续）→ 立即回退
  │   ├─ 🟡重要（质量隐患）  → 记录，本批次完成后回退
  │   └─ 🟢建议（可优化）   → 记录备查，不回退
  │
  ▼ 立即回退
回退到对应阶段
  │
  ├─ 附带问题描述和发现阶段
  ├─ 上游角色修正
  ├─ 对应评审官快速复核（1轮）
  ├─ 从回退阶段重新走流程到当前阶段
  └─ 更新合同记录（标记回退原因和次数）
```

### 回退次数限制（v4修正9：明确定义）

| 回退类型 | 定义 | 最大额度 | 超限处理 |
|---------|------|---------|---------|
| 阶段间回退 | 从阶段A回退到阶段B（跨阶段） | 消耗1次额度，总额度≤5次 | 超限暂停项目，与用户复盘 |
| 阶段内评审打回 | 同一阶段内评审官打回，角色修改后重新提交 | **不消耗额度** | 按迭代合同轮次限制处理 |
| 同阶段连续回退 | 同一阶段被回退后再次回退 | 同阶段连续回退≤2次 | 第3次升级到协调者+用户决策 |

**回退额度追踪（健康度看板维护）：**
- 总额度：5次
- 已消耗：N次
- 剩余：5-N次

### 回退记录

每次回退在 `.contracts/{project}/changes/` 下记录：

```markdown
## 回退记录 #{N}
- **发现阶段：** 阶段7（代码审查）
- **回退目标：** 阶段2（架构设计）
- **严重度：** 🔴 阻断
- **问题描述：** 分库分表策略导致联合查询无法实现，PRD要求的多条件筛选功能受限
- **发现者：** 架构评审官
- **处理结果：** 架构师调整为单库+读写分离方案
- **复核结果：** 架构师·评审官签收
- **额度消耗：** 1/5
```

## 机制二：端到端用户旅程验证

阶段9验收不只是汇总合同，还必须包含一个**完整用户旅程走查**。

### 为什么要做

各模块各自通过测试 ≠ 系统整体OK。可能存在：
- 模块A的输出格式 ≠ 模块B期望的输入格式
- 用户在A页面完成的操作，到B页面丢失了状态
- 每个页面的错误处理都OK，但串联起来用户被困住了

### 走查流程

```markdown
## 端到端用户旅程验证

### 1. 定义核心用户旅程
根据PRD中的用户故事，选择2-3个核心旅程：
- 旅程A：新用户注册 → 首次使用 → 完成关键操作
- 旅程B：老用户登录 → 使用核心功能 → 获得结果
- 旅程C（如适用）：异常路径（断网、超时、权限不足）

### 2. Playwright自动化走查
由QA评审官用Playwright按用户旅程逐步操作：
- 每一步截图
- 每一步记录实际结果 vs 预期结果
- 记录任何不一致

### 3. 问题分类
| 类型 | 说明 | 处理 |
|------|------|------|
| 模块衔接问题 | A的输出≠B的输入 | 打回开发修复 |
| 状态丢失 | 页面切换后数据丢失 | 打回开发修复 |
| 体验断裂 | 步骤之间缺少引导/过渡 | UI/UX评估是否需要改 |
| 数据不一致 | 不同页面显示的数据矛盾 | 打回开发修复 |

### 4. 结果判定
- 所有核心旅程全部走通 → ✅ 通过
- 存在🔴问题 → ❌ 修复后重新走查
- 存在🟡问题 → 记录为已知问题，可交付
```

## 机制三：需求变更管理

开发过程中用户可能提出需求变更。不能打补丁，必须有流程。

### 变更分级

| 变更类型 | 影响范围 | 处理方式 |
|---------|---------|---------|
| 🟢 小调整 | 不影响已有设计/代码 | 当前阶段内部消化 |
| 🟡 功能调整 | 影响部分设计，不改PRD核心 | 走快速变更流程 |
| 🔴 需求变更 | 改变PRD核心或增加重要功能 | 回退到阶段1重走流程 |

### 快速变更流程（🟡）

```
用户提出变更
  │
  ├─ 协调者评估变更影响
  │   ├─ 涉及哪些阶段？
  │   ├─ 已完成的哪些产出需要修改？
  │   ├─ 新增工作量多大？
  │
  ▼ 快速变更
PM更新PRD（只改受影响的部分）
  │
  ├─ 受影响的角色确认影响范围
  ├─ 各角色快速修改自己的产出
  ├─ 对应评审官复核修改部分（1轮）
  └─ 继续原流程
```

### 回退变更流程（🔴）

```
用户提出重大变更
  │
  ├─ 协调者 + 用户讨论
  │   ├─ 这个变更值得重做吗？
  │   ├─ 能不能放到V2？
  │   ├─ 如果重做，哪些阶段需要重走？
  │
  ▼ 用户确认重做
回退到阶段1
  │
  ├─ PM基于新需求重写PRD
  ├─ 保留已有的有价值产出（标注"基于旧需求"）
  ├─ 按正常流程从阶段1重新走
  └─ 在合同记录中标记为"需求变更v2"
```

### 变更记录

每次变更在 `.contracts/{project}/changes/` 下记录：

```markdown
## 变更记录 #{N}
- **时间：** YYYY-MM-DD HH:mm
- **变更类型：** 🟡 功能调整
- **变更描述：** 用户要求在注册流程中增加手机号验证
- **影响范围：** PRD（功能#1）、架构（需新增短信服务）、任务分解（新增子任务）
- **受影响阶段：** 1、2、5、6
- **处理方式：** 快速变更
- **当前进度：** PRD已更新，架构师确认新增短信服务可行
```

## 机制四：Agent 间沟通汇报

**所有 agent 之间的每一次沟通都必须记录并汇报给用户。用户拥有完全的知情权。**

### 什么是"agent间沟通"

以下场景都算 agent 间沟通，必须记录：
- 评审官签收 / 打回 + 签收理由或打回原因
- 交叉评审中各角色的评审意见
- 开发 Agent 收到打回后的修改方案
- 评审官复核结果
- 回退事件（谁发现、回退到哪、原因）
- 需求变更讨论
- 协调者的调度决策

### 沟通记录格式

每次沟通产出一个记录，存储在 `.contracts/{project}/communications/` 下：

```markdown
## 沟通记录 #{N}
- **时间：** YYYY-MM-DD HH:mm
- **发送方：** PM评审官
- **接收方：** PM
- **类型：** 打回
- **合同编号：** c02
- **阶段：** 阶段1（PRD评审）

### 沟通内容
🟡 建议2条，🔴 阻断0条

| # | 严重度 | 问题描述 | 修改建议 |
|---|--------|---------|----------|
| 1 | 🟡 | 验收标准AC-001-2缺少密码强度边界 | 补充：密码8-20位，必须包含字母+数字 |
| 2 | 🟡 | MVP范围偏大，"第三方登录"应为P1 | 从must-have移到nice-to-have |

### 期望响应
PM需要在本轮迭代中解决以上2个🟡问题。
```

### 汇报规则

**核心原则：用户看到的是原始沟通记录，不是协调者的总结。协调者不可信，只信任原文。**

每个节点（签收/打回/交叉评审/回退）完成后，协调者必须：

1. **直接发送沟通记录文件的完整内容**给用户（用 `read` 读取文件全文发送）
2. **不做精简、不做汇总、不写总结**
3. **附上文件路径**，用户可以随时自己查看

**协调者在每个节点完成后必须发送两条内容：**

1. **总结** — 协调者写的简短概述（谁→谁、结果、下一步）
2. **沟通记录原文** — 用 `read` 读取文件全文，一字不改地发送

**发送格式：**
```
📋 {阶段} | {节点} | 总结
━━━━━━━━━━━━━━━━━━━
PM评审官打回PRD v1，评分6.5/10，2个🟡+1个🟢。
PM已收到，开始修改。
下一步：PM修订后重新提交。
━━━━━━━━━━━━━━━━━━━

📋 {阶段} | {节点} | 沟通记录原文
📁 文件：.contracts/{project}/communications/comm-002.md
━━━━━━━━━━━━━━━━━━━
（read 文件全文，一字不改）
━━━━━━━━━━━━━━━━━━━
```

**总结的规则：**
- 总结只是索引，不能替代原文
- 总结必须包含：谁→谁、结果（签收/打回+分数）、问题数量
- **禁止在总结中过滤、隐藏或美化问题**
- 总结中提到的每个问题，在原文中必须能找到对应

**禁止：**
```
❌ 只发总结不发原文
❌ 只发原文不发总结
❌ 总结里说"没什么问题"但原文里有🔴
❌ 总结里漏掉了原文中的关键问题
❌ 跳过任何沟通记录不发
```

### 协调者发送格式

每个节点发两条：先总结，后原文。

**总结部分**（协调者写）：
```

下一步：PM修订后重新提交。
━━━━━━━━━━━━━━━━━━━
```

## 机制五：防偷懒强制约束

**LLM 会偷懒。必须用强制机制确保每个步骤真实执行，不允许跳过、合并、空签收。**

### 5.1 文件检查点（不产出文件 = 没执行）

每个阶段必须产出指定文件。**协调者进入下一阶段前，必须执行检查脚本验证。**

检查脚本：`scripts/pipeline-check.sh`（位于 agent-pipeline skill 目录）

```bash
# 协调者在每个阶段完成后必须执行
bash ~/.agents/skills/agent-pipeline/scripts/pipeline-check.sh ".contracts/{project}" "{阶段号}"
# 退出码 0 = 通过，1 = 不通过（流程暂停）
```

**⚠️ 协调者红线：不执行检查脚本就进入下一阶段 = 严重违规。**

| 阶段 | 必须产出的文件 | 最小行数 |
|------|---------------|----------|
| 0. 启动 | `PROJECT-PLAN.md` | 30行 |
| 1. PRD | `PRD.md` | 200行 |
| 1.5. PRD交叉评审 | `cross-review-pm.md` | 50行 |
| 2. 架构设计 | `ARCHITECTURE.md` | 100行 |
| 2.5. 架构交叉评审 | `cross-review-arch.md` | 40行 |
| 2.8. Spike | `spike-report.md` | 30行 |
| 3. UX设计 | `UX-DESIGN.md` | 80行 |
| 3.5. UX→UI交接 | `cross-review-ux-to-ui.md` | 20行 |
| 4. UI设计 | `UI-DESIGN.md` | 80行 |
| 4.5. UI→UX反向 | `cross-review-ui-to-ux.md` | 20行 |
| 5. 任务分解 | `TASK-LIST.md` | 50行 |
| 5.5. 分解确认 | `confirm-tasks.md` | 20行 |
| 6. 开发 | `dev/` 下有实际代码文件 | ≥1个 |
| 6.3. 代码集成 | `integration-report.md` | 20行 |
| 6.5. 开发审查 | `cross-review-dev.md` | 60行（每个reviewer≥20行）|
| 7. 代码审查 | `review-report.md` + `screenshots/` | 30行+截图 |
| 8. 测试 | `test-report.md` | 20行 |
| 9. 验收 | `ACCEPTANCE-REPORT.md` | 50行 |

**脚本自动检查的内容：**
- 文件是否存在且非空
- 文件行数是否达标（不足20%报错，不足20%报警告）
- 阶段7额外检查：审查报告是否包含≥3个检查点（防空签收）、截图是否存在
- 通用检查：沟通记录是否存在

**文件不存在的处理：**
1. 脚本返回退出码1 → 协调者必须暂停
2. 协调者汇报给用户："❌ 阶段{N}检查不通过，缺少文件 XXX"
3. 要求对应角色重新执行
4. 重新执行后再次运行脚本验证
5. 连续2次验证失败 → 汇报给用户介入

### 5.2 签收必须附带内容（禁止空签收）

**以下情况视为空签收，判定为不合格：**

| ❌ 空签收 | ✅ 合格签收 |
|-----------|------------|
| "看起来没问题，签收" | "检查了3个API端点的参数格式、错误码和鉴权方式，均符合架构设计规范。🟢建议：错误码可以考虑统一枚举类" |
| "PRD写得很好，通过" | "逐条检查了10个AC，其中8个可直接执行，2个🟡建议已在签收报告中标注。评分：功能完整度8/10" |
| "代码没有问题" | "Playwright走了3个核心流程，截图见screenshots/目录。安全扫描通过。🟡1条：注册表单缺少XSS转义" |
| "大家都没有意见" | 每个角色单独列出了3个检查过的具体点 |

**在所有评审官的 SOUL.md 中加入红线：**

```
⚠️ 红线（违反 = 你不合格）：

1. 禁止空签收。签收报告必须包含：
   - 至少3个你具体检查过的点（写明文件名/行号/功能点）
   - 至少1个🟢建议或观察（即使你觉得完美也要写）
   - 各维度评分（不能只给总分）

2. 禁止笼统打回。打回报告必须包含：
   - 每个问题的具体位置（文件名+行号/章节号）
   - 具体的修改建议（不能只说"需要修改"）
   - 预期修改后的效果

3. 禁止跳过检查项。REVIEW-STANDARDS.md 中的检查项不是建议，是必须逐条检查的。
   如果你因为token限制没检查完，必须在报告中说明哪些没检查到。

4. 禁止合并步骤。阶段3.5、4.5、5.5等"小步骤"不是可选的，必须独立执行并产出文件。
```

### 5.3 交叉评审独立输出（禁止合并为"大家说没问题"）

**错误示例：**
```markdown
## 交叉评审汇总
5个角色评审完成，均无异议。
```

**正确要求：** 每个角色必须单独输出，格式如下：

```markdown
## 架构师的评审意见

### 检查了以下具体点（至少3个）：
1. SSE实时推送方案 → PRD §F008 → 技术可行但需要区分H5和小程序
2. AI识别超时策略 → PRD §F008 异常流程 → 只说3分钟超时，没说部分结果保留
3. 图片压缩策略 → PRD全文 → 完全没有提到

### 评审结论：🟡 3条建议

| # | 严重度 | 具体问题 | 修改建议 |
|---|--------|---------|----------|
| 1 | 🟡 | SSE需区分平台实现 | PRD §F008补充H5用EventSource，小程序用uni.request |
| 2 | 🟡 | AI超时后部分结果处理 | 补充AC：超时后保留已识别结果，提供重新识别按钮 |
| 3 | 🟡 | 缺少图片压缩策略 | 补充：上传前压缩到500KB以内 |
```

**如果某个角色确实没有发现问题（全部通过）：** 也要写3个具体检查过的点 + 1个🟢观察。不能写"没有意见"就结束。

### 5.4 并发执行的真实性（禁止假并发）

**假并发的表现：** 声称3个开发Agent"并发"执行，但实际上是顺序执行然后用`sessions_spawn`同时发出去的。

**真并发的验证：**
- 协调者优先使用 `sessions_send` 发给固化 agent；一次性任务使用 `sessions_spawn` 同时启动多个 agent（不是顺序 spawn）
- 每个 agent 的启动时间戳应该接近（差异<30秒）
- 每个 agent 的产出文件独立
- 沟通记录中可以看到多条并发的签收/打回

**协调者红线：**
```
⚠️ 并发要求：
- 固化 agent 使用 `sessions_send` 在同一消息中同时发出
- 一次性任务使用 `sessions_spawn` 在同一调用中同时启动
- 或者在一个消息中同时发出多个 sessions_send / sessions_spawn 调用
- 绝对禁止：send A → 等 A 完成 → send B → 等 B 完成（这是串行不是并发）
```

### 5.5 协调者随机抽查

**当一个阶段"全部通过"时（评审官签收、交叉评审无异议），协调者不能盲目信任。**

**抽查规则：**
- 随机选择一个评审官/角色，要求它详细说明"你具体检查了什么"
- 如果回答笼统（"我检查了整体质量"），要求补充具体检查点
- 如果同一项目中连续2个阶段"全部无异议"，强制要求第3个阶段增加抽查

**抽查命令（协调者发给被抽查的agent）：**
```
📋 质量抽查 | 阶段{N}
━━━━━━━━━━━━━━━━━━━
本次抽查对象：{角色名}
请回答：
1. 你评审的文件路径是什么？
2. 你逐条检查了REVIEW-STANDARDS.md中的哪些检查项？（列出编号）
3. 你的签收报告中提到的3个检查点分别对应标准中的哪些条目？
4. 有没有因为token限制而跳过的检查项？如果有，是哪些？
━━━━━━━━━━━━━━━━━━━
```

### 5.6 产出物最小要求

**防止 agent 用"一句话文件"敷衍。**

| 产出物 | 最小要求 | 检查方式 |
|--------|---------|----------|
| PRD | ≥200行，包含目录+用户故事+AC | `wc -l` |
| 架构设计 | ≥100行，包含技术选型+数据模型+API | `wc -l` |
| UX设计 | ≥80行，包含用户流程+信息架构+交互规范 | `wc -l` |
| UI设计 | ≥80行，包含视觉规范+组件+响应式 | `wc -l` |
| 任务清单 | ≥50行，包含任务表+依赖+风险 | `wc -l` |
| 评审报告 | ≥30行，包含具体检查点+问题+评分 | `wc -l` |
| 交叉评审意见 | 每个角色≥20行 | `wc -l` |
| 测试报告 | ≥20行，包含测试用例+结果 | `wc -l` |

**行数不足的处理：**
- 轻微不足（差<20%）→ 🟡 警告，要求补充
- 严重不足（差>20%）→ ❌ 打回重做

### 5.7 用户随时介入

**用户是最终决策者，有权在任何阶段介入。**

- 用户说"暂停" → 立即暂停当前阶段，汇报当前状态
- 用户说"看看XX阶段做了什么" → 展示该阶段的产出文件和沟通记录
- 用户说"我觉得XX有问题" → 记录用户意见，根据严重度决定是否触发回退
- 用户说"跳过XX步骤" → 协调者记录用户决策，但必须向用户确认影响
- 用户说"这个阶段做得不行，重来" → 立即回退到该阶段，重新执行

## CI/CD集成附录（v4新增，可选）

> ⚠️ 本节为可选项。仅当项目使用 GitHub/GitLab CI 时启用。本地开发项目可跳过。

### GitHub Actions 模板

```yaml
# .github/workflows/pipeline.yml
name: Agent Pipeline CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run linter
        run: npm run lint

  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: npm run test

  security-scan:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Security scan
        run: bash ~/.agents/skills/agent-pipeline/scripts/security-scan.sh ./src
```

### 集成时机
- 阶段6.3（代码集成）后自动触发CI
- 阶段8（测试验证）结果同步到CI报告
- 阶段9（验收）通过后可自动部署
