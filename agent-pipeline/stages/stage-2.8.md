# 阶段 2.8：技术 Spike（可行性验证）

## 任务
针对架构设计中识别出的高风险技术点，进行快速原型验证，确认技术方案可行后再进入任务分解。

## 角色
- Spike Agent：执行技术验证（可以是架构师或指定开发）
- 协调者：调度、收集结论、决定是否回退

## 执行流程
1. 读取 ARCHITECTURE.md 中标记为「高风险」或「需验证」的技术点
2. 对每个风险点执行 Spike 验证：
   - 搭建最小可运行原型
   - 验证关键技术 API/库/框架的可行性
   - 记录验证结论（通过/不通过/有条件通过）
3. 产出 SPIKE-REPORT.md，包含：
   - 验证的技术点清单
   - 每个点的验证方法、代码片段、结论
   - 发现的新风险或约束
4. 结论路由：
   - 全部通过 → 推进阶段 5
   - 部分不通过 → 回退阶段 2，针对性调整架构
   - 全部不通过 → 回退阶段 2，重新设计

## 产出物
| 文件 | 说明 |
|------|------|
| pipeline/2.8/SPIKE-REPORT.md | 技术验证报告 |
| pipeline/2.8/spike-*/ | 各 Spike 原型代码目录 |

## 约束
- Spike 时间上限 4 小时，超时视为不通过
- 只验证关键技术点，不做完整实现
- 验证代码不合并到主分支
- pipeline-check.sh 检查 SPIKE-REPORT.md 存在且结论明确
