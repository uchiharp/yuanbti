# PRD 编写禁止清单

## 绝对禁止

| 禁止 | 为什么 | 正确做法 |
|------|--------|---------|
| 跳过 Discovery 直接写 PRD | 没有理解问题就出方案，大概率做错 | 先完成 10 个 Discovery 问题 |
| 编造用户没说的需求 | 用户不认，做出来浪费 | 标注 `[假设]` 并让用户确认 |
| 功能需求不编号（REQ-xxx） | 下游无法追踪，数据库无法入库 | 每个 REQ 必须有唯一编号 |
| 验收标准用模糊语言 | 不可测试 = 不可验收 | 量化：数字、时间、状态 |
| 不写 Out of Scope | 范围蔓延，永远做不完 | 明确列出不做的事和原因 |
| REQ 不标注模块归属 | 架构师无法按模块拆分，开发不知道放哪 | 每个 REQ 标注所属模块 |
| REQ 不标注依赖关系 | 任务无法排序，并行组划分失据 | 每个 REQ 标注 Dependencies |
| 不写 Problem Statement 直接写功能 | 不知道为什么做，容易做偏 | 先讲清楚问题再讲方案 |

## 中型+项目额外禁止

| 禁止 | 为什么 | 正确做法 |
|------|--------|---------|
| 跳过 System Overview | REQ 是孤岛，丢失模块间关联 | 先画模块关系+数据流+用户旅程 |
| REQ 不写 UI/UX 段落 | 开发凭想象做界面，和用户期望不一致 | 每个有界面的 REQ 写明交互+状态+组件 |
| 不标注关联用户故事 | 不知道这个功能服务谁 | 每个 REQ 标注关联的 User Story |

## 沟通规范

| 必须问用户 | 不需要问 | AI 绝不应该做 |
|-----------|---------|-------------|
| 项目方向和边界 | 技术细节的实现选择 | 替用户决定优先级 |
| 需求的理解确认 | 模板格式和排版 | 编造用户没说的需求 |
| 验收标准是否合理 | 编号规则（REQ-xxx） | 假设"用户肯定想要X" |
| Out of Scope 确认 | 质量评分的计算 | 跳过 Discovery 直接写 PRD |
| 确认点是否通过 | — | 在确认点未通过时继续 |
