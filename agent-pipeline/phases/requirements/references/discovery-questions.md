# Discovery 问题框架

## 必答题（5个）

### 1. 这个项目要解决什么问题？
- 目的：构建 Problem Statement
- 追问方向：是效率问题？体验问题？合规问题？成本问题？
- 标注 `[假设]`：如果用户回答模糊，AI 填充最佳猜测并标注

### 2. 目标用户是谁？他们现在的痛点是什么？
- 目的：构建 User Impact
- 追问方向：具体什么角色？什么场景下痛苦？严重程度？
- 必须获得：用户画像 + 痛点 + 严重程度证据

### 3. 你期望的解决方案是什么方向？
- 目的：构建 Solution Direction
- 追问方向：给 2-3 个方向选择，标注利弊
- 注意：方向不是实现方案，是"做工具 vs 做平台 vs 做流程优化"

### 4. 怎么算成功？能量化吗？
- 目的：构建 SMART Goals
- 追问方向：有没有数字？没有的话帮用户推算
- 必须获得：至少 1 个可量化的成功指标

### 5. 有什么硬性约束（时间/预算/技术/合规）？
- 目的：构建 Constraints
- 追问方向：有没有不能碰的技术？有没有截止日期？有没有合规要求？
- 注意：约束比需求更优先——约束决定了方案的边界

## 技术题（4个）

### 6. 从零开始还是已有代码库？
- 目的：技术考量
- 影响模板选择：从零用 comprehensive，改造用精简+迁移策略

### 7. 技术栈偏好或强制要求？
- 目的：Tech Stack
- 追问方向：有现成团队技能约束吗？有历史技术债限制吗？

### 8. 需要集成哪些外部系统？
- 目的：Dependencies
- 影响架构：外部系统数量直接影响复杂度判定

### 9. 性能/安全/合规有特殊要求吗？
- 目的：Non-Functional Requirements
- 追问方向：有没有合规标准？有没有 SLA 要求？有没有数据敏感度？

## 开放题（1个）

### 10. 还有什么我该知道的？
- 目的：捕获遗漏信息
- 不要小看这道题——很多关键信息在前9题里问不到

## 交互规则

- 逐个提问，不一次丢出10个问题
- 优先给选择题（2-3个方向 + 利弊标注），不要开放式空投
- 用户回答简略时，AI 用最佳猜测填充并标注 `[假设]`
- 全部答完后整理成 Project Brief，等用户确认
- 确认通过后进入 PRD 生成阶段
