# 日志/异常/错误处理架构规范

> 本 skill 供架构师在阶段 2 设计架构方案时参考，也可供开发在阶段 6 编码时查阅。

---

## 1. 错误码体系

### 分层编码规则

架构方案必须定义统一的错误码格式，推荐格式：`{模块}_{编号}`

| 模块前缀 | 说明 | 示例 |
|---------|------|------|
| `AUTH` | 认证相关 | `AUTH_001`（Token过期）、`AUTH_002`（密码错误） |
| `PERM` | 权限相关 | `PERM_001`（无权限）、`PERM_002`（策略拒绝） |
| `USER` | 用户管理 | `USER_001`（用户不存在）、`USER_002`（手机号已注册） |
| `SYS` | 系统级 | `SYS_001`（服务不可用）、`SYS_002`（参数校验失败） |

### HTTP 状态码映射

| HTTP 码 | 场景 | 业务码范围 |
|---------|------|----------|
| 400 | 参数校验失败 | `SYS_001` ~ `SYS_099` |
| 401 | 未认证 / Token无效 | `AUTH_001` ~ `AUTH_099` |
| 403 | 已认证但无权限 | `PERM_001` ~ `PERM_099` |
| 404 | 资源不存在 | 各模块 `*_001` |
| 409 | 业务冲突（重复注册等） | 各模块 `*_002` ~ `*_099` |
| 500 | 系统内部错误 | `SYS_100`+ |

### 错误响应格式

```json
{
  "code": "AUTH_001",
  "message": "Token已过期，请重新登录",
  "detail": null,
  "timestamp": "2026-05-11T10:30:00Z",
  "traceId": "abc-123-def"
}
```

**规则：**
- `message` 面向用户，可直接展示
- `detail` 面向开发，仅在非生产环境返回
- `traceId` 用于日志关联，每个请求唯一

## 2. 异常传播策略

### 各层职责

| 层 | 异常处理 |
|---|---------|
| Controller | **不捕获**，交给全局异常处理器 |
| Service | 捕获**业务异常**转为业务错误码，**不捕获**系统异常 |
| Repository | **不捕获**，让数据库异常冒泡到 Service |
| 外部调用（HTTP/RPC） | **必须捕获**，转为业务异常或降级 |

### 异常分类

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 业务异常 | 用户不存在、余额不足 | 返回业务错误码 + 用户友好提示 |
| 参数异常 | 缺少必填字段、格式错误 | 返回 400 + 具体字段错误 |
| 系统异常 | 数据库连接失败、NPE | 返回 500 + traceId，日志记录堆栈 |
| 外部异常 | 第三方 API 超时 | 重试 → 降级 → 返回系统错误 |

### 全局异常处理器

```
@RestControllerAdvice
GlobalExceptionHandler
  ├─ catch BusinessException → 业务错误码 + message
  ├─ catch MethodArgumentNotValidException → 400 + 字段错误列表
  ├─ catch AccessDeniedException → 403
  ├─ catch Exception → 500 + traceId（日志记录完整堆栈）
```

**禁止：**
- ❌ 每个方法都 try-catch Exception
- ❌ 吞掉异常返回 null 或空对象
- ❌ 把堆栈信息返回给前端

## 3. 日志规范

### 结构化日志字段

每条日志必须包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| `timestamp` | 时间戳 | `2026-05-11T10:30:00.123Z` |
| `level` | 日志级别 | `INFO` / `WARN` / `ERROR` |
| `traceId` | 请求链路ID | `abc-123-def` |
| `userId` | 操作人 | `user_123` |
| `action` | 操作 | `login` / `create_project` |
| `module` | 模块 | `auth` / `user` / `project` |
| `message` | 描述 | `用户登录成功` |

### 日志级别使用场景

| 级别 | 何时用 | 示例 |
|------|--------|------|
| `DEBUG` | 开发调试信息 | SQL 参数、缓存命中 |
| `INFO` | 业务关键节点 | 用户登录、订单创建、支付完成 |
| `WARN` | 可恢复的异常 | 第一次重试成功、降级触发 |
| `ERROR` | 需要人工介入 | 重试失败、数据不一致、系统异常 |

**规则：**
- ❌ 不用 `ERROR` 记录业务拒绝（余额不足是业务逻辑，不是错误）
- ❌ 不在循环里打 INFO（批量操作只打一条汇总）
- ✅ ERROR 必须带堆栈（`log.error("描述", e)`）
- ✅ 敏感操作必须打 INFO（登录、权限变更、数据删除）

### 审计日志（独立于应用日志）

审计日志记录「谁在什么时候对什么做了什么操作」，用于安全追溯。

| 字段 | 说明 |
|------|------|
| `user_id` | 操作人 |
| `action` | 操作类型（create/update/delete/login） |
| `resource` | 资源类型（user/role/project） |
| `resource_id` | 资源ID |
| `detail` | 变更内容（JSON，记录前后值） |
| `ip` | 请求IP |

**实现方式：** AOP 切面 + `@AuditLog` 注解，不侵入业务代码。

## 4. 敏感数据脱敏

### 必须脱敏的字段

| 字段类型 | 脱敏方式 | 示例 |
|---------|---------|------|
| 密码 | 不记录 | 日志中完全不出现 |
| Token | 截断 | `eyJhb...xxx` |
| 手机号 | 中间遮罩 | `138****5678` |
| 身份证 | 中间遮罩 | `110***********1234` |
| 银行卡 | 只显示后4位 | `****1234` |
| 邮箱 | 遮罩用户名 | `t***@example.com` |

### 脱敏时机

| 场景 | 脱敏方式 |
|------|---------|
| 日志输出 | 日志框架拦截器统一处理 |
| API 响应 | DTO 序列化时 `@JsonSerialize` 注解 |
| 数据库存储 | 密码 bcrypt，其他可逆加密 |
| 错误响应 | 不返回敏感字段 |

## 5. 重试与降级

### 重试策略

| 场景 | 重试次数 | 间隔 | 说明 |
|------|---------|------|------|
| 第三方 API 超时 | 3次 | 指数退避（1s/2s/4s） | 幂等接口才重试 |
| 数据库连接失败 | 3次 | 固定 1s | 连接池自动重连 |
| 消息发送失败 | 5次 | 指数退避 | 写入死信队列 |

**规则：**
- 非幂等操作不重试（如扣款）
- 重试必须有上限，不能无限重试
- 重试日志用 WARN 级别

### 降级策略

| 场景 | 降级方案 |
|------|---------|
| 缓存不可用 | 直接查库（性能降级） |
| 第三方服务不可用 | 返回默认值 / 缓存数据 |
| 搜索服务不可用 | 降级为数据库 LIKE 查询 |
