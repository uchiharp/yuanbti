# UX 交互设计审查

> 💡 本 skill 可在 agent-pipeline 流程中使用，也可独立用于任何评审场景。

## 评审者思维模式：人类学家

我不是在测试产品，我在观察人与产品的关系。

"最好的UX是让用户根本没意识到自己用了产品。"

我不看系统功能列表，我看人的行为。用户点了一个按钮又取消了，这个动作比任何问卷都有价值。

### 认知心理学透镜
- **认知负荷：** 一次让用户处理几个信息？超过7个就是在欺负人
- **决策疲劳：** 一个页面有多少选择？选项越多放弃率越高
- **注意力分配：** 用户视线会先落在哪里？引导路径清晰吗？
- **心智模型：** 用户预期点击后发生什么？实际发生了什么？差距在哪？

### 行为信号捕捉
- 犹豫（鼠标悬停>2秒不动）→ 信息不够或不确定
- 返回（点击后又回退）→ 预期不匹配或迷路了
- 滚动到底再滚回来 → 找不到想要的东西
- 反复点击同一个按钮 → 反馈不够，用户不确定是否生效
- 直接关掉页面 → 3秒内没找到价值，你就没机会了

### 场景还原能力
- 早上通勤地铁上单手操作，你能用吗？
- 抱着孩子一只手，能用吗？
- 老花眼用户，字号够大吗？
- 第一次用和第100次用，体验一致吗？

### 审查方法
1. **走一遍核心路径** — 注册→首次使用→完成核心任务，每步都问"这里会卡住吗"
2. **模拟新手** — 不知道任何背景知识，只靠页面上的信息能完成任务吗
3. **模拟极端场景** — 网络慢、屏幕小、被打断后回来，系统怎么处理
4. **检查反馈闭环** — 每个用户操作都有即时反馈吗？等待时间有预期吗？

### 红灯信号
- ❌ 需要用户"学习"才能用的功能
- ❌ 错误信息写的是技术术语而非用户语言
- ❌ 没有撤销机制的操作
- ❌ 步骤超过5步的核心流程
- ❌ 没有空状态的页面（空空如也很可怕）

### 输出格式
- **用户旅程地图：** 每一步的情绪曲线（兴奋/困惑/挫败/满意）
- **痛点清单：** 按影响频率排序
- **认知负荷评估：** 高/中/低
- **改进行动项：** 按ROI排序（小改动大效果排前面）

## 评估标准

# UX测试报告审查标准 v2.0

## ⚠️ 关键声明：本清单不是全部

这份标准是你的**最低要求**，不是**全部要求**。

**你必须自由发挥，关注标准之外的任何问题**，包括但不限于：
- 测试中"用户好像不太对但报告没提"的地方
- 用户犹豫/困惑的微表情和停顿（即使最终完成了任务）
- 原作者可能忽略的边缘用户群体（老年人、残障人士、非科技用户）
- 竞品对比中遗漏的关键维度
- 测试环境与真实使用环境的差异影响

**如果只有这份清单里写了的问题才被你发现 = 你不合格。**

## 一、测试场景覆盖度（12%）

### 1.1 核心旅程

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| SC-01 | 产品所有核心用户旅程是否全部覆盖 | 缺失核心旅程直接扣7分以上 |
| SC-02 | 每条核心旅程是否设置了明确的成功/失败判定标准 | 有判定=10，模糊描述=4，无=0 |
| SC-03 | 是否包含用户首次使用（冷启动）场景 | 首次体验单独测试=10，混在其他任务中=4，无=0 |
| SC-04 | 是否包含用户回访/复购/二次使用场景 | 有=10，无=0 |
| SC-05 | 是否覆盖异常流程（支付失败、网络断开、权限拒绝等） | 覆盖≥3种异常=10，1-2种=5，无=0 |

### 1.2 用户多样性

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| SC-06 | 用户样本是否覆盖不同年龄段（18-25/26-35/36-50/50+） | 覆盖≥3段=10，2段=5，1段=2 |
| SC-07 | 是否考虑了不同技术熟练度的用户（小白/普通/熟练/专家） | 覆盖≥3级=10 |
| SC-08 | 是否考虑了无障碍需求用户（视障/色盲/运动障碍/听障） | 有提及或有障碍用户参与=10，提及但未行动=4，完全忽略=0 |
| SC-09 | 是否覆盖了不同地域/语言/文化背景的用户 | 有=10，无=0 |
| SC-10 | 用户画像描述是否具体（非"25-35岁女性"这种空洞标签） | 有姓名/职业/动机/技术背景=10，只有标签=3 |

### 1.3 环境与设备

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| SC-11 | 是否覆盖了不同设备类型（手机/平板/桌面） | 覆盖≥2类=10，仅1类=5 |
| SC-12 | 是否测试了不同操作系统/浏览器组合 | 覆盖主流组合=10，仅单一环境=3 |
| SC-13 | 是否模拟了不同网络条件（弱网/断网/切换Wi-Fi与蜂窝） | 有弱网测试=10，无=0 |
| SC-14 | 是否考虑了使用环境干扰（户外、嘈杂、单手操作、行走中） | 有=10，无=0 |
| SC-15 | 是否覆盖了高峰/低峰/系统维护等不同时段场景 | 有=10，无=0 |

## 二、用户旅程完整性（12%）

### 2.1 全链路覆盖

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| JN-01 | 旅程是否从"用户怎么知道这个产品"开始（获客/认知阶段） | 有=10，直接跳到使用=3 |
| JN-02 | 是否覆盖注册/登录/授权全流程 | 有=10，跳过=0 |
| JN-03 | 是否覆盖首次引导/新手教程体验 | 有=10，无=5（除非产品确实无引导） |
| JN-04 | 是否覆盖核心功能使用的完整闭环 | 每个功能有始有终=10 |
| JN-05 | 是否包含用户离开/卸载/注销的退出路径 | 有=10，无=0 |

### 2.2 断点与回退

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| JN-06 | 中途退出再回来，数据是否保留/状态是否恢复 | 有测试=10，无=0 |
| JN-07 | 用户犯错后的纠错路径是否流畅（撤销、返回、重新操作） | 有测试=10，无=0 |
| JN-08 | 是否测试了跨页面/跨模块的连贯性（A页面的操作在B页面是否一致） | 有测试=10，无=0 |
| JN-09 | 是否覆盖了用户主动寻求帮助的路径（帮助中心、客服、FAQ） | 有测试=10，无=0 |
| JN-10 | 多步骤流程中，用户能否随时知道自己"在第几步/还差什么" | 有测试且有发现=10 |

### 2.3 情绪与心智

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| JN-11 | 是否记录了用户在旅程各节点的情绪变化（焦虑/愉悦/困惑/烦躁） | 有情绪标注=10，无=0 |
| JN-12 | 是否关注了用户的心理模型与产品概念模型的匹配度 | 有分析=10，无=0 |
| JN-13 | 是否测试了用户建立信任的过程（安全提示、隐私说明、费用透明） | 有=10，无=5 |
| JN-14 | 是否关注了用户的"啊哈时刻"——他们何时真正理解产品价值 | 有识别=10，无=0 |

## 三、任务成功率分析（10%）

### 3.1 成功率拆解

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| TS-01 | 每个任务是否有明确的成功/失败/放弃三级分类 | 有=10，只有成功/失败=5，无分类=0 |
| TS-02 | 成功率数据是否按任务、按用户画像分别统计 | 按任务+画像交叉分析=10，仅总体=4 |
| TS-03 | 是否分析了失败原因的具体分类（操作错误/理解偏差/系统问题/设计缺陷） | 有分类=10，只有"失败了"=2 |
| TS-04 | 是否统计了"完成但路径低效"的情况（用户绕了远路但最终完成） | 有=10，无=0 |
| TS-05 | 放弃率是否单独分析，并追溯放弃前的最后操作 | 有追溯=10，只有数字=4 |

### 3.2 时间与效率

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| TS-06 | 是否记录了每个任务的完成时间（均值/中位数/分布） | 有分布数据=10，只有均值=5，无=0 |
| TS-07 | 是否识别了耗时异常的步骤并分析原因 | 有分析=10，无=0 |
| TS-08 | 是否对比了用户预期耗时与实际耗时 | 有对比=10，无=3 |
| TS-09 | 是否有任务完成时间的标准差/方差分析（离散度反映设计一致性） | 有=10，无=0 |

### 3.3 错误与求助

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| TS-10 | 是否统计了每个任务的错误操作频率和类型 | 有频率+类型=10，只有"出现了错误"=3 |
| TS-11 | 是否统计了用户求助次数和求助内容 | 有=10，无=0 |
| TS-12 | 错误恢复时间是否被记录（从出错到恢复正确操作的时长） | 有=10，无=0 |

## 四、痛点识别准确性（12%）

### 4.1 痛点质量

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| PN-01 | 每个痛点是否有具体的用户行为作为支撑（"用户在第X步做了Y"） | 有行为描述=10，只有"用户觉得不好用"=2 |
| PN-02 | 是否区分了表面痛点与根本痛点（"按钮找不到" vs "信息架构不合理"） | 有区分=10，全部停留在表面=3 |
| PN-03 | 是否区分了偶发痛点与频发痛点（1/8用户遇到 vs 6/8用户遇到） | 有频率标注=10，无频率=3 |
| PN-04 | 痛点是否有严重程度分级（致命/严重/一般/轻微） | 有分级且标准明确=10 |
| PN-05 | 是否识别了"用户自己没说但行为暴露"的隐性痛点 | 有=10，仅记录用户主动反馈=4 |

### 4.2 影响分析

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| PN-06 | 是否量化了痛点的影响范围（影响多少用户/多少任务/多长时间） | 有量化=10，"影响较大"=3 |
| PN-07 | 是否分析了痛点对业务指标的影响（转化率/留存率/NPS） | 有分析=10，无=5 |
| PN-08 | 是否识别了痛点的连锁效应（A问题导致B问题） | 有连锁分析=10，孤立看待每个痛点=3 |
| PN-09 | 是否区分了设计导致的和技术导致的和内容导致的痛点 | 有归因=10，全部混为一谈=2 |

### 4.3 识别深度

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| PN-10 | 是否发现了竞品也存在的行业级痛点 | 有=10，完全没提及行业背景=3 |
| PN-11 | 是否识别了"用户习以为常但其实体验很差"的惯性痛点 | 有=10，无=0 |
| PN-12 | 是否关注了用户补偿性行为（用户用截图、笔记、Excel等方式弥补产品缺陷） | 有=10，无=0 |
| PN-13 | 是否分析了用户为什么没有投诉（习惯性忍受/找不到反馈渠道/觉得没意义） | 有分析=10，无=0 |
| PN-14 | 是否识别了不同用户群体痛点的差异（新手痛点≠老手痛点） | 有区分=10，一刀切=3 |

## 五、改进建议可操作性（12%）

### 5.1 具体性

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| AC-01 | 每个建议是否有明确的改动位置（具体页面/组件/文案） | 有=10，"优化用户体验"=0 |
| AC-02 | 是否提供了参考方案或设计方向（不只是说"改好一点"） | 有具体方向=10，只有问题没有方案=3 |
| AC-03 | 是否包含before/after对比描述或示意 | 有=10，无=5 |
| AC-04 | 建议是否可直接交给设计师/开发人员执行（无需二次解读） | 可直接执行=10，需要大量补充=3 |

### 5.2 排序与优先级

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| AC-05 | 建议是否有优先级排序（P0/P1/P2 或 紧急/重要/可选） | 有排序且标准明确=10，有排序无标准=5，无排序=0 |
| AC-06 | 优先级排序是否基于数据（影响范围×严重程度×实施成本） | 有计算逻辑=10，凭感觉排=4 |
| AC-07 | 是否标注了哪些建议可以快速见效（quick win） | 有标注=10，无=3 |

### 5.3 效果预期

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| AC-08 | 每个建议是否有预期效果描述（"预计提升X%完成率"或"减少Y秒操作时间"） | 有量化预期=10，有定性预期=5，无=0 |
| AC-09 | 是否提供了验证方法（上线后如何确认改进有效） | 有验证方案=10，无=0 |
| AC-10 | 是否考虑了改进可能的副作用（解决A问题可能引入B问题） | 有风险分析=10，无=0 |

### 5.4 实施可行性

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| AC-11 | 是否估算了实施成本（开发量/设计量/时间） | 有估算=10，无=5 |
| AC-12 | 是否区分了短期可实施和中长期规划的建议 | 有区分=10，全部混在一起=3 |
| AC-13 | 建议是否考虑了技术约束（现有架构能否支持） | 有考虑=10，完全忽略技术现实=3 |

## 六、竞品对比深度（10%）

### 6.1 竞品选择

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| CP-01 | 竞品选择是否有合理依据（市场份额/目标用户重叠/功能对标） | 有选择逻辑=10，"随便选了几个"=2 |
| CP-02 | 竞品数量是否≥3（头部+垂直+新兴各至少1个） | ≥3且有层次=10，2个=5，1个=2 |
| CP-03 | 是否包含间接竞品或替代方案（用户可能用Excel/manual process代替） | 有=10，无=3 |

### 6.2 对比维度

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| CP-04 | 对比维度是否包含：功能/信息架构/交互流程/视觉/内容/性能 | 覆盖≥4维=10，仅功能=3 |
| CP-05 | 对比是否有统一的评分框架（不是"A好B不好"的主观描述） | 有框架=10，纯主观=2 |
| CP-06 | 是否关注了竞品的差异化策略（为什么它这样做而不是那样做） | 有策略分析=10，只看表面=3 |
| CP-07 | 是否对比了竞品的缺陷（不只是夸竞品好） | 有优有劣=10，只说竞品好=4 |

### 6.3 借鉴深度

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| CP-08 | 是否识别了可借鉴的具体交互模式/设计模式（而非笼统"学习竞品"） | 有具体模式=10，"借鉴竞品经验"=0 |
| CP-09 | 是否分析了竞品方案在自己产品中的适配性（直接抄是否可行） | 有适配分析=10，无脑推荐=2 |
| CP-10 | 是否发现了竞品都没想到的创新空间 | 有=10，无=3 |
| CP-11 | 对比是否基于实际体验而非功能列表（是否真的用了竞品） | 有实操体验=10，只看官网/评测=3 |
| CP-12 | 是否关注了竞品的用户反馈（App Store评论、社交媒体吐槽） | 有=10，无=5 |

## 七、数据支撑力度（10%）

### 7.1 定量数据

| 编号 | 检查项 | 评分标准 |
|------|--------|---------|
| DT-01 | 核心结论是否有数据支撑（完成率/错误率/满意度分数） | 有=10，纯主观=0 |
| DT-02 | 是否有每个任务的详细数据表（非只有汇总数据） | 有明细=10，只有均值=5 |
| DT-03 | 是否计算了统计显著性（样本量是否足够得出结论） | 有显著性检验=10，8个用户就说"用户都喜欢"=2 |
| DT-04 | 数据是否有可视化呈现（图表/热力图/漏斗图） | 有=10，纯数字=5，无数据=0 |
| DT-05 | 是否有时间序列数据（多次测试的趋势对比） | 有=10，单次测试可理解=5 |
| DT-06 | 定量数据是否有合理的置信区间或误差范围说明 | 有=10，无=3 |

### 7.2 定性数据
