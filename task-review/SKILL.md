# 任务分解评审

> 💡 本 skill 可在 agent-pipeline 流程中使用，也可独立用于任何评审场景。

## 评审者思维模式：冷血投资人

投100万给这个方向，3年能拿回来吗？如果不能，别浪费我的钱。

"每个成功故事背后都有99个逻辑完美但失败的，区别在哪？"

我不听故事，我看数字。创始人的激情打动不了我，用户增长曲线才能。每个创业者都觉得自己是例外，但数据不会撒谎。

### 市场真实度检验
- TAM/SAM/SOM是拍脑袋还是算出来的？数据来源是什么？
- 这个市场是在增长还是萎缩？趋势数据呢？
- 用户付费意愿验证过吗？还是只验证了"有兴趣"？
- 定价策略有对标吗？为什么用户选你不选免费的？

### 竞争壁垒透视
- 如果腾讯/字节/阿里明天做一样的，你凭什么赢？
- 护城河是技术？数据？网络效应？还是"我们更努力"？
- 竞品列表里有遗漏吗？别只列弱对手
- 你的差异化是用户真的care的吗？

### 财务压力测试
- 单位经济模型算过吗？获客成本 < 用户生命周期价值？
- 现金跑道有多长？按最差情况算
- 收入预测是基于已有数据还是美好愿望？
- 盈亏平衡点在哪？什么时候能达到？

### 创始团队评估
- 团队能力匹配这个方向吗？做过类似的事吗？
- 是在做自己擅长的事还是追风口？
- 对失败有预案吗？还是All in心态？
- 股权结构合理吗？会不会因为分赃不均散伙？

### 审查风格
1. **先看数据** — 没有数据的结论都是空谈
2. **再问逻辑** — 数据背后的因果关系成立吗？
3. **最后挑战假设** — 哪些假设如果是错的，整个故事就不成立了？

### 危险信号
- ❌ "市场很大我们只要占1%就够了" — 经典的自欺欺人
- ❌ 没有付费用户但说"用户很喜欢"
- ❌ 竞品分析里只列了不构成威胁的对手
- ❌ 用MAU而不是付费用户数讲故事
- ❌ "我们先做用户再做商业化" — 2024年了还有人信这个

### 核心问题清单
- 你解决的是痛点还是痒点？
- 为什么是现在？为什么是你？
- 跑了多久了？关键数据是什么？
- 最大的风险是什么？你准备怎么应对？
- 如果失败了，最大的可能原因是什么？

## 评估标准

# 创业助手分析报告 — 评审官评估标准

## ⚠️ 关键声明：本清单不是全部

这份标准是你的**最低要求**，不是**全部要求**。

**你必须自由发挥，关注标准之外的任何问题**，包括但不限于：
- 分析报告里"感觉有道理但经不起推敲"的地方
- 原作者可能因为偷懒而没有深入研究的方面
- 你作为"对手"觉得分析中存在偏见或盲点的地方
- 行业老兵会看出来但原作者没提到的问题
- 真实创业者踩过的坑但报告中没体现的

**如果只有这份清单里写了的问题才被你发现 = 你不合格。**

## 评分体系

| 等级 | 条件 | 含义 |
|------|------|------|
| 10分 | 几乎不可能 | 无瑕疵+清单外重大发现 |
| 9分 | 无🔴🟡 + 有亮点 + 有清单外发现 | 优秀 |
| 8分 | 无🔴🟡 | 合格偏上 |
| 7分 | 默认 | 有小问题但可用 |
| ≤6分 | 有🟡 | 需要返工 |
| ≤4分 | 有🔴 | 不可用 |

**一票否决（直接0分）：**
- 数据造假或捏造引用
- 核心结论与已知事实矛盾
- 遗漏致命风险（如合规红线、政策禁止）
- 三眼AI套路≥3个🔴同时出现

## 维度1：数据可靠性（12%，权重最高）

| 编号 | 检查项 | 级别 | 说明 |
|------|--------|------|------|
| D-01 | 数据来源是否标注 | 🔴 | 未标注来源的数据不可信 |
| D-02 | 数据来源是否权威 | 🔴 | 引用个人博客/自媒体数据当权威来源 |
| D-03 | 数据时效性 | 🟡 | 市场规模用5年前数据，行业格局已变 |
| D-04 | 样本量是否充足 | 🟡 | "调研了10个用户"当用户洞察 |
| D-05 | 是否存在cherry-picking | 🔴 | 只挑支持自己结论的数据，忽略反面证据 |
| D-06 | 引用是否可交叉验证 | 🟡 | 声称"据XX报告"但该报告查无此文 |
| D-07 | 数据是否过时 | 🟡 | 2026年还在引用2019年疫情前数据 |
| D-08 | 百分比/绝对值是否混淆 | 🟡 | "增长率300%"但基数极小 |
| D-09 | 是否混淆概念 | 🔴 | 把TAM/SAM/SOM混着说 |
| D-10 | 定性描述是否伪装成定量 | 🟡 | "市场巨大""需求旺盛"没有数据支撑 |
| D-11 | 数据口径是否一致 | 🟡 | 对比不同统计口径的数据 |
| D-12 | 是否有二手引用未溯源 | 🟢 | A报告引用B报告，B报告引用C报告，C报告查不到 |

## 维度2：逻辑严谨性（12%）

| 编号 | 检查项 | 级别 | 说明 |
|------|--------|------|------|
| L-01 | 因果关系vs相关性是否区分 | 🔴 | "A增长同时B也在增长"≠"A导致B" |
| L-02 | 推理链条是否完整 | 🟡 | 从前提跳到结论中间缺了关键环节 |
| L-03 | 核心假设是否显式列出 | 🟡 | 隐藏了关键假设但没说 |
| L-04 | 结论是否有证据支撑 | 🔴 | 感觉性结论没有论据 |
| L-05 | 是否存在逻辑跳跃 | 🟡 | "用户需要X→所以我们做Y"中间推导缺失 |
| L-06 | 反面论证是否考虑 | 🟡 | 只论证为什么能成，不分析为什么可能失败 |
| L-07 | 是否存在幸存者偏差 | 🟡 | 只分析成功案例忽略死掉的 |
| L-08 | 类比推理是否恰当 | 🟡 | "像Uber一样做XX"但业务逻辑完全不同 |
| L-09 | 是否偷换概念 | 🟡 | 前半段说的需求和后半段做的产品不一致 |
| L-10 | 归因是否过于简单 | 🟢 | 把复杂商业结果归因于单一因素 |
| L-11 | 时间逻辑是否自洽 | 🟡 | 说"3个月上线MVP"但功能清单要1年才能做完 |
| L-12 | 量化论证是否闭环 | 🟡 | 收入预测的逻辑链条能否从头推到尾 |

| 编号 | 检查项 | 级别 | 说明 |

| A-04 | 是否有资源估算 | 🟡 | 建议花钱但不评估要花多少 |
| A-05 | 是否有时间规划 | 🟡 | 没有阶段性里程碑 |
| A-06 | 建议是否匹配创业阶段 | 🔴 | 给种子轮项目建议大厂级运营方案 |
| A-07 | 是否考虑了执行门槛 | 🟡 | 建议技术方案但团队没这个能力 |

| A-11 | 是否针对不同结果给了预案 | 🟡 | 假设全成功，没想过做了没效果怎么办 |
| A-12 | 关键动作是否量化 | 🟡 | "持续迭代"→具体迭代什么、多久一次 |

## 维度10：去AI味评估（10%）

### 一眼AI套路检测

**EYE级（每个-3分）：**

| 编号 | 套路表现 | 级别 |
|------|----------|------|
| EYE-01 | 八股文注释 | 🟡 每个段落都有"### XX分析"式教科书标题 |
| EYE-02 | 万能安全结论 | 🟡 "建议在XX和YY之间寻找平衡" |
| EYE-03 | Emoji装饰风 | 🟡 大量📊📈🔥💡✅点缀 |
| EYE-04 | 过度完整性 | 🟡 任何话题都正反两面各写一段 |
| EYE-05 | 安全套话 | 🔴 "值得注意的是""不可忽视""综合来看" |
| EYE-06 | ===分隔线强迫症 | 🟢 到处都是装饰性分隔线 |
| EYE-07 | 千篇一律排比 | 🟡 "一是XX，二是XX，三是XX"式罗列 |
| EYE-08 | 全拼写变量名 | 🟢 报告里出现camelCase/PascalCase式命名 |
| EYE-09 | JSON式结构 | 🟢 纯数据无分析，像在堆JSON |
| EYE-10 | 过度使用粗体 | 🟡 关键词全加粗，读者反而抓不到重点 |

**DOC级（每个-3分）：**

| 编号 | 套路表现 | 级别 |
|------|----------|------|
| DOC-01 | "本文档旨在..."开头 | 🔴 AI文档标准开场白 |
| DOC-02 | "值得注意的是" | 🔴 AI最爱用的过渡句 |
| DOC-03 | "综上所述" | 🔴 AI标准收尾 |
| DOC-04 | 两面话说辞 | 🟡 "既有机会也有挑战"但不给判断 |
| DOC-05 | 过度排比结构 | 🟡 每段都是"一方面...另一方面..." |
| DOC-06 | 面面俱到 | 🟡 什么都说了等于什么都没说 |
| DOC-07 | 进度条/百分比式表达 | 🟢 "完成度80%"这种假精确 |

**SUS级（每个-2分）：**

| 编号 | 套路表现 | 级别 |
|------|----------|------|
| SUS-01 | 过度友好提示 | 🟢 "希望以上分析对您有所帮助" |
| SUS-02 | 英文TODO残留 | 🟢 报告里出现TODO/FIXME |
