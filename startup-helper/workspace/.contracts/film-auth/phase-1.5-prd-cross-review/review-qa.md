# QA PRD 评审

## 评审信息
- 评审者：QA
- 评审时间：2026-05-11 19:51
- 总体评分：6/10

## 检查点

### 检查点1：可测性 — P0功能验收标准

**评分：5/10**

PRD 功能描述详细，但**缺少明确的验收标准（Acceptance Criteria）**。每个P0功能只有功能描述，没有"给定-当-则"格式的验收条件，导致测试用例设计缺乏依据。

具体问题：
- **SSO登录流程**：7步流程描述清晰，但缺少异常场景的预期行为（如code过期、code重复使用、redirect_uri不匹配时应返回什么）
- **Token刷新**：refresh_token过期后应返回什么？是跳转登录页还是返回特定错误码？未明确
- **批量创建用户**：用户名"只允许中文、英文、数字"——但未定义非法输入时的错误提示和响应格式
- **权限计算公式** `角色权限 ∩ 项目权限 ∩ 数据范围` 已给出，但缺少具体测试场景的预期结果
- **审计日志导出Excel**：导出格式、字段、最大行数未定义

**可自动化评估：** 大部分接口测试可自动化（API测试），但SSO流程涉及页面跳转，需要E2E测试框架（Playwright/Cypress）支持。

### 检查点2：测试数据准备

**评分：4/10**

PRD 未提供任何测试数据方案，以下问题需要在开发前明确：

- **初始数据**：super_admin 预设角色的初始账号密码是什么？首次部署如何初始化？
- **测试用户集**：需要覆盖4种角色（管理员/项目负责人/普通用户/外部协作方）× 3种项目角色（owner/editor/viewer）× 3种数据范围（own/project/all），组合矩阵至少需要10+测试账号
- **权限矩阵测试数据**：17个权限点的排列组合，需要预定义测试角色的权限配置
- **批量创建测试数据**：需要准备各种边界输入（超长用户名、特殊字符、重复用户名、空行、纯空格行）
- **Token相关测试**：需要能精确控制Token过期时间的测试手段（时钟mock或短过期配置）
- **并发登录测试**：需要模拟多设备登录的测试工具/脚本

### 检查点3：边界场景覆盖

**评分：6/10**

已识别但缺少细化的边界场景：

| 边界场景 | PRD覆盖情况 | 测试关注点 |
|----------|------------|-----------|
| 密码策略边界 | ✅ 已定义 | 8位整、7位、含特殊字符、最近3次密码循环、90天过期临界点 |
| 登录失败锁定 | ✅ 已定义 | 5次失败锁定、第6次尝试、30分钟自动解锁、锁定期间管理员重置密码 |
| Token过期 | ⚠️ 部分覆盖 | access_token过期刷新成功/失败、refresh_token过期、两端同时过期 |
| 并发登录 | ⚠️ 策略可配 | 三种策略切换、单设备模式下旧会话被踢时的用户体验 |
| 权限边界 | ⚠️ 缺少细节 | 用户无角色时能否登录？角色被禁用后已登录用户的行为？多角色权限冲突？ |
| 批量操作 | ⚠️ 缺少细节 | 批量创建1000个用户时的性能？部分成功部分失败的处理？ |
| 软删除 | ❌ 未定义 | 删除后用户名能否复用？删除用户的审计日志是否保留？删除后Token是否立即失效？ |

**未覆盖的关键边界：**
- 同一用户同时被分配和移除角色（并发操作）
- 角色权限修改时，正在进行的API请求如何处理
- 批量踢出大量用户（如修改super_admin权限）的性能影响
- 用户名长度上限、密码长度上限
- 并发修改同一角色权限

### 检查点4：安全测试

**评分：5/10**

PRD提到了安全要求（4.2节），但缺少具体的安全测试方案：

| 安全维度 | PRD提及 | 测试方案缺口 |
|----------|---------|-------------|
| 密码存储 | "加密存储，不可逆" | 未指定算法（bcrypt/argon2？），需验证哈希强度 |
| SQL注入 | "参数化查询" | 需测试所有用户输入字段（用户名、搜索关键词、JSON导入） |
| XSS | "输入输出过滤" | 需测试昵称、角色描述、权限名称等字段 |
| CSRF | 提及 | 需验证SSO回调流程的CSRF防护 |
| 暴力破解 | "接口限流" | 未定义限流阈值（每分钟几次？IP级还是账号级？） |
| 权限越权 | 未明确 | 需测试水平越权（A用户修改B用户信息）和垂直越权（普通用户调用管理API） |
| Token泄露 | JWT签名 | 需验证Token是否能被伪造、篡改、重放 |
| redirect_uri验证 | OAuth流程 | 需测试开放重放攻击（redirect_uri指向恶意站点） |

**关键安全测试遗漏：**
- ❌ 未提及密码在传输过程中的处理（是否前端加密？还是明文传HTTPS？）
- ❌ client_secret 的存储和传输安全
- ❌ 审计日志本身的防篡改机制

### 检查点5：假设清单风险

**评分：7/10**

假设清单识别较好，但对测试影响最大的风险：

| 假设 | 不成立时对测试的影响 | 风险等级 |
|------|---------------------|---------|
| Redis可用（Token黑名单） | 降级为DB轮询，性能下降，需准备两套测试环境 | 🔴 高 |
| QPS<500 | 超出后现有测试工具可能无法模拟，需准备压测方案 | 🟡 中 |
| PostgreSQL | 如果开发阶段换数据库，所有SQL相关测试用例需重写 | 🟡 中 |
| HTTPS部署 | 如果HTTP部署，JWT安全测试场景完全不同 | 🟡 中 |
| 业务系统可改造接入 | 无法测试完整SSO流程，只能mock测试 | 🟢 低 |

## 问题清单

- 🔴 P0: **缺少验收标准** — 所有P0功能没有明确的Given-When-Then验收条件，无法判断功能是否完成，测试用例设计无依据
- 🔴 P0: **Token刷新+并发登录的边界行为未定义** — refresh_token过期时的用户体验、单设备模式下旧会话被踢的通知机制、Token黑名单与并发请求的竞态条件，这些直接影响核心流程的测试通过率
- 🔴 P0: **批量踢出性能指标缺失** — 修改角色权限触发批量踢出，5000用户场景下的响应时间要求未定义，无法设定性能测试基线
- 🟡 P1: **安全测试方案不完整** — 缺少限流阈值、越权测试方案、redirect_uri开放重放防护、密码传输方案
- 🟡 P1: **初始数据方案缺失** — super_admin初始化流程、测试数据准备方案未说明，影响冒烟测试启动
- 🟡 P1: **软删除行为未定义** — 删除用户后用户名复用、审计日志保留、Token失效时机不明确
- 🟢 P2: **错误码规范缺失** — 各接口的错误响应格式、错误码体系未定义，增加自动化测试断言难度

## 建议

- 🟢 建议PRD补充每个P0功能的验收标准（Acceptance Criteria），格式采用 Given-When-Then，至少覆盖正常流程+1个异常场景。这是测试团队启动用例设计的前置条件，不补齐会导致开发完成后才能设计测试用例，延长交付周期。
