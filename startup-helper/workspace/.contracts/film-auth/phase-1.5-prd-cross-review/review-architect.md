# 架构师 PRD 评审

## 评审信息
- 评审者：架构师
- 评审时间：2026-05-11 19:51
- 总体评分：7/10

## 检查点

- **技术可行性**: 整体可行。OAuth2 授权码模式 + JWT + Redis 黑名单是成熟方案。权限三级命名（`系统:模块:功能点`）设计合理，易于扩展。但 Token 黑名单在高并发场景下存在性能风险（见P0）。权限校验 <50ms 的目标在本地验签场景可达，但如果每次都调认证中心 API 则很难保证。

- **架构假设合理性**: 假设清单覆盖充分。`QPS<500` 和 `用户500-5000+` 的假设对初期合理，但权限校验走远程 API 的假设与 <50ms 目标矛盾。假设"所有业务系统为自研"降低了 V1.0 复杂度，合理。Redis 依赖作为核心组件，降级方案（数据库轮询）在 500 QPS 下不可接受，需明确 Redis 高可用方案。

- **非功能需求可达性**: 登录 <500ms 合理。Token 验证 <10ms（本地验签）合理。但 **权限校验 <50ms** 存在歧义——如果是远程调用认证中心，网络延迟就可能超标；如果是本地缓存+验签，则需明确缓存更新机制和一致性要求。500 并发登录目标合理，但未考虑 Token 黑名单查询对 Redis 的压力。

- **可扩展性**: 权限体系设计良好。三级权限命名支持按需细分，`系统` 前缀天然支持新业务系统接入。项目级权限 + 数据范围（own/project/all）覆盖了常见场景。V1.1（第三方应用接入）和 V2.0（LDAP、多租户）的扩展路径清晰。但多租户支持在当前数据模型下改动较大，建议 V1.0 数据模型预留 tenant_id 字段。

- **安全性考量**: 密码策略合理（8位、大小写+数字、90天过期、历史密码检查）。JWT 签名 + HTTPS 是标配。但存在几个安全隐患：（1）Token 黑名单只靠 Redis，Redis 故障时踢人失效；（2）client_secret 的存储和传输安全未明确；（3）批量踢出的原子性（修改角色权限时踢出所有相关用户）可能存在竞态条件。

## 问题清单

- 🔴 P0: **Token 黑名单方案在 Redis 故障时失效** — 修改角色权限后需批量踢出所有相关用户，如果 Redis 不可用，踢人失败导致旧权限持续生效。建议：Redis 用主从/哨兵保证可用性；黑名单同时写数据库作为兜底；JWT payload 中加入 `role_version` 字段，角色权限变更时递增版本号，本地验签时可低成本检测权限是否过期。

- 🟡 P1: **权限校验接口性能目标模糊** — PRD 写了"权限校验 <50ms"但未说明是本地校验还是远程调用。如果是远程调用，建议各业务系统实现本地权限缓存（带 TTL），认证中心权限变更时通过黑名单机制通知刷新。建议在架构设计中明确校验路径。

- 🟡 P1: **批量踢出的性能和一致性** — 修改角色权限时需踢出该角色所有用户，如果角色有几百个用户，逐个写 Redis 黑名单是否有性能问题？建议用 `role:{role_id}:version` 递增方案替代逐个踢出，JWT 中嵌入角色版本号，验签时比对即可。

- 🟡 P1: **数据模型未预留多租户字段** — V2.0 规划了多租户，但 V1.0 数据模型如果不含 tenant_id，后续改造成本高。建议 V1.0 核心表（users、roles、permissions）预留 `tenant_id` 字段，默认值为 `1`。

- 🟢 P2: **权限模板硬编码** — 权限模板（项目管理员、内容编辑等）写死在 PRD 中，建议做成可配置的，管理员可自定义模板。

- 🟢 P2: **审计日志未考虑分表/归档策略** — 随着用户量增长，审计日志表会膨胀。建议在架构阶段规划分区或定期归档策略。

## 建议

- 🟢 **用角色版本号替代逐个踢出**：JWT payload 中增加 `roles: [{id, version}]` 字段，认证中心维护每个角色的当前版本号（存 Redis）。权限校验时只需比对版本号，无需逐个用户写黑名单，大幅简化踢出逻辑并提升性能。这是对 Token 黑名单方案的架构级优化建议。
