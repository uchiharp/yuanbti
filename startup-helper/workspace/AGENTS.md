## 🚨 核心规则：大任务分解

> **所有大型任务（开发、测试、审查、修复）都必须用大任务分解方法执行。**

参见 `task-decomposer` skill：
- 大任务按维度拆成小任务
- 用表格模板约束Agent输出
- 分轮执行，逐轮验证
- 产出可量化（行数、行号、字段完整性）

**什么时候用大任务分解：**
- 任务需要读3个以上文件
- 任务描述含"全面"、"所有"、"整体"
- 产出包含多种类型

**什么时候直接做：**
- 只需读1个文件、产出单一
- 简单的编译检查、单个curl测试

## 🔧 验证管道（用大任务分解）

### 后端验证

收到代码后，用大任务分解启动验证Agent：

```
第一轮（并行）
  ├─ Agent A → mvn compile → 报告编译结果
  ├─ Agent B → 启动服务 + 健康检查 → 报告启动状态
  └─ Agent C → curl测试核心接口 → 报告每个接口的HTTP状态和JSON结构
```

### 集成验证（E2E）

```
第一轮（并行）
  ├─ Agent A → 编译+启动后端 → 健康检查
  ├─ Agent B → 编译+启动前端 → 页面检查
  └─ Agent C → API冒烟测试（注册/登录/CRUD/搜索）→ 报告结果

验证标准：
- 单元测试通过 ≠ 功能正常
- 必须连真实服务（PostgreSQL + Redis）测试
- 空数据场景必须覆盖（新用户、无物品）
- 边界场景必须覆盖（超长输入、无效参数）
```

### 验证汇报格式

```
## 验证结果

### 单元测试
- mvn test：159/159 通过 ✅
- 注意：这是Mock环境，不代表真实数据库行为

### E2E验证
- 后端启动：健康检查 UP ✅
- 前端启动：页面正常响应 ✅
- API冒烟：X个通过 / Y个失败
- 空数据场景：X个通过 / Y个失败

### 待人工验证
- 请在浏览器中打开 http://localhost:5173 操作确认
```

## 📚 经验教训

- **2026-04-09**：所有 Agent 都在"假装测试"，自测报告全是假的
  - 根因：LLM 生成"测试通过"的文字比真正执行命令更容易
- **2026-04-10**：单元测试全通过但E2E有bug（findTop20空结果异常）
  - 根因：单元测试用Mock，不测真实数据库行为
  - 教训：单元测试通过 ≠ 功能正常，必须E2E验证
  - 教训：汇报时要说清楚测试范围，不要误导用户以为"全部验证通过"
  - 教训：空数据场景是新用户第一体验，必须重点测试
- **2026-04-10**：大任务分解方法论验证成功
  - 先产出事实（清单/表格），再做判断（分析）
  - 表格模板约束输出，防止Agent偷懒
  - 已固化为 task-decomposer skill
- **2026-04-11**：前后端接口不一致导致4个API返回400
  - 根因：前后端Agent分别开发，URL没对齐
  - 教训：同一功能的前后端必须由同一个Agent做，不拆分
  - 教训：spawn Agent 时必须发详细指令（行号、路径、验证方式），不能省略
  - 教训：给用户看的计划和给Agent的指令必须完全一致
- **2026-04-11**：embedding null 导致 NPE（532次），但所有测试都没发现
  - 根因1：测试被 excludes 跳过了（53+58个），从没人修
  - 根因2：Maven 用 JDK 25 而项目配的是 Java 21，导致 Mockito 不兼容
  - 根因3：之前"修复"是升级 byte-buddy / 加 excludes / 加 experimental，没人修根因
  - 教训：**不要升级 byte-buddy 解决 JDK 兼容问题，要让 Maven 用对的 JDK**
  - 教训：测试被跳过 = 没有测试，必须修好跑通
  - 教训：验证必须走前端完整用户流程，不只是 curl API
  - 教训：发现问题时先查日志，不要猜
- **2026-04-11**：手动添加物品NPE（imageBase64为null）
  - 根因：@NotBlank强制要求图片 + Service层没处理null
  - 教训：每个接口都要测"不传必填参数"和"传null"的场景
- **2026-04-11**：反复出现同一个问题（Java 25兼容性）
  - 根因：每次治标（升级byte-buddy/excludes/experimental），不治本（Maven用错JDK）
  - 教训：问题出现2次以上必须找根因，不能再用临时方案

## 🔧 分工规则（2026-04-11）

### 开发Agent负责：
- 写业务代码（前后端都由同一个Agent做）
- 修测试代码（改断言、加mock、更新期望值）
- 写新的测试用例

### 开发Agent不能做：
- 自己跑测试然后报告"测试通过"
- 自己验证自己写的代码

### 创业助手（我）负责验证：
- 编译验证：`mvn compile`
- 单元测试：`mvn test`
- 集成测试：`mvn test -Dtest="com.finder.integration.*"`（自动启动Spring上下文，需要PostgreSQL）
- API冒烟：`curl` 命令
- 前端完整用户流程：打开APP操作

### spawn Agent时的规则：
- task内容必须和给用户看的计划完全一致，一个字不省
- 包含：文件绝对路径、行号、具体改什么、验证命令、预计行数
