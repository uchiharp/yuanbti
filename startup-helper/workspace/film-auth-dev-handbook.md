# 1. UI 设计规范

## 1.1 设计哲学

**"内容是主角，工具是透明的。"**

影视工作者每天面对大量视频素材，UI 的职责是让内容呼吸、让操作无感。每一个像素都在回答：**这个界面是否让创作者更快地完成工作？**

### 核心原则

- 暗色优先 — 剪辑室、调色室、放映室全是暗环境，亮色 UI 刺眼
- 视频内容是视觉焦点 — 背景不抢注意力
- 信息密度适中 — 连续看素材几小时不会疲劳
- 时间线驱动 — 影视制作本质是时间轴上的工作流

### 参考产品

- DaVinci Resolve — 暗色专业工具标杆
- Frame.io — 视频审批协作，品牌橙色系
- ShotGrid（Autodesk）— 项目管理
- Notion 暗色模式 — 内容管理参考

## 1.2 色彩体系

### 背景层级（暖灰，去蓝紫倾向）

| 名称 | 色值 | 用途 |
|------|------|------|
| Surface-0 | `#0D0D0F` | 全局最底色（body） |
| Surface-1 | `#151518` | 侧边栏、弹窗底色 |
| Surface-2 | `#1E1E22` | 卡片、表格行底色 |
| Surface-3 | `#27272C` | 输入框、下拉框底色 |
| Surface-4 | `#313138` | Hover 态、分割线 |
| Surface-Warm | `#E8753A@3%` | 选中态、播放器区域底色（带温度的灰） |

> **为什么不选冷灰？** 冷色调（含蓝紫）长时间观看产生"工业冷感"，像监控室。暖灰更贴近影视后期的胶片质感，和品牌橙也更搭。

### 文字层级

| 名称 | 色值 | 用途 | 对比度 |
|------|------|------|--------|
| Text-Primary | `#F2F0ED` | 标题、重要信息（微暖白） | ≥ 7:1 |
| Text-Secondary | `#A8A4A0` | 正文、描述（暖灰） | ≥ 4.5:1 |
| Text-Tertiary | `#6E6A66` | 占位符、禁用态 | ≥ 3:1 |

### 强调色

| 名称 | 色值 | 用途 |
|------|------|------|
| Brand-Primary | `#E8753A` | 品牌橙，主 CTA 按钮 |
| Brand-Light | `#F09860` | 品牌橙浅色，hover 态 |
| Brand-Dark | `#C45E28` | 品牌橙深色，active 态 |

> **为什么选橙？** Frame.io 用橙，DaVinci 用橙。暗环境中橙色的视觉舒适度最高，且与影视行业的"胶片暖调"有隐约呼应。

### 功能色

| 名称 | 色值 | 用途 |
|------|------|------|
| Success | `#4ADE80` | 完成/通过 |
| Warning | `#FBBF24` | 待处理/进行中 |
| Danger | `#EF4444` | 错误/拒绝/紧急 |
| Info | `#60A5FA` | 提示/链接 |

### 特殊色

| 色值 | 用途 |
|------|------|
| `#E8753A@20%` | 选中态背景 |
| `#E8753A@8%` | hover/聚焦微暖底色 |
| `#FFFFFF@8%` | 通用 hover 叠加 |

### 渐变规则

**不使用渐变。** 唯一例外：视频播放器下边缘可做 1px `#E8753A@15%` 发光线，暗示"内容在此聚焦"。其余位置用纯色 + 透明度叠加表达层级。

## 1.3 字体

### 字体栈

```css
--font-primary: 'Inter', 'SF Pro Display', 'Noto Sans SC', -apple-system, sans-serif;
--font-mono: 'JetBrains Mono', 'SF Mono', 'Menlo', monospace;
--font-display: 'Inter', 'SF Pro Display', sans-serif;
```

### 字号阶梯

| 用途 | 大小 | 字重 | 行高 | 场景 |
|------|------|------|------|------|
| Display | 28px | 600 | 1.2 | 页面大标题 |
| H1 | 22px | 600 | 1.3 | 模块标题 |
| H2 | 18px | 500 | 1.4 | 区域标题 |
| Body | 14px | 400 | 1.5 | 正文、表格 |
| Caption | 12px | 400 | 1.4 | 辅助信息、时间戳 |
| Micro | 11px | 500 | 1.3 | 标签、badge |

### 中文注意事项

- 正文最小 14px，不小于此
- 标题用 600 字重（semi-bold），中文 bold 太粗会糊
- 数字和英文用等宽字体（版本号、时长、分辨率）

| 倍数 | 用途 |

- 侧边栏固定宽度：**240px**（收起 64px）
- 内容区：自适应，最大宽度 **1600px**（超出居中）
- 卡片网格：4 列（1600px）/ 3 列（1200px）/ 2 列（800px）

## 1.10 暗色模式专属细节

- 视频播放器区域永远是页面最暗的区域（`#0A0A0E`），比页面背景还深一级
- 文字对比度 ≥ 4.5:1（WCAG AA），标题 ≥ 7:1
- 阴影在暗色模式下用发光替代：`box-shadow: 0 0 20px rgba(232,117,58,0.15)`
- 图标用线条风格（outline），不填充，减少视觉噪音

## 2.2 单元测试

### AI 最容易犯的错

| 错误 | 后果 | 反例 |
|------|------|------|
| Mock 太完美 | 测了个寂寞，真实场景全挂 | Mock 返回永远正确的数据 |
| 只测 happy path | 异常输入直接 500 | 只传正常参数 |
| 自证预言 | 测试复述代码逻辑，不验证行为 | 先写实现再补测试 |

### 规则

1. **先写测试再写实现**（TDD），哪怕只是写个空的断言框架
2. **三种输入必须测**：正常值、空值/null、超出边界值
3. **Mock 原则**：只 mock 外部依赖（DB、API），不 mock 自己的业务逻辑
4. **命名公式**：`methodName_场景_预期结果`，例：`findTop20_emptyData_returnsEmptyList`

### 示例

```java
@Test
void createProject_withNullName_throwsException() {
    assertThatThrownBy(() -> projectService.create(null, "description"))
        .isInstanceOf(IllegalArgumentException.class)
        .hasMessageContaining("name");
}

@Test
void findTop20_emptyData_returnsEmptyList() {
    // 不 mock，用真实空数据库
    List<Project> result = projectService.findTop20();
    assertThat(result).isEmpty();  // 不是 null
}
```

### 真实教训

> Finder App 的 `findTop20` 空结果异常 — 单元测试全绿但 E2E 挂了。根因：Mock 返回了 `emptyList()`，但真实代码返回 `null` 导致 NPE。

## 2.3 集成测试

### AI 最容易犯的错

| 错误 | 后果 |
|------|------|
| 每个测试启停容器 | 10 分钟跑完 → 开发者跳过 |
| 测试间互相污染 | 随机失败，不可复现 |
| 不测接口契约 | 前后端字段名不一致，上线才发现 |

### 规则

1. **用 Testcontainers**，不 mock 数据库
2. **每个用例自包含**：setup → execute → assert → cleanup（或 `@Transactional` 回滚）
3. **前后端契约是最高优先级**：字段名、类型、必填/可选、null 处理
4. **少而精**：只测模块间接缝，不重复单元测试逻辑

### 接口契约测试清单

每个 API 接口必须验证：

```
□ 请求：URL、Method、Headers、Body（字段名、类型、必填/可选）
□ 响应：状态码、Body（字段名、类型、null 处理）
□ 不传必填参数 → 400
□ 传 null → 400 或有默认值
□ 传超长字符串 → 截断或 400
□ 不存在的资源 ID → 404
□ 无权限 → 403
□ 未登录 → 401
```

### 真实教训

> 前后端接口不一致导致 4 个 API 返回 400。embedding null 导致 NPE 532 次，所有单元测试都没发现。

## 2.4 E2E 测试

### AI 最容易犯的错（重灾区）

| 错误 | 后果 |
|------|------|
| 伪造测试结果 | LLM 生成"通过"比执行命令容易 |
| 用空数据测试 | 新用户啥都没有，当然"正常" |
| 只测 API 不测 UI | curl 200 ≠ 用户能完成操作 |
| 不截图不留痕 | 失败了没法复现 |

### 规则

1. **每个用例必须**：打开真实浏览器 → 填写真实数据 → 验证可见结果 → 失败时截图
2. **测试数据独立**：每个 suite 有独立 setup/teardown，不依赖其他用例
3. **必须覆盖三种场景**：空数据（新用户第一体验）、正常流程、错误输入
4. **失败必须留痕**：截图 + 日志 + 错误上下文

### E2E 用例模板

```typescript
import { test, expect } from '@playwright/test';

test.describe('项目管理', () => {
  const projectName = `测试项目-${Date.now()}`;  // 唯一，不冲突

  test('创建项目', async ({ page }) => {
    await page.goto('/projects');
    await page.click('button:text("新建项目")');
    await page.fill('#name', projectName);
    await page.fill('#description', 'E2E 测试项目');
    await page.click('button:text("确定")');

    // 验证：列表中出现新项目
    await expect(page.locator(`text=${projectName}`)).toBeVisible();
  });

  test('删除项目', async ({ page }) => {
    // 自包含：先创建再删除
    const tempName = `待删除-${Date.now()}`;
    // ... 创建逻辑 ...
    // ... 删除逻辑 ...
    // 验证：列表中不再出现
    await expect(page.locator(`text=${tempName}`)).not.toBeVisible();
  });
});
```

### 真实教训

> "所有 Agent 都在假装测试" — 2026-04-09，这是最重要的一条。每个验证项必须实际执行命令，不能编造结果。
>
> 用户指出"从前端页面进入实际体验" — 不能只 curl API。
> — 2026-04-08

## 2.5 性能测试

### 工具选择

| 工具 | 适合场景 | 学习成本 |
|------|---------|---------|
| **k6** | API 压测，JS 脚本 | 低 ✅ 推荐 |
| JMeter | 复杂场景、GUI 操作 | 中 |
| Gatling | 高并发、好看报告 | 中 |

### 快速上手（k6）

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
  stages: [
    { duration: '10s', target: 20 },   // 慢慢升到 20 并发
    { duration: '30s', target: 50 },   // 升到 50 并发
    { duration: '10s', target: 0 },    // 降回 0
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],   // P95 < 500ms
    http_req_failed: ['rate<0.01'],      // 错误率 < 1%
  },
};

export default function () {
  let res = http.get('http://localhost:8080/api/projects',
    { headers: { Authorization: `Bearer ${__ENV.TOKEN}` } });
  check(res, { 'status 200': (r) => r.status === 200 });
  sleep(1);
}
```

```bash
# 运行
k6 run --env TOKEN=xxx script.js
```

### 快速检查（curl 自带计时）

```bash
curl -o /dev/null -s -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \
  http://localhost:8080/api/projects
```

### 必须关注的指标

| 指标 | 警戒值 | 说明 |
|------|--------|------|
| P50 响应时间 | < 200ms | 一半请求的响应时间 |
| P95 响应时间 | < 500ms | 95% 请求的响应时间 |
| P99 响应时间 | < 1000ms | 尾部延迟 |
| 错误率 | < 1% | 超过就要查 |
| 数据库慢查询 | > 500ms | 需要索引优化 |

## 2.6 边界条件测试

**每个接口必须过一遍的清单：**

| 边界类型 | 测试输入 | 正确行为 |
|---------|---------|---------|
| 空值/null | `username: null` | 返回 400，不是 500 |
| 超长字符串 | 10000 字符的名称 | 截断或报错，不崩溃 |
| 负数/零 | `pageSize: 0`, `pageSize: -1` | 返回空列表或默认值 |
| 最大值 | `pageSize: 99999` | 不 OOM |
| 特殊字符 | `<script>alert(1)</script>` | 不被 XSS 执行 |
| SQL 注入 | `'; DROP TABLE users; --` | 参数化查询，不受影响 |
| 并发冲突 | 两人同时改同一条 | 乐观锁 / 409 Conflict |
| 已删除数据 | 访问已删除的资源 ID | 404 |
| 权限边界 | A 用户操作 B 的资源 | 403 |
| 时间边界 | 过期 token、未来日期 | 正确拒绝 |
| 极端数量 | 列表 0 条 / 10000 条 | 不崩溃，合理分页 |
| Unicode | 中文、emoji、右左文 | 正确存储和显示 |

## 2.7 错误场景测试

**原则：每个外部依赖都假设它会挂。**

| 依赖挂了 | 模拟方法 | 预期行为 |
|---------|---------|---------|
| 数据库连接断 | Testcontainers 关掉容器再请求 | 503 + 友好错误信息 |
| Redis 不可用 | 停掉 Redis 服务 | 降级到无缓存模式或 503 |
| 第三方 API 超时 | Mock 设置 30s 延迟 | 不超过 5s 就返回超时错误 |
| 磁盘满 | 模拟写入失败 | 不丢数据，队列暂存 |
| 请求体格式错误 | 发非法 JSON | 400 不是 500 |
| 并发超限 | 压测超过线程池容量 | 排队或降级，不崩溃 |

## 2.8 测试数据隔离

### 核心问题

测试用例之间共享状态 → A 创建的数据影响 B → 测试不可重复执行。

### 方案一：每个测试自建自清（推荐）

```java
@BeforeEach
void setup() {
    testUser = userService.create("test-" + UUID.randomUUID(), "Pass123!");
    testProject = projectService.create(testUser.getId(), "项目-" + UUID.randomUUID());
}

@AfterEach
void cleanup() {
    // 清理，或者用 @Transactional 自动回滚
}
```

### 方案二：数据库快照（适合 E2E）

```bash
# 测试前
pg_dump -Fc finder > /tmp/test-snapshot.dump

# 测试后
pg_restore --clean -d finder /tmp/test-snapshot.dump
```

### 方案三：Testcontainers 隔离数据库（最佳实践）

```java
@Testcontainers
class ProjectIntegrationTest {
    @Container
    static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("pgvector/pgvector:pg16");
    // 每次运行都是全新空数据库，测试完自动销毁
}
```

### E2E 测试隔离（Playwright）

```typescript
test.beforeEach(async ({ page }) => {
  // 每个测试用唯一标识，不会冲突
  testName = `测试-${Date.now()}-${Math.random().toString(36).slice(2, 6)}`;
});
```

### 铁律

> **任何测试用例跑 100 遍，结果必须一致。如果不一致，就是设计错了。**

### E2E 测试覆盖率

E2E 覆盖率按**业务场景**衡量，不按代码行衡量。用场景清单打勾的方式追踪。

### 覆盖率目标

| 层级 | 关注指标 | 目标 |
|------|---------|------|
| 单元测试 | 行覆盖率 / 分支覆盖率 | > 80% / > 70% |
| 集成测试 | 接口覆盖 | 100% |
| E2E 测试 | 业务场景覆盖 | P0 100%，P1 > 80% |
| 性能测试 | 关键接口响应时间 | P95 < 500ms |

> **不要追求 100% 行覆盖率。** 那会导致写一堆无意义的测试。关键路径 100% 覆盖，其他路径够用就行。

## 2.10 通用铁律

| # | 铁律 | 说明 |
|---|------|------|
| 1 | **执行 > 声明** | 必须实际跑命令，不能写"假设通过" |
| 2 | **空数据是新用户第一体验** | 必须第一个测 |
| 3 | **前后端接口是最高风险的接缝** | 字段名、类型、null 处理必须每个都验 |
| 4 | **失败要留痕** | 截图 + 日志 + 错误上下文 |
| 5 | **测试不过 = 不通过** | 没有"差不多过了" |
| 6 | **根因 > 重试** | 同一问题出现 2 次以上必须找根因，不能贴膏药 |
| 7 | **测试必须幂等** | 跑 100 遍结果一致 |
| 8 | **每个外部依赖都可能挂** | 错误场景不是可选的 |
| 9 | **覆盖率是手段不是目的** | 关键路径 100%，其他够用就行 |
